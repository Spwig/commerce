title: OAuth 및 소셜 로그인 설정
---

OAuth 및 소셜 로그인은 고객이 기존의 Google, Apple 또는 Microsoft 계정을 사용하여 가게에 로그인할 수 있도록 해줍니다. 새로운 비밀번호를 생성하고 기억할 필요가 없습니다.

![OAuth 설정](/static/core/admin/img/help/oauth-social-login/oauth-settings.webp)

## OAuth / 소셜 로그인 이란 무엇인가요?

OAuth는 Google, Apple 또는 Microsoft와 같은 신뢰할 수 있는 제공업체의 자격 증명을 사용하여 고객이 로그인할 수 있도록 해주는 보안 인증 표준입니다.

### 이점

- **빠른 결제** — 고객은 등록 양식을 건너뛰고 한 번의 클릭으로 로그인합니다.
- **마찰 감소** — 비밀번호 생성, 확인 이메일, 잊어버린 비밀번호 흐름이 필요 없습니다.
- **더 나은 전환율** — 연구에 따르면 소셜 로그인이 전환율을 20-40% 증가시킬 수 있습니다.
- **보안 강화** — 자격 증명은 가게를 통해 전달되지 않으며 인증은 제공업체에서 처리됩니다.
- **고객 신뢰** — 고객은 기존 제공업체에 로그인 자격 증명을 신뢰합니다.

### 작동 방식

1. 고객이 로그인 페이지에서 "Google로 로그인"(또는 Apple/Microsoft)을 클릭합니다.
2. 제공업체의 보안 로그인 페이지로 리디렉션됩니다.
3. 고객은 제공업체의 자격 증명으로 인증합니다.
4. 제공업체는 확인된 신원 정보를 가게로 다시 보냅니다.
5. 고객은 자동으로 로그인됩니다.

처음 로그인 시, 제공업체의 이메일 및 프로필 정보를 사용하여 새로운 고객 계정이 자동으로 생성됩니다.

## 지원되는 제공업체

Spwig은 세 가지 주요 OAuth 제공업체를 지원합니다:

| 제공업체 | 사용 사례 | 자격 증명 요구 사항 |
|----------|----------|------------------------|
| **Google** | 가장 인기 있고 설정이 가장 간단한 | 클라이언트 ID, 클라이언트 비밀 |
| **Apple** | iOS 앱에 필수적이며 개인정보 보호에 중점을 둔 | 클라이언트 ID, 팀 ID, 키 ID, 개인 키 |
| **Microsoft** | 기업 고객, Office 365 사용자 | 클라이언트 ID, 클라이언트 비밀, 테넌트 ID |

한 개, 두 개 또는 세 개의 제공업체를 활성화할 수 있습니다. 각각은 독립적으로 작동합니다.

## Google OAuth 설정

Google OAuth는 가장 인기 있는 옵션으로 설정이 가장 간단합니다.

### 사전 조건

- Google 계정
- Google Cloud Console에 대한 액세스

### 단계별 설정

1. **OAuth 설정으로 이동**
   - 관리자 패널에서 **설정 > 가게 설정**으로 이동
   - **OAuth 제공업체** 섹션으로 스크롤
   - **Google 설정**을 클릭합니다.

2. **Google Cloud 프로젝트 생성**
   - [Google Cloud Console](https://console.cloud.google.com/)에 방문
   - **프로젝트 생성**을 클릭
   - 프로젝트 이름을 입력 (예: "My Store OAuth")
   - **생성**을 클릭

3. **Google+ API 활성화**
   - 왼쪽 사이드바에서 **API 및 서비스 > 라이브러리**로 이동
   - "Google+ API"를 검색
   - **활성화**를 클릭

4. **OAuth 자격 증명 생성**
   - **API 및 서비스 > 자격 증명**으로 이동
   - **자격 증명 생성 > OAuth 클라이언트 ID**를 클릭
   - 애플리케이션 유형을 선택: **웹 애플리케이션**
   - 이름을 입력 (예: "가게 로그인")

5. **리디렉션 URI 구성**
   - **인증된 리디렉션 URI** 아래에 추가:
     ```
     https://yourdomain.com/accounts/google/login/callback/
     ```
   - `yourdomain.com`을 실제 도메인으로 대체
   - **생성**을 클릭

6. **자격 증명 복사**
   - 팝업에서 **클라이언트 ID** 및 **클라이언트 비밀**을 복사합니다.

7. **Spwig에 자격 증명 입력**
   - Spwig 관리자 OAuth 설정으로 돌아갑니다.
   - 클라이언트 ID 및 클라이언트 비밀을 붙여넣습니다.
   - **저장**을 클릭합니다.
   - **Google OAuth 활성화**를 토글하여 활성화합니다.

### 테스트

- 가게 전면 로그인 페이지를 방문합니다.
- "Google로 로그인" 버튼을 찾습니다.
- 클릭하여 Google 계정으로 인증합니다.
- 로그인되어 고객 대시보드로 리디렉션됩니다.

## Apple OAuth 설정

Apple OAuth는 키 기반 인증 시스템으로 인해 Google보다 더 복잡합니다.

### 사전 조건

- Apple 개발자 계정 (유료 가입 필요)
- Apple 개발자 포털에 대한 액세스

### 단계별 설정

1. **OAuth 설정으로 이동**
   - **설정 > 가게 설정 > OAuth 제공업체**로 이동
   - **Apple 설정**을 클릭합니다.

2. **서비스 ID 생성**
   - [Apple 개발자](https://developer.apple.com/account/)에 로그인
   - **인증서, 식별자 및 프로파일**로 이동
   - **식별자**를 클릭한 후 **+** 버튼을 클릭
   - **서비스 식별자**를 선택하고 **계속**을 클릭
   - 설명을 입력 (예: "가게 로그인")
   - 식별자를 입력 (예: `com.yourstore.login`)
   - **계속**을 클릭한 후 **등록**을 클릭합니다.

3. **서비스 ID 구성**
   - 새로 생성한 서비스 ID를 클릭합니다.
   - **Apple로 로그인**을 선택합니다.
   - **설정**을 클릭합니다.
   - 도메인과 반환 URL을 추가합니다:
     - **도메인**: `yourdomain.com`
     - **반환 URL**: `https://yourdomain.com/accounts/apple/login/callback/`
   - **저장**을 클릭한 후 **계속**을 클릭하고 다시 **저장**을 클릭합니다.

4. **키 생성**
   - 왼쪽 사이드바에서 **키**를 클릭한 후 **+** 버튼을 클릭합니다.
   - 키 이름을 입력 (예: "가게 OAuth 키")
   - **Apple로 로그인**을 선택합니다.
   - **설정**을 클릭하고 주요 앱 ID를 선택합니다.
   - **저장**을 클릭한 후 **계속**을 클릭하고 **등록**을 클릭합니다.
   - **다운로드된 키 파일** (.p8) — 다시 다운로드할 수 없습니다.

5. **필요한 정보 수집**
   필요 정보:
   - **클라이언트 ID (서비스 ID)**: 생성한 식별자 (예: `com.yourstore.login`)
   - **팀 ID**: Apple 개발자 포털 상단 오른쪽에 있습니다.
   - **키 ID**: 키를 생성할 때 표시됩니다.
   - **개인 키**: 다운로드한 .p8 파일의 내용

6. **Spwig에 자격 증명 입력**
   - Spwig OAuth 설정으로 돌아갑니다.
   - 클라이언트 ID, 팀 ID 및 키 ID를 붙여넣습니다.
   - 텍스트 편집기에서 .p8 파일을 열고 내용을 복사합니다.
   - 전체 키 (헤더 포함)를 개인 키 필드에 붙여넣습니다.
   - **저장**을 클릭합니다.
   - **Apple OAuth 활성화**를 토글하여 활성화합니다.

### 테스트

- Apple ID가 있는 장치에서 가게 전면 로그인 페이지를 방문합니다.
- "Apple로 로그인"을 클릭합니다.
- Apple ID로 인증합니다.
- 성공적으로 로그인되어야 합니다.

## Microsoft OAuth 설정

Microsoft OAuth는 Office 365 또는 Azure AD를 사용하는 기업 고객을 대상으로 하는 가게에 이상적입니다.

### 사전 조건

- Microsoft 계정
- Azure 포털에 대한 액세스

### 단계별 설정

1. **OAuth 설정으로 이동**
   - **설정 > 가게 설정 > OAuth 제공업체**로 이동
   - **Microsoft 설정**을 클릭합니다.

2. **Azure에서 애플리케이션 등록**
   - [Azure 포털](https://portal.azure.com/)에 방문
   - **Azure Active Directory > 애플리케이션 등록**으로 이동
   - **새 등록**을 클릭합니다.
   - 이름을 입력 (예: "가게 OAuth")
   - **모든 조직 디렉토리 및 개인 Microsoft 계정**을 선택합니다.
   - **리디렉션 URI** 아래에서 **웹**을 선택하고 다음을 입력:
     ```
     https://yourdomain.com/accounts/microsoft/login/callback/
     ```
   - **등록**을 클릭합니다.

3. **애플리케이션 ID 복사**
   - 애플리케이션 개요 페이지에서 **애플리케이션 (클라이언트) ID**를 복사합니다.

4. **클라이언트 비밀 생성**
   - 왼쪽 사이드바에서 **인증서 및 비밀**을 클릭합니다.
   - **새 클라이언트 비밀**을 클릭합니다.
   - 설명을 입력 (예: "OAuth 비밀")
   - 만료 기간을 선택 (권장: 24개월)
   - **추가**를 클릭합니다.
   - **즉시 비밀 값 복사** — 다시 표시되지 않습니다.

5. **Spwig에 자격 증명 입력**
   - Spwig OAuth 설정으로 돌아갑니다.
   - 애플리케이션 (클라이언트) ID를 클라이언트 ID로 붙여넣습니다.
   - 비밀 값을 클라이언트 비밀로 붙여넣습니다.
   - 선택적으로 테넌트 ID를 입력합니다 (단일 테넌트 앱의 경우; 다중 테넌트인 경우 빈칸으로 남김)
   - **저장**을 클릭합니다.
   - **Microsoft OAuth 활성화**를 토글하여 활성화합니다.

### 테스트

- 가게 전면 로그인 페이지를 방문합니다.
- "Microsoft로 로그인"을 클릭합니다.
- Microsoft 계정으로 인증합니다.
- 성공적으로 로그인되어야 합니다.

## OAuth 연결 관리

### 고객 뷰

고객은 계정 대시보드에서 연결된 OAuth 제공업체를 볼 수 있고 관리할 수 있습니다:

- **내 계정 > 연결된 계정**으로 이동
- 연결된 제공업체(Google, Apple, Microsoft)를 확인합니다.
- 연결된 제공업체를 해제하려면 **연결 해제**를 클릭합니다.
- 다시 연결하려면 해당 제공업체로 다시 로그인합니다.

### 여러 제공업체

단일 고객 계정은 여러 OAuth 제공업체에 연결될 수 있습니다. 예를 들어, 고객은 동일한 계정에 Google과 Apple을 모두 연결할 수 있습니다.

동일한 이메일 주소로 다른 OAuth 제공업체로 로그인을 시도하는 경우, Spwig은 기존 계정에 자동으로 연결합니다.

### 관리자 관리

관리자로서 고객의 OAuth 연결을 볼 수 있습니다:

- **고객 > 고객**으로 이동
- 고객 기록을 열고
- **연결된 계정** 섹션으로 스크롤합니다.
- 연결된 제공업체와 연결 시간을 확인합니다.

고객 대신 제공업체를 해제할 수 없습니다 — 보안상 이유로 고객이 직접 해야 합니다.

## 문제 해결

### 리디렉션 URI 불일치

**에러**: "리디렉션 URI 불일치" 또는 "잘못된 redirect_uri"

**해결 방법**:
- 제공업체 설정에서 리디렉션 URI가 Spwig에 있는 것과 정확히 일치하는지 확인합니다.
- 끝에 슬래시가 있는지 확인합니다 — 반드시 일치해야 합니다.
- `https://`를 사용하는지 확인합니다 (아니면 `http://`가 아님)
- 브라우저 캐시를 지우고 다시 시도합니다.

### 유효하지 않은 자격 증명

**에러**: "잘못된 클라이언트 ID" 또는 "인증 실패"

**해결 방법**:
- Client ID 및 Client Secret이 올바르게 복사되었는지 다시 확인합니다.
- 추가 공백이나 줄 바꿈이 없는지 확인합니다.
- 자격 증명이 올바른 프로젝트/앱에서 가져왔는지 확인합니다.
- Apple의 경우, 개인 키가 .p8 파일의 전체 내용을 포함하는지 확인합니다.

### 제공업체 API 비활성화

**에러**: "API 비활성화" 또는 "접근 설정되지 않음"

**해결 방법**:
- Google의 경우: Google Cloud 프로젝트에서 Google+ API가 활성화되어 있는지 확인합니다.
- Microsoft의 경우: 애플리케이션 등록이 승인되고 활성화되어 있는지 확인합니다.
- Apple의 경우: Service ID에 "Apple로 로그인"이 활성화되어 있는지 확인합니다.

### SSL 필요

**에러**: "OAuth는 HTTPS를 요구함" 또는 "비보안 리디렉션 URI"

**해결 방법**:
- OAuth 제공업체는 보안을 위해 SSL/TLS (HTTPS)를 요구합니다.
- 가게에 유효한 SSL 인증서가 설치되어 있는지 확인합니다.
- 리디렉션 URI를 `https://` 대신 `http://`로 사용하지 않도록 업데이트합니다.
- 로컬에서 테스트하는 경우, ngrok과 같은 서비스를 사용하여 HTTPS 터널을 생성합니다.

### 버튼이 나타나지 않음

**문제**: "Google/Apple/Microsoft로 로그인" 버튼이 로그인 페이지에 나타나지 않습니다.

**해결 방법**:
- OAuth 설정에서 제공업체가 활성화되어 있는지 확인합니다.
- 브라우저 캐시를 지우고 페이지를 새로고침합니다.
- 테마가 소셜 로그인 템플릿을 포함하고 있는지 확인합니다.
- 브라우저 콘솔에서 JavaScript 오류를 확인합니다.

## 팁 및 최선의 실천

### 보안

- **정기적으로 비밀을 교체** — 12-24개월마다 Client Secrets를 업데이트합니다.
- **실패한 로그인 시도를 모니터링** — 비정상적인 인증 패턴을 감시합니다.
- **환경별로 별도의 자격 증명 사용** — 스테이징 및 프로덕션에 대해 다른 자격 증명을 사용합니다.
- **리디렉션 URI를 제한** — 필요한 URI만 추가합니다.

### 사용자 경험

- **세 가지 제공업체 모두 활성화** — 고객에게 선택지를 제공하고, 다른 인구 통계는 다른 제공업체를 선호합니다.
- **버튼을 눈에 띄게 배치** — 소셜 로그인 버튼은 이메일/비밀번호 양식 위에 배치되어야 합니다.
- **인식 가능한 브랜딩 사용** — 표준 Google/Apple/Microsoft 버튼 스타일을 유지합니다.
- **모바일에서 테스트** — OAuth 흐름은 모바일 브라우저에서 다르게 작동합니다.

### 준수

- **개인정보 보호 정책** — OAuth 제공업체를 사용하고 어떤 데이터를 받는지 공개합니다.
- **이용 약관** — 제공업체의 이용 약관을 준수합니다 (Google, Apple, Microsoft 각각에 요구사항이 있습니다).
