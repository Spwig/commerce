---
title: Вход в административную панель с использованием одноразового входа (SSO)
---

Одноразовый вход (SSO) позволяет вашему персоналу входить в административную панель с использованием поставщика удостоверений вашей организации вместо отдельного имени пользователя и пароля. Spwig поддерживает любой поставщик удостоверений, использующий протокол OpenID Connect (OIDC), включая Microsoft Entra ID, Google Workspace, Okta, Auth0, Keycloak и другие.

## Что такое корпоративный SSO?

Корпоративный SSO отличается от социальной авторизации (входа с помощью личного аккаунта Google или Facebook). С использованием корпоративного SSO:

- Персонал аутентифицируется через **поставщика удостоверений вашей организации** — ту же систему, которую они используют для электронной почты, внутренних инструментов и других корпоративных приложений
- Ваша команда IT централизованно управляет доступом — когда кто-то покидает организацию, отключение его аккаунта в поставщике удостоверений мгновенно отзывает его доступ к Spwig
- Многофакторная аутентификация (MFA) обеспечивается поставщиком удостоверений, обеспечивая последовательную политику безопасности для всех приложений
- Персонал не должен запоминать отдельный пароль для Spwig

## Как это работает

Когда SSO включен, страница входа в админ-панель отображает кнопку **Войти через [Поставщик]**. Процесс аутентификации работает следующим образом:

1. Сотрудник нажимает кнопку SSO на странице входа в Spwig
2. Его перенаправляют на страницу входа в вашем поставщике удостоверений (например, вход в Microsoft)
3. Он аутентифицируется через поставщика удостоверений (включая любую MFA, требуемую поставщиком)
4. Поставщик удостоверений перенаправляет его обратно в Spwig с безопасным кодом авторизации
5. Spwig обменивает код на информацию о пользователе и создает сессию
6. Сотрудник попадает в административную панель, полностью аутентифицированный

Это использует отраслевой стандартный протокол **OpenID Connect (OIDC)**, который поддерживается почти всеми корпоративными поставщиками удостоверений.

## Включение SSO

SSO настраивается в двух местах:

1. **Настройки сайта > вкладка Безопасность** — включить или отключить SSO и управлять видимостью входа по паролю
2. **Настройка поставщика SSO** — введите детали OIDC вашего поставщика удостоверений

### Шаг 1: Настройка вашего поставщика удостоверений

Перед включением SSO в Spwig необходимо зарегистрировать Spwig как приложение в вашем поставщике удостоверений. См. руководства, специфичные для поставщика:

- **Microsoft Entra ID** — см. руководство по настройке Microsoft Entra ID
- **Google Workspace** — см. руководство по настройке Google Workspace
- **Okta** — см. руководство по настройке Okta
- **Другие поставщики** — работает любой поставщик, соответствующий OIDC. Зарегистрируйте веб-приложение с URI перенаправления `https://your-store.com/oidc/callback/` и обратитесь к документации вашего поставщика для получения URL OIDC Discovery, Client ID и Client Secret.

### Шаг 2: Настройка поставщика SSO в Spwig

Перейдите на страницу **Настройка поставщика SSO** (ссылка из вкладки Безопасности или доступна по пути **Корпоративный SSO > Настройка поставщика SSO** в боковом меню администратора). Введите:

1. **Название поставщика** — отображается на кнопке входа (например, «Microsoft Entra ID»)
2. **OIDC Discovery URL** — URL вашего поставщика `.well-known/openid-configuration`. Нажмите **Автоматическое обнаружение**, чтобы автоматически заполнить поля конечной точки.
3. **Client ID** и **Client Secret** — из регистрации приложения вашего поставщика удостоверений

Секрет клиента хранится зашифрованным и никогда не отображается после сохранения.

### Шаг 3: Включение SSO в настройках сайта

Перейдите в **Настройки сайта > вкладка Безопасность** и установите флажок **Включить SSO для входа в админ-панель**. Кнопка SSO сразу появится на странице входа в админ-панель.

## Настройки SSO

| Настройка | Описание |
|---------|-------------|
| **Включить SSO для входа в админ-панель** | Отображает кнопку SSO на странице входа в админ-панель. Не влияет на обычный вход по паролю, если вы также не отключите его. |
| **Разрешить вход по паролю на странице администратора** | Если не отмечено, форма ввода пароля скрывается за съёмной вкладкой. По умолчанию сотрудники видят только кнопку SSO. Форма ввода пароля всё ещё может быть доступна, нажав «Войти через локальный аккаунт» или добавив `?password=1` к URL входа. |

### Поведение страницы входа

| SSO включён | Вход по паролю | Результат |
|-------------|---------------|--------|
| Выкл | Вкл | Стандартная страница входа с формой имени пользователя/пароля |
| Вкл | Вкл | Кнопка SSO вверху, разделитель "или", затем форма пароля ниже |
| Вкл | Выкл | Только кнопка SSO. Форма пароля скрыта за переключателем "Войти с локальной учётной записью" |
| Выкл | Выкл | Невозможно — вход по паролю автоматически включается, если SSO отключён или не настроен |

## Соответствие пользователей

Когда сотрудник входит в систему через SSO, Spwig сопоставляет его с существующей учётной записью пользователя по **электронной почте** (без учёта регистра). Электронная почта из утверждений идентификационного провайдера должна совпадать с электронной почтой учётной записи сотрудника в Spwig.

Если совпадающая учётная запись не найдена:

- **Создание пользователей автоматически отключено** (по умолчанию) — вход запрещён. Сначала необходимо создать учётную запись сотрудника в Spwig с совпадающим адресом электронной почты.
- **Создание пользователей автоматически включено** — создаётся новая учётная запись пользователя автоматически с именем и электронной почтой из утверждений идентификационного провайдера.

Настройка **Ограничить только для сотрудников** (включена по умолчанию) добавляет дополнительную проверку: даже если учётная запись пользователя существует, вход запрещён, если у пользователя нет статуса сотрудника. Это предотвращает доступ к панели администратора через SSO для несоставных учётных записей.

## Сопоставление ролей

Если ваш идентификационный провайдер передаёт информацию о членстве в группах в утверждениях OIDC, Spwig может автоматически назначить статус сотрудника и суперпользователя на основе членства в группах.

Чтобы настроить сопоставление ролей:

1. В конфигурации провайдера SSO установите поле **Утверждение групп** на имя утверждения, используемое вашим провайдером (по умолчанию: `groups`)
2. В **Группы сотрудников** введите имена или идентификаторы групп, разделённые запятыми. Пользователи, входящие в любую из этих групп, получают статус сотрудника.
3. В **Группы суперпользователей** введите имена или идентификаторы групп, разделённые запятыми. Пользователи, входящие в любую из этих групп, получают статус суперпользователя.

Сопоставление ролей проверяется каждый раз, когда пользователь входит в систему через SSO. Если пользователь удаляется из группы в идентификационном провайдере, его статус сотрудника или суперпользователя обновляется при следующем входе через SSO.

**Важно:** Microsoft Entra ID по умолчанию передаёт **идентификаторы объектов групп** (UUID), а не имена групп. Скопируйте идентификатор объекта из портала Azure при настройке сопоставления ролей. Другие провайдеры, такие как Okta, обычно передают имена групп.

## Сопоставление утверждений

Spwig считывает информацию о пользователях из стандартных утверждений OIDC. По умолчанию это работает с большинством провайдеров, но вы можете настроить названия полей утверждений в конфигурации провайдера SSO:

| Настройка | По умолчанию | Описание |
|---------|---------|-------------|
| **Утверждение электронной почты** | `email` | Утверждение, содержащее адрес электронной почты пользователя |
| **Утверждение имени** | `given_name` | Утверждение, содержащее имя пользователя |
| **Утверждение фамилии** | `family_name` | Утверждение, содержащее фамилию пользователя |
| **Утверждение групп** | `groups` | Утверждение, содержащее членство в группах (оставьте пустым, чтобы отключить сопоставление ролей) |

## Поведение MFA

Когда сотрудник входит в систему через SSO, встроенное требование двухфакторной аутентификации (2FA) в Spwig автоматически обходит. Это связано с тем, что идентификационный провайдер отвечает за выполнение MFA в рамках потока входа SSO.

Если ваша организация требует MFA, настройте её в политиках условного доступа вашего идентификационного провайдера, а не в настройках 2FA в Spwig. Это позволяет централизованно управлять MFA для всех ваших приложений.

## Восстановление доступа

Если ваш идентификационный провайдер испытывает сбой или неправильную настройку, вы всё равно можете получить доступ к форме входа в администратора:

- **Нажмите переключатель** — Если вход по паролю отключён, нажмите "Войти с локальной учётной записью" на странице входа, чтобы отобразить форму пароля
- **Параметр URL** — Добавьте `?password=1` к URL входа в администратора (например, `https://your-store.com/en/admin/login/?password=1`), чтобы отобразить форму пароля напрямую
- **Вход по паролю всегда доступен** — Даже если форма скрыта в интерфейсе, задняя часть аутентификации по паролю остаётся активной. Влияется только видимость формы.

Spwig также предотвращает отключение входа по паролю, если SSO не включен и правильно настроен — вы не можете случайно заблокировать себя.

## Поддерживаемые провайдеры

Spwig работает с любым идентификационным провайдером, поддерживающим протокол OpenID Connect (OIDC). Подробные руководства по настройке доступны для:

- **Microsoft Entra ID** (ранее Azure Active Directory)
- **Google Workspace** (Google Cloud Identity)
- **Okta**

Для других провайдеров, соответствующих OIDC (Auth0, Keycloak, OneLogin, Ping Identity, JumpCloud и т.д.), шаги настройки в Spwig одинаковы — вам понадобится URL-адрес OIDC Discovery провайдера, Client ID и Client Secret. Обратитесь к документации вашего провайдера, чтобы узнать, как зарегистрировать веб-приложение и получить эти учетные данные. Используемый URI перенаправления всегда будет `https://your-store.com/oidc/callback/`.

## Советы

- **Начните с включения входа по паролю** — сначала включите SSO вместе с входом по паролю. После того как вы подтвердите, что SSO работает для вашей команды, вы можете по желанию отключить вход по паролю.
- **Тестируйте в режиме инкогнито** — используйте частное/инкогнито окно браузера для тестирования SSO, чтобы не повлиять на ваш текущий сеанс администратора.
- **Сначала создайте учетные записи сотрудников** — если вы не включите Auto-Create Users, сотрудники должны иметь существующий аккаунт Spwig с совпадающим адресом электронной почты, чтобы войти через SSO.
- **Используйте кнопку Авто-открытия** — введите URL-адрес OIDC Discovery провайдера и нажмите Авто-открытие, чтобы автоматически заполнить все поля конечных точек. Это быстрее и менее подвержено ошибкам, чем ввод конечных точек вручную.
- **Сохраните локальный аккаунт администратора** — всегда сохраняйте как минимум один локальный аккаунт администратора с паролем в качестве резервного варианта на случай проблем с провайдером идентификации.
- **Контролируйте срок действия секрета клиента** — некоторые провайдеры (в частности Microsoft Entra ID) выдают секреты клиентов со сроком действия. Установите напоминание в календаре, чтобы обновить секрет до его истечения.
