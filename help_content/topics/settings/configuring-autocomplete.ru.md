---
title: Настройка автодополнения
---

Автодополнение, также называемое предиктивным поиском или поиском при вводе, отображает результаты, пока клиенты вводят свои запросы. Это значительно улучшает пользовательский опыт, помогая клиентам быстрее находить товары и уменьшая количество поисков без результатов. Это руководство объясняет, как настроить поведение автодополнения, настройки отображения и компромиссы по производительности.

Автодополнение включено по умолчанию со сбалансированными настройками. Регулируйте их только в случае специфических проблем производительности или предпочтений отображения.

![Настройки автодополнения](/static/core/admin/img/help/configuring-autocomplete/autocomplete-settings-main.webp)

## Включение автодополнения

Перейдите в **Поиск > Настройки поиска** и нажмите вкладку **Автодополнение**.

**Включить автодополнение** - главный переключатель для предиктивного поиска. При включении поля ввода поиска показывают выпадающий список результатов при вводе клиентами.

**Максимальное количество результатов по типу** - по умолчанию: 8 элементов. Сколько результатов показывать для каждого типа контента (товары, категории, бренды, блог-посты). Низкие значения (5-6) уменьшают размер нагрузки API и ускоряют отображение. Высокие значения (10-12) дают клиентам больше вариантов, но замедляют ответ.

## Задержка автодополнения

⚠️ **ПРЕДУПРЕЖДЕНИЕ О ПРОИЗВОДИТЕЛЬНОСТИ** - Задержка автодополнения значительно влияет на нагрузку на сервер.

**Задержка автодополнения** - по умолчанию: 300мс. Как долго ждать после последнего нажатия клавиши перед запуском запроса автодополнения.

Эта настройка сбалансирует отзывчивость и нагрузку на сервер:

| Задержка | Пользовательский опыт | Влияние на сервер |
|----------|----------------------|-------------------|
| **100мс** | Очень отзывчиво     | 3x больше вызовов API, чем 300мс - высокая нагрузка |
| **200мс** | Отзывчиво          | 1.5x больше вызовов API, чем 300мс |
| **300мс** | Хорошее соотношение (рекомендуется) | Базовое значение |
| **400мс** | Слегка медленно     | Меньше вызовов API - меньшая нагрузка |
| **500мс** | Заметно задержано   | 50% меньше вызовов, но ощущается медленно |

**Рекомендация**: Оставайтесь в диапазоне 250-350мс. Увеличивайте выше 350мс только если ваш сервер испытывает трудности с нагрузкой автодополнения. Никогда не уменьшайте ниже 200мс, если только у вас не очень быстрый сервер и небольшой каталог.

## Настройки отображения для товаров

Эти переключатели управляют тем, какая информация отображается в результатах автодополнения товаров:

**Показать миниатюру** - по умолчанию: ВКЛ. Отображает изображение товара рядом с результатом. **Влияние на производительность**: Добавляет запрос изображения и увеличивает размер JSON-нагрузки. Отключите, чтобы ускорить автодополнение при медленных соединениях.

**Показать описание** - по умолчанию: ВЫКЛ. Отображает краткое описание товара. **Влияние на производительность**: Добавляет обработку текста и значительно увеличивает размер нагрузки. Оставьте отключенным, если описания не критичны для выбора товара.

**Показать цену** - по умолчанию: ВКЛ. Отображает цену товара. **Влияние на производительность**: Низкое - данные о цене уже загружены с товаром. Безопасно оставить включенным.

**Показать SKU** - по умолчанию: ВКЛ. Отображает SKU товара. **Влияние на производительность**: Низкое - SKU уже индексирован. Необходимо для B2B-магазинов.

**Показать статус наличия** - по умолчанию: ВЫКЛ. ⚠️ **ВАЖНОЕ ПРЕДУПРЕЖДЕНИЕ О ПРОИЗВОДИТЕЛЬНОСТИ**

Отображает значки "В наличии", "Низкий запас" или "Нет в наличии". **Никогда не включайте это для больших каталогов**.

Статус наличия требует агрегации `with_stock_totals()` - вычисления количества товара на руках по всем складам для каждого товара в результатах автодополнения. Это добавляет:
- Значительную нагрузку на базу данных (агрегирующие запросы)
- Дополнительную задержку 200-500мс для каталогов с более чем 1000 товарами
- Возможные тайм-ауты для каталогов с более чем 10000 товарами

Включайте только если это абсолютно критично и у вас менее 500 товаров.

## Настройки отображения для блог-постов

**Показать основное изображение** - по умолчанию: ВКЛ. Миниатюра блог-поста в результатах автодополнения.

**Показать выдержку** - по умолчанию: ВКЛ. Краткий текст из содержимого поста.

**Длина выдержки** - по умолчанию: 60 символов. Сколько текста выдержки показывать.

Эти настройки имеют минимальное влияние на производительность, поскольку блог-посты обычно в небольшом количестве по сравнению с товарами.

## Настройки отображения для категорий и брендов

**Показать миниатюру/логотип** - по умолчанию: ВКЛ. Изображение категории или бренда в результатах.

**Показать количество товаров** - по умолчанию: ВЫКЛ. ⚠️ **ПРЕДУПРЕЖДЕНИЕ О ПРОИЗВОДИТЕЛЬНОСТИ**

Отображает, сколько товаров в каждой категории или бренде (например, "Электроника (234)").

**Никогда не включайте это для больших каталогов**. Количество товаров пересчитывается при каждом запросе автодополнения:
- Каждый тип контента с включенным подсчетом добавляет 2 дополнительных запроса
- Запросы включают соединения и агрегации
- Типичная дополнительная задержка 100-300мс
- Увеличивается линейно с количеством категорий/брендов

Включайте только если у вас менее 50 категорий/брендов И менее 1000 товаров в общей сложности.

## Кэширование

**TTL кэша автодополнения** - по умолчанию: 60 секунд (установлено в вкладке Кэширование).

Результаты автодополнения кэшируются для улучшения производительности. TTL в 60 секунд означает:
- Первый клиент, ищущий "laptop", запускает запрос к базе данных
- В течение следующих 59 секунд все поиски "laptop" возвращают кэшированные результаты
- После 60 секунд кэш истекает, и следующий поиск обновляет данные

**Рекомендации по TTL**:
- **45-60с**: Хорошее соотношение для большинства магазинов (по умолчанию)
- **90-120с**: Лучшая производительность, если инвентарь товаров меняется редко
- **30с**: Более актуальные результаты, если вы часто добавляете товары

Увеличение TTL кэша - это самый простой способ улучшить производительность автодополнения.

## Автодополнение с поддержкой нескольких языков

Если у вас настроены несколько языков, автодополнение автоматически ищет переведенное содержание, хранящееся в полях JSONField перевода.

**Как это работает**:
- Клиент ищет на испанском: "zapatos"
- Система ищет испанские переводы названий товаров
- Результаты показывают испанские названия товаров из данных JSONField
- Если испанский перевод отсутствует, переходит к основному языку

**Производительность**: Минимальная перегрузка для 1-3 языков. При 5+ языках небольшое увеличение сложности запроса.

## Тестирование автодополнения

После настройки параметров протестируйте опыт автодополнения:

1. **Откройте главную страницу вашего магазина** в режиме инкогнито
2. **Нажмите поле поиска**, чтобы активировать его
3. **Выведите название распространенного товара** медленно (например, "laptop")
4. **Наблюдайте**:
   - Как быстро появляются результаты после того, как вы прекратите ввод (работает ли задержка?)
   - Какая информация отображается (миниатюры, цены, SKU, как настроено)
   - Соответствуют ли результаты вашему запросу (проверьте веса релевантности, если нет)
5. **Проверьте на мобильных устройствах** - Убедитесь, что выпадающий список легко нажимается и читается на телефонах

## Советы

- **Отключите описания товаров для скорости** - Описания значительно увеличивают размер нагрузки с минимальной пользой в контексте автодополнения
- **Никогда не включайте статус наличия для больших каталогов** - Агрегация запасов убивает производительность автодополнения
- **Проверяйте на мобильных устройствах с целями нажатия** - Результаты автодополнения должны легко нажиматься на телефонах
- **Контролируйте время отклика еженедельно** - Цель: <200мс для запросов автодополнения
- **Увеличьте TTL кэша, если медленно** - Самый простой оптимизация производительности
- **Количество товаров дорого - отключите, если не критично** - Каждый подсчет категории/бренда добавляет 2 запроса к каждому запросу автодополнения
