---
title: การเข้าสู่ระบบเดียว (SSO) สำหรับผู้ดูแลระบบ
---

การเข้าสู่ระบบเดียว (SSO) ช่วยให้พนักงานของคุณสามารถเข้าสู่ระบบในแดชบอร์ดผู้ดูแลระบบโดยใช้ระบบรับรองตัวตนขององค์กรแทนชื่อผู้ใช้และรหัสผ่านที่แยกต่างหาก การสนับสนุน SSO ของ Spwig รองรับระบบรับรองตัวตนใด ๆ ที่ใช้โปรโตคอล OpenID Connect (OIDC) รวมถึง Microsoft Entra ID, Google Workspace, Okta, Auth0, Keycloak และอื่น ๆ อีกมากมาย

## อะไรคือ Enterprise SSO?

Enterprise SSO แตกต่างจาก Social Login (การเข้าสู่ระบบด้วยบัญชีส่วนตัวของ Google หรือ Facebook) ด้วย Enterprise SSO:

- พนักงานจะยืนยันตัวตนผ่าน **ระบบรับรองตัวตนขององค์กร** — ระบบเดียวกันที่พวกเขาใช้สำหรับอีเมล เครื่องมือภายใน และแอปพลิเคชันธุรกิจอื่น ๆ
- ทีม IT ของคุณสามารถควบคุมการเข้าถึงแบบรวมศูนย์ — เมื่อใครสักคนออกจากองค์กร การปิดใช้งานบัญชีในระบบรับรองตัวตนจะยกเลิกการเข้าถึง Spwig ทันที
- การยืนยันตัวตนหลายปัจจัย (MFA) จะถูกบังคับโดยระบบรับรองตัวตน ทำให้คุณมีนโยบายความปลอดภัยที่สอดคล้องกันทั่วทั้งแอปพลิเคชัน
- พนักงานไม่จำเป็นต้องจำรหัสผ่านแยกต่างหากสำหรับ Spwig

## วิธีการทำงาน

เมื่อเปิดใช้งาน SSO หน้าเข้าสู่ระบบผู้ดูแลระบบจะแสดงปุ่ม **Sign in with [Provider]** ขั้นตอนการยืนยันตัวตนทำงานดังนี้:

1. พนักงานคลิกปุ่ม SSO บนหน้าเข้าสู่ระบบ Spwig
2. พวกเขาจะถูกเปลี่ยนเส้นทางไปยังหน้าเข้าสู่ระบบของระบบรับรองตัวตนของคุณ (เช่น การเข้าสู่ระบบของ Microsoft)
3. พวกเขาจะยืนยันตัวตนกับระบบรับรองตัวตน (รวมถึง MFA ที่ระบบรับรองตัวตนต้องการ)
4. ระบบรับรองตัวตนจะเปลี่ยนเส้นทางกลับไปยัง Spwig พร้อมกับรหัสการอนุญาตที่ปลอดภัย
5. Spwig จะแลกเปลี่ยนรหัสเพื่อขอข้อมูลผู้ใช้และสร้างเซสชัน
6. พนักงานจะเข้าสู่แดชบอร์ดผู้ดูแลระบบโดยมีการยืนยันตัวตนครบถ้วน

สิ่งนี้ใช้โปรโตคอลมาตรฐานของอุตสาหกรรม **OpenID Connect (OIDC)** ซึ่งได้รับการสนับสนุนจากระบบรับรองตัวตนขององค์กรเกือบทุกแห่ง

## การเปิดใช้งาน SSO

SSO ถูกกำหนดค่าในสองสถานที่:

1. **Site Settings > Security tab** — เปิดหรือปิดใช้งาน SSO และควบคุมการมองเห็นการเข้าสู่ระบบด้วยรหัสผ่าน
2. **SSO Provider Configuration** — ใส่รายละเอียด OIDC ของระบบรับรองตัวตนของคุณ

### ขั้นตอนที่ 1: กำหนดค่าระบบรับรองตัวตนของคุณ

ก่อนเปิดใช้งาน SSO ใน Spwig คุณต้องลงทะเบียน Spwig เป็นแอปพลิเคชันในระบบรับรองตัวตนของคุณ ดูคู่มือเฉพาะของผู้ให้บริการ:

- **Microsoft Entra ID** — ดูคู่มือการตั้งค่า Microsoft Entra ID
- **Google Workspace** — ดูคู่มือการตั้งค่า Google Workspace
- **Okta** — ดูคู่มือการตั้งค่า Okta
- **ผู้ให้บริการอื่น ๆ** — ผู้ให้บริการใด ๆ ที่เป็นไปตาม OIDC ทำงานได้ ลงทะเบียนแอปพลิเคชันเว็บโดยใช้ URI การเปลี่ยนเส้นทาง `https://your-store.com/oidc/callback/` และดูเอกสารของผู้ให้บริการของคุณเพื่อหา URL การค้นหา OIDC, Client ID และ Client Secret

### ขั้นตอนที่ 2: กำหนดค่า SSO Provider ใน Spwig

ไปที่หน้า **SSO Provider Configuration** (เชื่อมโยงจากแท็บ Security หรือเข้าถึงได้ที่ **Enterprise SSO > SSO Provider Configuration** ในแถบด้านข้างของผู้ดูแลระบบ) ใส่:

1. **Provider Name** — แสดงบนปุ่มเข้าสู่ระบบ (เช่น "Microsoft Entra ID")
2. **OIDC Discovery URL** — URL `.well-known/openid-configuration` ของผู้ให้บริการของคุณ คลิก **Auto-Discover** เพื่อเติมข้อมูลปลายทางอัตโนมัติ
3. **Client ID** และ **Client Secret** — จากการลงทะเบียนแอปพลิเคชันของผู้ให้บริการของคุณ

Client Secret จะถูกเก็บในรูปแบบที่เข้ารหัสและจะไม่แสดงหลังจากบันทึก

### ขั้นตอนที่ 3: เปิดใช้งาน SSO ใน Site Settings

ไปที่ **Site Settings > Security** และตรวจสอบ **Enable SSO for admin login** ปุ่ม SSO จะปรากฏทันทีบนหน้าเข้าสู่ระบบผู้ดูแลระบบ

## การตั้งค่า SSO

| การตั้งค่า | คำอธิบาย |
|---------|-------------|
| **Enable SSO for admin login** | แสดงปุ่ม SSO บนหน้าเข้าสู่ระบบผู้ดูแลระบบ ไม่ส่งผลต่อการเข้าสู่ระบบด้วยรหัสผ่านเว้นแต่คุณจะปิดใช้งานมันด้วย |
| **Allow password login on admin page** | เมื่อไม่ได้ตรวจสอบ จะซ่อนฟอร์มรหัสผ่านไว้เบื้องหลังปุ่มที่สามารถพับได้ พนักงานจะเห็นเพียงปุ่ม SSO ทั่วไป ฟอร์มรหัสผ่านยังสามารถเข้าถึงได้โดยการคลิก "Sign in with local account" หรือโดยการเพิ่ม `?password=1` ไปยัง URL เข้าสู่ระบบ |

### พฤติกรรมหน้าเข้าสู่ระบบ


| SSO ได้รับการเปิดใช้งาน | การเข้าสู่ระบบด้วยรหัสผ่าน | ผลลัพธ์ |
|------------------|------------------|--------|
| ปิด | เปิด | หน้าเข้าสู่ระบบมาตรฐานที่มีฟอร์มชื่อผู้ใช้/รหัสผ่านเท่านั้น |
| เปิด | เปิด | ปุ่ม SSO อยู่ด้านบน, ตัวแบ่ง "หรือ", แล้วตามด้วยฟอร์มรหัสผ่านด้านล่าง |
| เปิด | ปิด | ปุ่ม SSO เท่านั้น ฟอร์มรหัสผ่านอยู่เบื้องหลังตัวเลือก "เข้าสู่ระบบด้วยบัญชีท้องถิ่น" |
| ปิด | ปิด | ไม่สามารถทำได้ — การเข้าสู่ระบบด้วยรหัสผ่านจะถูกเปิดใช้งานอัตโนมัติหาก SSO ถูกปิดใช้งานหรือไม่ได้กำหนดค่า |

## การจับคู่ผู้ใช้

เมื่อพนักงานเข้าสู่ระบบผ่าน SSO, Spwig จะจับคู่พวกเขาเข้ากับบัญชีผู้ใช้ที่มีอยู่โดยใช้ **ที่อยู่อีเมล** (ไม่คำนึงถึงตัวพิมพ์ใหญ่/เล็ก) ที่อยู่อีเมลจากข้อมูลการยืนยันตัวตนของผู้ให้บริการต้องตรงกับที่อยู่อีเมลในบัญชี Spwig ของพนักงาน

หากไม่พบผู้ใช้ที่ตรงกัน:

- **การสร้างผู้ใช้อัตโนมัติถูกปิดใช้งาน** (ค่าเริ่มต้น) — การเข้าสู่ระบบถูกปฏิเสธ คุณต้องสร้างบัญชีพนักงานใน Spwig ก่อนด้วยที่อยู่อีเมลที่ตรงกัน
- **การสร้างผู้ใช้อัตโนมัติถูกเปิดใช้งาน** — บัญชีผู้ใช้ใหม่จะถูกสร้างขึ้นโดยอัตโนมัติพร้อมชื่อและอีเมลจากข้อมูลการยืนยันตัวตนของผู้ให้บริการ

การตั้งค่า **จำกัดเฉพาะพนักงาน** (เปิดใช้งานเป็นค่าเริ่มต้น) จะเพิ่มการตรวจสอบเพิ่มเติม: แม้ผู้ใช้จะมีบัญชีอยู่แล้ว การเข้าสู่ระบบจะถูกปฏิเสธหากผู้ใช้ไม่มีสถานะพนักงาน นี่ช่วยป้องกันไม่ให้บัญชีที่ไม่ใช่พนักงานเข้าถึงแดชบอร์ดผ่าน SSO

## การจัดการบทบาท

หากผู้ให้บริการของคุณส่งข้อมูลการเป็นสมาชิกกลุ่มในข้อมูล OIDC, Spwig สามารถตั้งค่าสถานะพนักงานและผู้ใช้ระดับสูงโดยอัตโนมัติโดยอิงจากกลุ่มที่ผู้ใช้อยู่

เพื่อตั้งค่าการจัดการบทบาท:

1. ในการตั้งค่าผู้ให้บริการ SSO, ตั้งค่าฟิลด์ **กลุ่มที่อ้างอิง** เป็นชื่อข้อมูลที่ผู้ให้บริการใช้ (ค่าเริ่มต้น: `groups`)
2. ใน **กลุ่มพนักงาน**, ใส่ชื่อกลุ่มหรือ ID ที่คั่นด้วยเครื่องหมายจุลภาค ผู้ใช้ที่อยู่ในกลุ่มใดกลุ่มหนึ่งจะได้รับสถานะพนักงาน
3. ใน **กลุ่มผู้ใช้ระดับสูง**, ใส่ชื่อกลุ่มหรือ ID ที่คั่นด้วยเครื่องหมายจุลภาค ผู้ใช้ที่อยู่ในกลุ่มใดกลุ่มหนึ่งจะได้รับสถานะผู้ใช้ระดับสูง

การจัดการบทบาทจะถูกประเมินทุกครั้งที่ผู้ใช้เข้าสู่ระบบผ่าน SSO หากผู้ใช้ถูกนำออกจากรายชื่อกลุ่มในผู้ให้บริการ สถานะพนักงานหรือผู้ใช้ระดับสูงของผู้ใช้นั้นจะถูกอัปเดตเมื่อเข้าสู่ระบบผ่าน SSO ครั้งต่อไป

**สำคัญ:** Microsoft Entra ID ส่ง **Object ID ของกลุ่ม** (UUID) ตามค่าเริ่มต้น ไม่ใช่ชื่อกลุ่ม คัดลอก Object ID จากพอร์ทัล Azure เมื่อตั้งค่าการจัดการบทบาท ผู้ให้บริการอื่นๆ เช่น Okta มักส่งชื่อกลุ่ม

## การจับคู่ข้อมูลการยืนยันตัวตน

Spwig อ่านข้อมูลผู้ใช้จากข้อมูล OIDC มาตรฐาน ค่าเริ่มต้นทำงานกับผู้ให้บริการส่วนใหญ่ แต่คุณสามารถปรับชื่อฟิลด์ข้อมูลการยืนยันตัวตนในการตั้งค่าผู้ให้บริการ SSO:

| ตั้งค่า | ค่าเริ่มต้น | คำอธิบาย |
|--------|-----------|-----------|
| **ข้อมูลอีเมล** | `email` | ข้อมูลที่มีที่อยู่อีเมลของผู้ใช้ |
| **ข้อมูลชื่อจริง** | `given_name` | ข้อมูลที่มีชื่อจริงของผู้ใช้ |
| **ข้อมูลนามสกุล** | `family_name` | ข้อมูลที่มีนามสกุลของผู้ใช้ |
| **ข้อมูลกลุ่ม** | `groups` | ข้อมูลที่มีการเป็นสมาชิกกลุ่ม (ทิ้งว่างเพื่อปิดการจัดการบทบาท) |

## พฤติกรรมการยืนยันตัวตนหลายปัจจัย (MFA)

เมื่อพนักงานเข้าสู่ระบบผ่าน SSO, ข้อกำหนดการยืนยันตัวตนสองปัจจัย (2FA) ที่สร้างขึ้นโดย Spwig จะถูกข้ามโดยอัตโนมัติ เนื่องจากผู้ให้บริการต้องรับผิดชอบในการบังคับใช้ MFA เป็นส่วนหนึ่งของขั้นตอนการเข้าสู่ระบบ SSO

หากองค์กรของคุณต้องการ MFA ให้ตั้งค่าในนโยบายการเข้าถึงแบบมีเงื่อนไขของผู้ให้บริการ ไม่ใช่ในการตั้งค่า 2FA ของ Spwig นี่จะให้คุณจัดการ MFA แบบรวมศูนย์สำหรับแอปพลิเคชันทั้งหมดของคุณ

## การเข้าถึงการกู้คืน

หากผู้ให้บริการของคุณประสบกับการหยุดทำงานหรือการตั้งค่าผิดพลาด คุณยังสามารถเข้าถึงฟอร์มการเข้าสู่ระบบแดชบอร์ดได้:

- **คลิกสวิตช์** — หากการเข้าสู่ระบบด้วยรหัสผ่านถูกปิดใช้งาน คลิก "เข้าสู่ระบบด้วยบัญชีท้องถิ่น" บนหน้าการเข้าสู่ระบบเพื่อเปิดเผยฟอร์มรหัสผ่าน
- **พารามิเตอร์ URL** — เพิ่ม `?password=1` ที่ URL การเข้าสู่ระบบแดชบอร์ด (เช่น `https://your-store.com/en/admin/login/?password=1`) เพื่อแสดงฟอร์มรหัสผ่านโดยตรง
- **การเข้าสู่ระบบด้วยรหัสผ่านมีอยู่เสมอ** — แม้จะถูกซ่อนจาก UI การยืนยันตัวตนด้วยรหัสผ่านยังคงทำงานอยู่ แค่การมองเห็นฟอร์มเท่านั้นที่ได้รับผลกระทบ

Spwig ยังช่วยป้องกันไม่ให้คุณปิดการเข้าสู่ระบบด้วยรหัสผ่าน เว้นแต่ SSO จะถูกเปิดใช้งานและตั้งค่าอย่างถูกต้อง — คุณไม่สามารถล็อกตัวเองออกโดยไม่ได้ตั้งใจได้

## ผู้ให้บริการที่รองรับ

Spwig ทำงานร่วมกับผู้ให้บริการตัวตนใด ๆ ที่รองรับโปรโตคอล OpenID Connect (OIDC) มีคู่มือตั้งค่าที่ละเอียดสำหรับ:

- **Microsoft Entra ID** (เดิมคือ Azure Active Directory)
- **Google Workspace** (Google Cloud Identity)
- **Okta**

สำหรับผู้ให้บริการอื่น ๆ ที่เป็นไปตาม OIDC (Auth0, Keycloak, OneLogin, Ping Identity, JumpCloud, ฯลฯ) ขั้นตอนการตั้งค่าของ Spwig จะเหมือนกัน — คุณจำเป็นต้องมี URL สำหรับ OIDC Discovery ของผู้ให้บริการ, Client ID และ Client Secret โปรดตรวจสอบเอกสารของผู้ให้บริการเพื่อดูวิธีลงทะเบียนแอปเว็บและรับข้อมูลเหล่านี้ URL ที่ใช้จะเป็น `https://your-store.com/oidc/callback/` เสมอ

## เคล็ดลับ

- **เริ่มต้นด้วยการเปิดใช้งานการเข้าสู่ระบบด้วยรหัสผ่าน** — เปิดใช้งาน SSO พร้อมกับการเข้าสู่ระบบด้วยรหัสผ่านก่อน เมื่อคุณยืนยันว่า SSO ทำงานสำหรับทีมของคุณแล้ว คุณสามารถปิดการเข้าสู่ระบบด้วยรหัสผ่านได้ตามต้องการ
- **ทดสอบในหน้าต่าง Incognito** — ใช้หน้าต่างเบราว์เซอร์แบบส่วนตัว/Incognito เพื่อทดสอบ SSO โดยไม่ได้รับผลกระทบจากเซสชันผู้ดูแลระบบปัจจุบันของคุณ
- **สร้างบัญชีพนักงานก่อน** — เว้นแต่คุณจะเปิดใช้งาน Auto-Create Users พนักงานจะต้องมีบัญชี Spwig ที่มีอยู่แล้วพร้อมที่อยู่อีเมลที่ตรงกันก่อนที่พวกเขาจะสามารถเข้าสู่ระบบผ่าน SSO ได้
- **ใช้ปุ่ม Auto-Discover** — ใส่ URL สำหรับ OIDC Discovery ของผู้ให้บริการของคุณและคลิก Auto-Discover เพื่อเติมข้อมูลทุกจุดปลายทางอัตโนมัติ นี่จะเร็วกว่าและลดข้อผิดพลาดมากกว่าการป้อนจุดปลายทางด้วยตนเอง
- **รักษาบัญชีผู้ดูแลระบบในท้องถิ่นไว้** — ต้องมีบัญชีผู้ดูแลระบบในท้องถิ่นอย่างน้อยหนึ่งบัญชีที่มีรหัสผ่านไว้เป็นทางเลือกสำหรับการกู้คืนในกรณีที่มีปัญหากับผู้ให้บริการตัวตน
- **ติดตามการหมดอายุของ Client Secret** — ผู้ให้บริการบางราย (โดยเฉพาะ Microsoft Entra ID) จะออก Client Secret ที่มีวันหมดอายุ ตั้งเตือนในปฏิทินเพื่อเปลี่ยน Secret ก่อนที่จะหมดอายุ
