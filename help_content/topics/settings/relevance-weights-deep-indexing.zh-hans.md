---
title: 相关性权重和深度索引
---

相关性权重和深度索引控制搜索结果的排序方式以及哪些产品数据会被搜索。权重是重要性乘数 - 2.0 的权重意味着该字段的匹配比 1.0 的权重重要两倍。深度索引决定搜索是否超越基本产品名称，深入到 SKU、属性、评论，甚至文档内容中。本指南将解释这两个系统，何时调整它们，以及关键的性能影响。

默认设置对大多数电子商务商店都适用。只有在有特定的排序或索引需求时才进行调整。

![Weights Tab](/static/core/admin/img/help/search-settings-overview/search-settings-weights.webp)

## 理解权重

权重是应用于不同字段中文字匹配的乘数（0.0-2.0 范围）。更高的权重意味着该字段的匹配在结果中排名更高。

**示例**：如果产品名称（权重 1.50）和描述（权重 0.80）中都有“laptop”：
- 名称匹配对相关性评分贡献 1.50
- 描述匹配对相关性评分贡献 0.80
- 综合评分决定与其他产品的排名

权重允许你在排序搜索结果时优先考虑某些字段。

## 权重类别和默认值

导航到 **Search Settings > Weights tab** 查看所有权重设置：

| Field | Default Weight | Rationale |
|-------|---------------|-----------|
| **weight_name** | 1.50 | 产品名称最重要 - 用户期望在搜索结果顶部看到精确的名称匹配 |
| **weight_sku** | 1.20 | SKU 是具体标识符 - 对于 B2B 和回头客非常重要 |
| **weight_description** | 0.80 | 描述提供上下文，但不如精确名称匹配重要 |
| **weight_categories** | 0.80 | 分类匹配对浏览有帮助，但不如名称/SKU 具体 |
| **weight_attributes** | 0.70 | 颜色、尺寸、材质搜索 - 有用但属于支持信息 |
| **weight_brands** | 0.70 | 品牌过滤很重要，但对大多数商店来说不是主要的搜索标准 |
| **weight_blog_posts** | 0.60 | 博客内容在以电子商务为主的搜索中重要性较低（最低优先级） |
| **weight_reviews** | 0.50 | 用户生成的内容最不受控制 - 权重最低 |

这些默认值假设了一个典型的电子商务商店，其中产品发现是主要的搜索目标。

## 何时调整权重

当您的商店优先级与典型的电子商务模式不同时，调整权重：

**SKU 为主的商店（B2B、批发）** - 将 `weight_sku` 增加到 1.8-2.0，使产品代码搜索主导结果。B2B 客户通常通过精确的 SKU 进行搜索。

**以品牌为主的商店** - 当客户主要按品牌购物时（设计师服装、奢侈品），将 `weight_brands` 增加到 1.2-1.5。

**内容为主的商店** - 如果您是内容发布商或教育零售商，其中博客文章与产品一样重要，将 `weight_blog_posts` 增加到 0.9-1.2。

**属性为主的商店（时尚）** - 当客户经常按颜色、尺寸、风格属性搜索时，将 `weight_attributes` 增加到 1.0-1.2。

## 权重调整示例

| Store Type | Recommended Adjustments |
|-----------|------------------------|
| **B2B Wholesale** | weight_sku: 2.0, weight_name: 1.3, weight_description: 0.6 - 优先考虑产品代码 |
| **Fashion Boutique** | weight_attributes: 1.2, weight_brands: 1.2, weight_name: 1.4 - 颜色/风格/品牌重要 |
| **Content Publisher** | weight_blog_posts: 1.2, weight_name: 1.3, weight_reviews: 0.7 - 内容与产品一样重要 |
| **General E-commerce** | 使用默认值 - 对于典型的在线商店是平衡的 |

一次调整一个权重，并在进行其他更改之前进行测试。

## 深度索引概述

⚠️ **性能警告** - 每个深度索引选项都会增加查询复杂性和开销。

深度索引将搜索范围扩展到基本产品名称/描述之外，深入到其他数据中：

![Deep Indexing Tab](/static/core/admin/img/help/search-settings-overview/search-settings-deep-indexing.webp)

导航到 **Search Settings > Deep Indexing tab** 进行配置。

## 索引 SKU

**默认**：ON，**性能影响**：低

在搜索索引中包含产品和变体的 SKU。触发变体 JOIN（轻微成本）。

**保持开启的时机**：对于 B2B 商店，客户知道产品代码时必不可少。对于记得之前订单 SKU 的回头客也有帮助。

**关闭的时机**：除非你真的没有任何 SKU，否则不要关闭。性能影响可以忽略不计。

## 索引属性

**默认**：ON，**性能影响**：中等

在搜索索引中包含产品属性（颜色、尺寸、材质、自定义属性）。连接到属性表。

**保持开启的时机**：对于时尚、可配置产品或任何客户通过产品特性（如“红色连衣裙”、“大号T恤”）进行搜索的商店非常重要。

**关闭的时机**：产品目录超过20,000个，且每个产品有大量属性时，可能会有50-100ms的开销。只有在性能至关重要且客户不通过属性搜索时才关闭。

## 索引自定义字段

**默认**：ON，**性能影响**：中等

在搜索中包含商家定义的自定义字段（来自 JSONField）。需要遍历 JSONField。

**保持开启的时机**：如果你使用自定义字段来存储可搜索的产品数据（保修信息、规格、兼容性细节）。

**关闭的时机**：如果你不使用自定义字段，或者自定义字段包含非搜索数据（内部注释、会计代码）。关闭可以节省 JSONField 处理开销。

## 索引评论

**默认**：ON，**性能影响**：中等-高

在搜索中包含已批准的评论标题和评论。连接到评论表并增加文本搜索开销。

**保持开启的时机**：评论为主的目录，客户根据评论内容搜索产品（如“防水笔记本包”可能出现在评论文本中）。

**关闭的时机**：产品目录超过20,000个或每个产品有大量评论的商店。在大型目录上增加100-200ms的开销。

## 索引文档

**默认**：OFF，**性能影响**：非常高 🚨

**不要随意启用** - 最昂贵的搜索功能。

文档索引从附加到数字产品的PDF、DOCX和XLSX文件中提取文本，使文件内容可搜索。

**技术细节**：
- 使用 PyPDF2、python-docx 和 openpyxl 库
- 搜索时同步文件 I/O 和文本提取
- 通过 MD5 校验和跟踪文件（仅在文件更改时重新索引）
- 大文件（>10MB PDF）可能导致超时

**性能影响**：
- 非常昂贵的初始索引（大型库需要几分钟到几小时）
- 显著的查询开销（增加100-500ms的延迟）
- 大型文档内存消耗大

**仅在以下情况启用**：
- 你销售带有可搜索文档的数字产品（电子书、报告、手册）
- 目录较小（<500个数字产品）
- 服务器有足够的资源
- 你已彻底测试了影响

**对于数字产品商店**：考虑客户是否真的需要搜索文档内容，或者搜索产品名称/描述是否足够。

## 性能影响表

| Feature | Default | Impact | Use When |
|---------|---------|--------|----------|
| Index SKUs | ON | Low | Always (B2B essential) |
| Index Attributes | ON | Medium | Configurable products |
| Index Custom Fields | ON | Medium | Using custom fields |
| Index Reviews | ON | Medium-High | Review-heavy store |
| Index Documents | OFF | Very High | Digital products only (test first) |

影响基于典型目录。大型目录（>50,000 产品）会经历成比例更高的开销。

## 测试权重更改

调整权重时，请遵循以下测试流程：

1. **一次只更改一个权重** - 不要同时调整多个权重；否则你将不知道哪个更改导致了结果变化
2. **小幅度调整** - 每次调整 ±0.2（例如，1.0 → 1.2，而不是 1.0 → 1.8）
3. **使用真实查询进行测试** - 使用来自分析的实际客户搜索词，而不是随机测试
4. **监控分析数据** - 使用顶级查询比较更改前后的结果相关性
5. **等待 1-2 周** - 给客户时间与新的排名互动
6. **测量点击率** - 客户点击结果的数量比之前更多或更少吗？

## 性能与准确性的权衡

更多的索引 = 更好的搜索结果但更慢的性能：

**场景：小目录（<1,000 产品）**
- 启用所有索引选项（SKU、属性、自定义字段、评论）
- 性能影响最小
- 全面的搜索功能

**场景：中等目录（1,000-10,000 产品）**
- 保持 SKU、属性、自定义字段开启
- 如果每个产品的平均评论数超过10个，考虑关闭评论
- 监控响应时间

**场景：大型目录（>10,000 产品）**
- 保持 SKU 开启（低影响）
- 关闭评论索引（高影响）
- 如果未使用，关闭自定义字段
- 永远不要启用文档索引
- 在产品数量超过50,000时考虑使用Elasticsearch

根据您的目录大小和服务器资源进行平衡。

## 引擎特定的权重覆盖

通过向导创建搜索引擎（第3步）时，可以为该特定引擎覆盖全局权重。

**使用场景**：以博客为主的引擎
- 创建“blog”引擎
- 将 `weight_blog_posts` 覆盖为 1.5（相对于全局的 0.60）
- 博客内容现在在博客引擎搜索中排名更高

大多数引擎不应覆盖权重 - 留空以继承全局设置。

## 小贴士

- **除非绝对必要，否则不要启用文档索引** - 这是任何搜索功能中性能成本最高的
- **B2B 商店：将 weight_sku 增加到 2.0** - 产品代码是主要的搜索方法
- **在低流量时段测试权重更改** - 在高峰时段之前观察性能影响
- **启用索引后监控响应时间** - 检查分析仪表板是否有延迟
- **在产品目录超过20K时禁用评论索引** - 有显著的性能影响
- **一次只更改一个权重进行测试** - 同时更改无法确定因果关系
- **文档提取需要 PyPDF2/docx/openpyxl** - 在启用文档索引前确认这些库已安装
