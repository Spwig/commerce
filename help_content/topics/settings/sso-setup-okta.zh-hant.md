---
title: SSO設定：Okta
---

本指南將引導您將 Spwig 連接到 Okta，以實現管理員的單點登入（SSO）。設定完成後，您的員工可以使用其 Okta 帳戶登入 Spwig 管理面板。

**注意**：Okta 可能會隨著時間推移更新其管理員控制台介面。這些說明是根據 2026 年初的 Okta 管理員控制台撰寫的。如果任何步驟與您看到的內容不同，請參閱 Okta 的官方文件 [建立 OIDC 應用程式整合](https://developer.okta.com/docs/guides/sign-into-web-app-redirect/main/)。

## 前提條件

- 一個 Okta 組織（任何層級 — 免費開發者帳戶可用於測試）
- 在 Okta 中具有 **Super Administrator** 或 **Application Administrator** 角色
- 您的 Spwig 商店 URL（例如，`https://your-store.com`）
- 員工的電子郵件地址必須與其 Okta 帳戶相符

## 第 1 步：建立應用程式

1. 使用 [Okta 管理員控制台](https://your-org-admin.okta.com) 登入
2. 點擊 **Applications > Applications**
3. 點擊 **Create App Integration**
4. 選擇：

| 欄位 | 值 |
|-------|-------|
| **登入方式** | OIDC - OpenID Connect |
| **應用程式類型** | Web Application |

5. 點擊 **Next**

## 第 2 步：設定應用程式

填寫應用程式設定：

| 欄位 | 值 |
|-------|-------|
| **應用程式整合名稱** | `Spwig Admin SSO`（或您偏好的任何名稱） |
| **授權類型** | Authorization Code（預設應已選中） |
| **登入重導向 URI** | `https://your-store.com/oidc/callback/` |
| **登出重導向 URI** | `https://your-store.com/en/admin/login/` |
| **受控存取** | 根據您的需求選擇（請參閱下方） |

對於 **受控存取**，請選擇以下其中一種：

- **允許組織內所有用戶存取** — 所有 Okta 用戶都可以登入（您仍然可以使用限制為員工的設定來控制 Spwig 的存取）
- **限制存取到選定的群組** — 只有特定 Okta 群組中的用戶可以登入
- **暫時跳過群組指派** — 您稍後會手動指派用戶或群組

點擊 **儲存**。

**重要：** 登入重定向 URI 必須完全匹配 `https://your-store.com/oidc/callback/` — 包括結尾的斜線。

## 第 3 步：取得用戶端憑證

儲存後，應用程式會在 **一般** 索引標籤中顯示您的憑證：

| 值 | 找到位置 |
|-------|-----------------|
| **用戶端 ID** | 一般索引標籤，用戶端憑證區 |
| **用戶端密碼** | 一般索引標籤，用戶端憑證區（點擊眼睛圖示以顯示） |

複製這兩個值 — 您需要它們來設定 Spwig。

## 第 4 步：建立發現 URL

發現 URL 取決於您的 Okta 組織和授權伺服器：

**預設授權伺服器（最常見）：**
```
https://your-org.okta.com/.well-known/openid-configuration
```

**自訂授權伺服器（如果已設定）：**
```
https://your-org.okta.com/oauth2/{authorization-server-id}/.well-known/openid-configuration
```

將 `your-org.okta.com` 替換為您的實際 Okta 域名。您可以在管理員控制台的 URL 栏中或在 **設定 > 帳戶** 下找到您的 Okta 域名。

**提示：** 大多數組織使用 Org 授權伺服器（預設）。只有在 Okta 管理員特別設定自訂授權伺服器時，才使用自訂授權伺服器 URL。

## 第 5 步：指派用戶或群組

如果您在第 2 步中選擇了 "跳過群組指派"，您需要在用戶能夠登入之前指派用戶：

1. 在應用程式的 **指派** 索引標籤中，點擊 **指派**
2. 選擇 **指派給人員** 或 **指派給群組**
3. 選擇用戶或群組，然後點擊 **指派**
4. 點擊 **完成**

未指派給應用程式的用戶在嘗試 SSO 時會看到錯誤。

## 第 6 步：設定群組聲明（可選）

如果您希望 Spwig 根據 Okta 群組成員資格自動設定員工或超級用戶狀態：

1.

導航至管理員控制台的 **安全性 > API**
2.

選擇您的 **授權伺服器**（如果您尚未建立自訂伺服器，請使用 "default" 或 Org 授權伺服器）
3.

前往 **聲明** 索引標籤
4.

保留所有 markdown 格式、圖片路徑、程式碼區塊和技術術語。

點擊 **新增聲明**
5.

配置聲明：

| 欄位 | 值 |
|-------|-------|
| **名稱** | `groups` |
| **包含在令牌類型** | ID Token, Always |
| **值類型** | Groups |
| **過濾** | 匹配正則表達式：`.*`（用以包含所有群組） |
| **包含在** | 任何範圍（或 `openid` 如果您想要限制） |

6. 點擊 **建立**

**提示：** 與 Microsoft Entra ID 會傳送物件 ID 不同，Okta 默認會傳送 **群組名稱**。這使得角色映射更加直觀 — 您可以直接在 Spwig 的員工群組和超級用戶群組欄位中使用 Okta 群組的顯示名稱。

### 篩選群組

如果您的用戶屬於許多 Okta 群組，但您只想包含特定的群組在令牌中：

- 將過濾器從 `.*` 改為更特定的正則表達式，例如 `^Spwig.*` 來僅包含以 "Spwig" 開頭的群組
- 或使用 **開頭為**、**等於** 或 **包含** 過濾器，而不是正則表達式

## 第 7 步：在 Spwig 中進行配置

1. 在 Spwig 管理介面中，導航至 **企業 SSO > SSO 提供者配置**
2. 將 **提供者名稱** 設為 `Okta`
3. 輸入來自第 4 步的發現 URL
4. 點擊 **自動發現** — 這會自動填寫所有端點欄位
5. 輸入來自第 3 步的 **客戶端 ID**
6. 輸入來自第 3 步的 **客戶端密碼**
7. 如果您在第 6 步中配置了群組聲明：
   - 將 **群組聲明** 設為 `groups`
   - 在 **員工群組** 中，輸入應為員工的 Okta 群組名稱（以逗號分隔）
   - 在 **超級用戶群組** 中，輸入應為超級用戶的 Okta 群組名稱（以逗號分隔）
8. 點擊 **儲存**

## 第 8 步：啟用並測試

1.

導航至 **網站設定 > 安全性** 索引標籤
2.

勾選 **啟用 SSO 用於管理員登入**
3.

點擊 **儲存**
4.

在 **私人/無痕視窗** 中打開管理員登入頁面
5.

您應該會看到 **使用 Okta 登入** 按鈕
6.

點擊它 — 您應該會被導向 Okta 的登入頁面
7.


使用已分配到應用程式且其電子郵件與 Spwig 員工用戶相符的 Okta 帳戶登入
8.

您應該會被重新導向回 Spwig 管理儀表板

## 常見問題

| 問題 | 原因 | 解決方案 |
|---------|-------|----------|
| **重定向 URI 不被允許** | 重定向 URI 與應用程式配置不匹配 | 確認登入重定向 URI 為 `https://your-store.com/oidc/callback/`，並包含結尾斜線 |
| **用戶未分配到客戶端應用程式** | 用戶未分配到 Okta 應用程式 | 在「指派」選項卡中，將用戶或其群組分配到應用程式 |
| **在 Okta 登入成功但在 Spwig 失敗** | Spwig 中沒有匹配的用戶 | 確保 Spwig 中存在與電子郵件相同的員工帳戶。檢查「限制為員工」設定。 |
| **群組聲明為空** | 授權伺服器上未配置群組聲明 | 按照步驟 6 添加群組聲明。確保您將其添加到正確的授權伺服器。 |
| **錯誤的授權伺服器** | 發現 URL 使用與配置群組聲明的授權伺服器不同的伺服器 | 確認發現 URL 與您配置群組聲明的授權伺服器匹配 |
| **"提供的 client_id 無效"** | client_id 不匹配或應用程式處於非活動狀態 | 確認 client_id 是否正確，並在 Okta 中確認應用程式狀態為「Active」 |

## 小技巧

- **Okta 傳送群組名稱，而非 ID** — 這使得角色映射變得簡單。

在 Spwig 的員工群組或超級用戶群組欄位中，輸入精確的群組顯示名稱（例如：`Spwig Admins`）。
- **使用群組指派進行存取控制** — 將特定的 Okta 群組指派給 Spwig 應用程式，而不是允許所有用戶。

# 這樣一來，只有指定的員工才能登入。
- **Okta 用戶端密鑰預設不會過期** — 但為符合安全最佳實踐，您可以隨時從應用程式的 General 索引標籤中輪替這些密鑰。
- **使用非管理員帳戶進行測試** — 使用一個普通 Okta 用戶（而非超級管理員）並指派給應用程式，以驗證 SSO 是否如預期般運作。
- **Okta 中的多重驗證 (MFA)** — 設定 Okta 的全域會話策略或驗證策略，以要求多重驗證。

這將套用於所有對 Spwig 的 SSO 登入，無需在 Spwig 中另行設定 MFA。
