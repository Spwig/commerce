---
title: 了解搜尋設定
---

SearchSettings 介面控制您 Spwig 商店中所有全域搜尋行為。此單一設定頁面使用 8 索引標籤介面來組織搜尋選項，從基本啟用到高階效能調整。除非在引擎層級覆寫，否則這裡的更改會套用到所有搜尋引擎。

本指南將逐步介紹每個標籤，說明每個設定的作用以及何時進行調整。

![Search Settings General Tab](/static/core/admin/img/help/search-settings-overview/search-settings-general.webp)

## 8-Tab 介面

SearchSettings 是一個單例模型 - 您的整個商店只有一個設定記錄 (pk=1)。介面分為八個標籤：

| 標籤 | 目的 |
|-----|---------|
| **一般** | 啟用/停用搜尋，設定基本參數 |
| **自動完成** | 設定預測搜尋下拉選單行為 |
| **內容類型** | 選擇可搜尋的內容類型 |
| **深度索引** | 控制哪些產品資料會被索引（效能影響） |
| **模糊匹配** | 輸入錯誤容忍度和相似度閾值 |
| **權重** | 排序結果的相關性乘數 |
| **快取** | 回應時間與新鮮度的權衡 |
| **分析** | 查詢追蹤和隱私設定 |

每個標籤專注於搜尋設定的特定方面。

## 一般標籤

一般標籤包含影響所有搜尋的核心設定：

**啟用搜尋** - 搜尋系統的主控開關。停用時，商店內所有搜尋功能（包括自動完成和搜尋結果頁面）都將處於非活動狀態。

**最小查詢長度** - 預設：2 個字元。短於此長度的搜尋將被拒絕。設為 1 可允許單字元搜尋（例如 "A"），但會增加伺服器負載。

**每頁結果數** - 預設：20 項。控制搜尋結果頁面的分頁。較高的值（30-50）可減少分頁點擊次數，但會增加頁面載入時間。

## 內容類型標籤

![Content Types Settings](/static/core/admin/img/help/search-settings-overview/search-settings-content-types.webp)

切換哪些內容類型會出現在搜尋結果中：

- **產品** - 實體、數位和訂閱產品
- **分類** - 產品分類
- **品牌** - 產品品牌
- **部落格文章** - 部落格內容

**效能提示**：較少的內容類型 = 更快的搜尋。每個啟用的類型都會增加額外的資料庫查詢。如果您沒有部落格，請停用部落格文章以提高回應時間。

## 深度索引標籤

⚠️ **效能警告** - 這些設定會對效能產生重大影響。

![Deep Indexing Settings](/static/core/admin/img/help/search-settings-overview/search-settings-deep-indexing.webp)

深度索引控制哪些與產品相關的資料會被包含在搜尋中：

**索引 SKU** - 預設：開啟，低影響。將產品和變體 SKU 包含在搜尋中。對於客戶透過產品編碼進行搜尋的 B2B 商店來說，這非常重要。

**索引屬性** - 預設：開啟，中等影響。將產品屬性（顏色、尺寸、材質）包含在搜尋中。會加入屬性表的 JOIN。對於時尚和可配置產品非常重要。

**索引自定義欄位** - 預設：開啟，中等影響。將商家定義的自定義欄位包含在搜尋結果中。需要 JSONField 遍歷。

**索引評論** - 預設：開啟，中等-高影響。將已批准的評論標題和評論包含在搜尋中。會加入評論表並增加文字搜尋負荷。對於評論密集型目錄很有幫助。

**索引文件** - 預設：關閉，**極高影響** ⚠️

文件索引會從附加到數位產品的 PDF、DOCX 和 XLSX 檔案中提取文字。此功能：

- 需要非常昂貴的初始索引
- 在每次搜尋時增加顯著的查詢負荷
- 可能導致大型文件超時
- **僅在有可搜尋文件的數位產品商店中啟用**
- **切勿隨意啟用** - 首先徹底測試效能影響

## 模糊匹配標籤

![Fuzzy Matching Settings](/static/core/admin/img/help/search-settings-overview/search-settings-fuzzy-matching.webp)

模糊匹配使用 Levenshtein 距離來處理拼寫錯誤：

**啟用模糊匹配** - 允許搜尋匹配相似的詞（例如 "laptop" 匹配 "labtop"）

**相似度閾值** - 預設：0.80（80% 相似度）。範圍：0.0-1.0。較高的值需要更接近的匹配，運行速度更快。較低的值可以捕捉更多拼寫錯誤，但可能會返回不相關的結果。

**最大編輯距離** - 預設：2 個字元更改。允許的最大插入、刪除或替換數量。較低的值（1）可以提高效能，但捕捉的拼寫錯誤較少。

## 權重標籤

權重控制相關性評分 - 結果的排序方式。權重標籤顯示每個可搜尋欄位的預設乘數：

- weight_name: 1.50（產品名稱最重要）
- weight_sku: 1.20
- weight_description: 0.80
- weight_categories: 0.80
- weight_attributes: 0.70
- weight_brands: 0.70
- weight_blog_posts: 0.60
- weight_reviews: 0.50

這些預設值對大多數電商商店都很有效。如需詳細資訊關於調整權重和理解其影響，請參閱 [相關性權重和深度索引](/en/admin/help/relevance-weights-deep-indexing/) 主題。

## 快取標籤

![Caching Settings](/static/core/admin/img/help/search-settings-overview/search-settings-caching.webp)

快取通過儲存最近的結果來大幅提升搜尋效能：

**自動完成快取 TTL** - 預設：60 秒。自動完成結果的快取時間。較短的 TTL（30-45 秒）= 更新鮮的結果，但更多資料庫查詢。較長的 TTL（90-120 秒）= 更快，但可能結果較舊。

**結果快取 TTL** - 預設：300 秒（5 分鐘）。完整搜尋結果頁面快取持續時間。較長的 TTL 显著提升效能，但會延遲新產品的可見性。

**權衡**：快取是效能優化的最有效方法。如果搜尋速度慢，請先增加這些值，而不是停用功能。

## 分析標籤

![Analytics Settings](/static/core/admin/img/help/search-settings-overview/search-settings-analytics.webp)

**追蹤搜尋查詢** - 啟用搜尋分析儀表板。記錄查詢文字、結果數量、回應時間和時間戳。

**追蹤使用者資訊** - 將搜尋與已登入的使用者關聯。為符合隱私法規（如 GDPR、CCPA）請停用。

**追蹤會話資訊** - 使用會話 ID 來追蹤匿名使用者的搜尋。在不使用個人資料的情況下識別搜尋模式很有用。

## 單例模式

SearchSettings 使用單例模式 - 您的資料庫中只有一個設定記錄 (pk=1)。當您在管理介面導航到搜尋設定時，您始終在編輯相同的記錄。

沒有 "新增" 或 "刪除" 選項 - 只有 "更改"。所有搜尋引擎都會繼承這些設定，除非它們指定了每個引擎的覆寫（很少見）。

## 小技巧

- **除非有特定需求，否則保留預設值** - 預設設定已針對典型電商商店進行優化
- **切勿隨意啟用文件索引** - 僅針對有可搜尋文件的數位產品商店，並首先測試效能影響
- **在分析中監控回應時間** - 目標是自動完成 <200ms，完整搜尋 <500ms
- **如果效能較慢，請增加快取 TTL** - 快取是最簡單的效能優化
- **每週審查零結果查詢** - 它們會顯示缺少的產品或需要的同義詞
- **停用未使用的內容類型** - 如果您沒有部落格，請關閉部落格文章以加快搜尋速度
