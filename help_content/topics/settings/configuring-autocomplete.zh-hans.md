---
title: 配置自动补全
---

自动补全，也称为预测搜索或边输入边搜索，在客户输入查询时显示结果。这通过帮助客户更快地找到产品并减少无结果搜索，显著改善了用户体验。本指南将解释如何配置自动补全行为、显示设置和性能权衡。

默认情况下，自动补全已启用并带有合理的设置。只有在有特定的性能问题或显示偏好时才进行调整。

![自动补全设置](/static/core/admin/img/help/configuring-autocomplete/autocomplete-settings-main.webp)

## 启用自动补全

导航到 **搜索 > 搜索设置** 并点击 **自动补全** 选项卡。

**启用自动补全** - 预测搜索的主控开关。启用后，搜索输入框会在客户输入时显示结果下拉列表。

**每种类型的最大结果数** - 默认：8 项。每种内容类型（产品、分类、品牌、博客文章）显示的结果数量。较低的值（5-6）可减少 API 数据负载并加快渲染速度。较高的值（10-12）可为客户提供更多选项，但会减慢响应速度。

## 延迟时间设置

⚠️ **性能警告** - 延迟时间显著影响服务器负载。

**延迟时间** - 默认：300 毫秒。在最后一次按键后等待多长时间再触发自动补全请求。

此设置在响应速度与服务器负载之间取得平衡：

| 延迟 | 用户体验 | 服务器影响 |
|-------|----------------|---------------|
| **100 毫秒** | 非常响应迅速 | 比 300 毫秒多 3 倍的 API 调用 - 高负载 |
| **200 毫秒** | 响应迅速 | 比 300 毫秒多 1.5 倍的 API 调用 |
| **300 毫秒** | 良好平衡（推荐） | 基准 |
| **400 毫秒** | 稍显迟缓 | 更少的 API 调用 - 负载较低 |
| **500 毫秒** | 明显延迟 | 调用减少 50%，但感觉缓慢 |

**建议**：保持在 250-350 毫秒之间。只有在服务器因自动补全负载而挣扎时，才将延迟时间提高到 350 毫秒以上。除非您有非常快速的服务器和小型目录，否则不要低于 200 毫秒。

## 产品的显示设置

这些切换控制产品自动补全结果中显示的信息：

**显示缩略图** - 默认：开启。在结果旁边显示产品图片。**性能影响**：增加图片查询并增大 JSON 数据负载。在连接较慢的情况下，关闭此功能可加快自动补全速度。

**显示描述** - 默认：关闭。显示产品的简短描述。**性能影响**：增加文本处理并显著增大数据负载。除非描述对产品选择至关重要，否则建议保持关闭。

**显示价格** - 默认：开启。显示产品价格。**性能影响**：低 - 价格数据已随产品加载。可以安全地保持开启。

**显示SKU** - 默认：开启。显示产品SKU。**性能影响**：低 - SKU已建立索引。对于B2B商店至关重要。

**显示库存状态** - 默认：关闭。**⚠️ 重大性能警告**

显示“有库存”、“库存不足”或“无库存”的徽章。**在大型目录中千万不要启用此功能**。

库存状态需要 `with_stock_totals()` 聚合 - 计算每个自动补全结果中所有仓库的产品在库数量。这会增加：
- 显著的数据库负载（聚合查询）
- 目录超过1,000个产品时，额外200-500毫秒的延迟
- 目录超过10,000个产品时，可能出现超时

只有在绝对必要且产品数量少于500个时才启用。

## 博客文章的显示设置

**显示特色图片** - 默认：开启。在自动补全结果中显示博客文章的缩略图。

**显示摘要** - 默认：开启。从文章内容中提取的简要预览文本。

**摘要长度** - 默认：60个字符。显示多少预览文本。

由于博客文章的数量通常比产品少，因此这些设置对性能影响很小。

## 分类和品牌的显示设置

**显示缩略图/Logo** - 默认：开启。在结果中显示分类或品牌的图片。

**显示产品数量** - 默认：关闭。**⚠️ 性能警告**

显示每个分类或品牌中的产品数量（例如，“电子产品（234）”）。

**在大型目录中千万不要启用此功能**。产品数量会在每次自动补全请求时重新计算：
- 每个启用了数量统计的内容类型会增加2个额外查询
- 查询包括连接和聚合
- 通常增加100-300毫秒的延迟
- 随着分类/品牌数量的增加，延迟呈线性增长

只有在分类/品牌数量少于50个且总产品数量少于1,000个时才启用。

## 缓存

**自动补全缓存TTL** - 默认：60秒（在缓存选项卡中设置）。

自动补全结果被缓存以提高性能。60秒的TTL意味着：
- 第一个搜索“笔记本电脑”的客户会触发数据库查询
- 接下来的59秒内，所有“笔记本电脑”搜索都会返回缓存结果
- 60秒后，缓存过期，下一次搜索会刷新数据

**TTL建议**：
- **45-60秒**：对大多数商店来说，平衡良好（默认）
- **90-120秒**：如果产品库存很少变化，性能更好
- **30秒**：如果您经常添加产品，结果会更新得更快

增加缓存TTL是提高自动补全性能最简单的方法。

## 多语言自动补全

如果您配置了多种语言，自动补全会自动搜索存储在JSONField翻译中的翻译内容。

**工作原理**：
- 客户用西班牙语搜索：“zapatos”
- 系统搜索西班牙语产品名称翻译
- 结果显示来自JSONField数据的西班牙语产品名称
- 如果西班牙语翻译缺失，则回退到基础语言

**性能**：1-3种语言的额外开销很小。5种以上语言会略微增加查询复杂度。

## 测试自动补全

配置设置后，测试自动补全体验：

1. **在无痕窗口中打开您的商店首页**
2. **点击搜索框** 以聚焦它
3. **缓慢输入一个常见的产品名称**（例如，“laptop”）
4. **观察**：
   - 在您停止输入后结果出现的速度（延迟是否正常？）
   - 显示的信息（缩略图、价格、SKU是否按配置显示）
   - 结果是否相关（如果没有，请检查相关性权重）
5. **测试移动设备** - 确保下拉菜单在触摸屏上易于点击和阅读

## 小贴士

- **为了速度关闭产品描述** - 描述在自动补全上下文中显著增加数据负载，但价值不大
- **在大型目录中千万不要启用库存状态** - 库存聚合会严重影响自动补全性能
- **使用触摸目标测试移动设备** - 自动补全结果必须在手机上容易点击
- **每周监控响应时间** - 自动补全请求的目标时间小于200毫秒
- **如果速度慢，增加缓存TTL** - 这是最简单的性能优化方法
- **产品数量统计代价高昂 - 除非必要否则关闭** - 每个分类/品牌数量统计会增加每个自动补全请求的2个查询
