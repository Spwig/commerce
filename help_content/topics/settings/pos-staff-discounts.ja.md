---
title: POSスタッフ割引と端末セキュリティ
---

POSスタッフ割引設定は、各スタッフが売上時に適用できる割引額を制御するためのものです。端末ロックイベントは、端末がロックまたはアンロックされたたびに監査トレースを提供し、端末にアクセスした人物や、ログイン試行の失敗が発生したかどうかを追跡するのに役立ちます。

## スタッフ割引上限

POSを使用する各スタッフは、個別の割引権限を持つことができます。デフォルトでは、スタッフは商品またはカート全体に対して最大10％の割引を適用できます。この上限を個人ごとに上げたり下げたりし、または標準的な上限を超える割引を承認できる管理者としてスタッフを指定することもできます。

### スタッフの割引上限を設定する

1. **POS > スタッフ割引**に移動します
2. **+ POSスタッフ割引の追加**をクリックするか、既存のスタッフを編集するためにクリックします
3. リストから**スタッフメンバー**を選択します
4. 割引上限を設定します：

| フィールド | 説明 |
|-------|-------------|
| **最大割引％** | この人が適用できる最大の割引率（例：`10`で10％） |
| **最大割引金額** | 1回の取引あたりの最大固定金額（固定上限なしの場合は空白にします） |
| **個別商品割引を適用可能** | 個別の商品項目に割引を適用できるようにする |
| **カート全体割引を適用可能** | カート全体の合計に割引を適用できるようにする |
| **理由が必要** | これをチェックすると、スタッフは割引を適用する前に理由を入力する必要があります |

5. **保存**をクリックします

### POSでの割引上限の動作方法

レジ係が割引を適用しようとした場合：
- 割引がその人の上限内にある場合、すぐに適用されます
- 割引がその人の上限を超える場合、端末は**管理者の承認**を求めるプロンプトを表示します
- 管理者は自分のPINを入力して上書きを承認し、割引が適用されます

このワークフローは、正当な割引が必要な場合に柔軟性を保ちながら、不正な高額割引を防止します。

## 管理者ロール

**Is Manager**フラグを持つスタッフは、他のスタッフの上限を超える割引を承認できます。

管理者は、承認が要求されたときに端末でPINを入力することで識別されます。

### 管理者の設定

1. スタッフメンバーの割引レコードを開きます
2. **管理者である**を確認します
3. **管理者PIN**（4〜6桁）を入力します — これは保存時にセキュアにハッシュ化されます
4. **保存**をクリックします

管理者PINは、端末のロック/アンロックに使用されるキャッシャーPINとは別です。管理者は、割引承認用の管理者PINと、端末アクセス用のキャッシャーPINの両方を持つことができます。

### 管理者PINのセキュリティ

管理者フォームでPINを入力して保存すると、Spwigは自動的にそれをハッシュ化します — プレーンなPINは決して保存されません。保存後にプレーンなPINフィールドは空になります。これは予期された動作です。

## キャッシャーPINとカードアクセス

各スタッフメンバーは、端末のロックとアンロックに使用できる**キャッシャーPIN**も持つことができます:

- **キャッシャーPIN** — 端末が自動的にロックされるか、手動でロックされた後に端末をアンロックするために使用される4〜6桁のPIN
- **カード識別子** — 登録されたカード（スワイプカードまたはNFC）も、端末をアンロックするために使用できます

キャッシャーPINを設定するには、**キャッシャーPIN**フィールドにPINを入力して保存します。管理者PINと同様に、保存時に自動的にハッシュ化されます。

## 端末ロックイベント

端末がロックまたはアンロックされるたびに、Spwigは端末ロックイベントを記録します。これにより、完全なセキュリティ監査トレースが作成されます。

### ロックイベントの表示

**POS > 端末ロックイベント**にナビゲートして、履歴全体を確認できます。イベントを次の項目でフィルタリングできます:
- 端末
- イベントタイプ
- 日付範囲

### イベントタイプ

| イベント | 意味 |
|-------|---------|
| **手動ロック** | スタッフが意図的に端末をロックしました |
| **自動ロック（アイドルタイムアウト）** | 空き時間が原因で端末が自動的にロックされました |
| **レジ係によるロック解除** | レジ係が認証し、端末をロック解除しました |
| **管理者によるロック解除** | 管理者が自分の資格情報を使用してロック解除しました |
| **カードによるロック解除** | 登録されたスワイプカードを使用して端末がロック解除されました |
| **生体認証によるロック解除** | 指紋または顔認証を使用して端末がロック解除されました |
| **ロック解除の失敗** | 間違った資格情報を使用してロック解除の試みがありました |
| **ロックアウト（3回以上の失敗）** | 複数回の失敗の後、端末がロックアウトされました |

### ロックイベントの記録に含まれるもの

各イベントは以下を記録します：
- 涉及された **端末**
- **イベントタイプ**
- アクションを実行した人物 (**Performed By**) およびロックが発生した際のログインユーザー (**Locked By**)
- **管理者による上書き**が使用されたかどうか
- **ロック解除方法**（PIN、カード、または生体認証）
- このイベント前の **失敗試行**（ブルートフォースパターンの検出に役立ちます）
- イベント発生時の **カート合計**とアイテム数
- リクエストのIPアドレス

### セキュリティ上の懸念の調査

端末への不正アクセスが疑われる場合：

1. **POS > Terminal Lock Events** に移動します
2. 質問の端末でフィルタを適用します
3. **Failed Unlock Attempt** または **Lockout** のタイプのイベントを確認します — これらは繰り返しの失敗アクセスを示しています
4. 成功したロック解除の **Performed By** フィールドを確認して、アクセスを取得した人物を確認します
5. **POS > Shifts** と照合して、当直予定のレジ係を確認します

## ヒント

- スタッフの経験レベルに基づいて割引の上限を設定します — 新人スタッフは5%から始め、経験豊富なスタッフは10-15%、管理者はさらに高い割引を承認できます。
- 高い割引上限を持つスタッフに **Requires Reason** を有効にします。

# ファイルに保存された理由があると、割引のパターンを分析し、不正使用を特定しやすくなります。
- 店舗に複数のスタッフやスタッフの交代が頻繁にある場合は、週次で端末ロックイベントを確認してください — 不規則なアクセスパターンは、問題になる前に簡単に特定できます。
- スタッフが退職した場合は、すぐにレジのPINとカード識別子を削除し、端末へのアクセスを防止してください。
- ロックアウトイベントを使用して、必要に応じて自動ロックタイムアウトを調整する必要がある端末を特定してください — 顧客が頻繁に誤ってロックアウトを引き起こしている場合、アイドルタイムアウトが短すぎる可能性があります。
- マネージャーのPINは定期的に変更する必要があります。

スタッフの割引記録で更新してください — 新しいPINは保存時にハッシュ化されます。
