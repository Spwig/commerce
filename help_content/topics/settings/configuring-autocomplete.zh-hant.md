---
title: 設定自動完成
---

自動完成，也稱為預測搜索或邊輸入邊搜索，在顧客輸入查詢時顯示結果。這大大改善了用戶體驗，幫助顧客更快找到產品，並減少無結果的搜索。本指南說明如何配置自動完成行為、顯示設置和性能權衡。

默認情況下，自動完成已啟用，並具有合理的設置。只有在有特定的性能問題或顯示偏好時才需要調整這些設置。

![自動完成設置](/static/core/admin/img/help/configuring-autocomplete/autocomplete-settings-main.webp)

## 啟用自動完成

導航到 **Search > Search Settings** 並點擊 **Autocomplete** 紙介。

**Enable Autocomplete** - 預測搜索的主控開關。啟用後，搜索輸入框會在顧客輸入時顯示結果下拉選單。

**Max Results Per Type** - 默認值：8 項。每個內容類型（產品、分類、品牌、博客文章）顯示的結果數量。較低的值（5-6）可減少 API 數據負載大小並加快渲染速度。較高的值（10-12）會給顧客更多選擇，但會減慢響應速度。

## 延遲時間

⚠️ **性能警告** - 延遲時間會顯著影響服務器負載。

**Debounce Delay** - 默認值：300ms。在最後一次鍵盤輸入後等待多長時間才觸發自動完成請求。

此設置在響應速度與服務器負載之間取得平衡：

| 延遲 | 用戶體驗 | 服務器影響 |
|-------|----------------|---------------|
| **100ms** | 非常響應 | 比 300ms 多 3 倍的 API 調用 - 高負載 |
| **200ms** | 響應 | 比 300ms 多 1.5 倍的 API 調用 |
| **300ms** | 良好平衡（推薦） | 基準 |
| **400ms** | 稍微遲鈍 | 更少的 API 調用 - 降低負載 |
| **500ms** | 明顯延遲 | 比 300ms 少 50% 的調用但感覺較慢 |

**建議**：保持在 250-350ms 之間。只有在服務器因自動完成負載而困難時才增加到 350ms 以上。除非您有非常快的服務器和小規模的目錄，否則不要低於 200ms。

## 產品顯示設置

這些開關控制產品自動完成結果中顯示的資訊：

**Show Thumbnail** - 默認值：ON。在結果旁顯示產品圖片。**性能影響**：增加圖片查詢和 JSON 數據負載大小。在連接較慢的情況下，禁用以加快自動完成速度。

**Show Description** - 默認值：OFF。顯示產品簡短描述。**性能影響**：增加文本處理和顯著增加數據負載大小。除非描述對產品選擇至關重要，否則請保持禁用。

**Show Price** - 默認值：ON。顯示產品價格。**性能影響**：低 - 價格數據已經與產品一起加載。可以安全地保持啟用。

**Show SKU** - 默認值：ON。顯示產品 SKU。**性能影響**：低 - SKU 已經被索引。對 B2B 商店至關重要。

**Show Stock Status** - 默認值：OFF。**⚠️ 重大性能警告**

顯示 "In Stock"、"Low Stock" 或 "Out of Stock" 的徽章。**在大型目錄中永遠不要啟用此功能**。

庫存狀態需要 `with_stock_totals()` 聚合 - 在所有倉庫中計算每個自動完成結果產品的庫存數量。這會增加：
- 显著的數據庫負載（聚合查詢）
- 在目錄數量超過 1,000 的情況下，增加 200-500ms 的延遲
- 在目錄數量超過 10,000 的情況下，可能導致超時

只有在絕對必要且產品數量少於 500 時才啟用。

## 博客文章顯示設置

**Show Featured Image** - 默認值：ON。在自動完成結果中顯示博客文章的特色圖片。

**Show Excerpt** - 默認值：ON。從文章內容中顯示簡短預覽文字。

**Excerpt Length** - 默認值：60 個字符。顯示多少預覽文字。

這些設置對性能的影響很小，因為與產品相比，博客文章通常較少。

## 分類和品牌顯示設置

**Show Thumbnail/Logo** - 默認值：ON。在結果中顯示分類或品牌的圖片/標誌。

**Show Product Count** - 默認值：OFF。**⚠️ 性能警告**

顯示每個分類或品牌中的產品數量（例如，"Electronics (234)"）。

**在大型目錄中永遠不要啟用此功能**。產品數量在每次自動完成請求時都會重新計算：
- 每個啟用數量的內容類型會增加 2 個額外查詢
- 查詢包括連接和聚合
- 通常增加 100-300ms 的延遲
- 隨著分類/品牌數量的增加而線性增加

只有在分類/品牌數量少於 50 個且總產品數少於 1,000 個時才啟用。

## 缓存

**Autocomplete Cache TTL** - 默認值：60 秒（在緩存標籤中設置）。

自動完成結果被緩存以提高性能。60 秒的 TTL 意味著：
- 第一位搜索 "laptop" 的顧客觸發數據庫查詢
- 接下來的 59 秒內，所有 "laptop" 搜索返回緩存結果
- 60 秒後，緩存過期，下一次搜索會刷新數據

**TTL 建議**：
- **45-60s**：大多數商店的良好平衡（默認）
- **90-120s**：如果產品庫存很少變化，性能更好
- **30s**：如果您經常添加產品，結果會更新更及時

增加緩存 TTL 是提高自動完成性能最簡單的方法。

## 多語言自動完成

如果您配置了多種語言，自動完成會自動搜索存儲在 JSONField 翻譯中的翻譯內容。

**工作原理**：
- 顧客以西班牙語搜索："zapatos"
- 系統搜索西班牙語產品名稱翻譯
- 結果顯示來自 JSONField 數據的西班牙語產品名稱
- 如果西班牙語翻譯缺失，則回退到基本語言

**性能**：1-3 種語言的額外開銷很小。5 種語言以上會略微增加查詢複雜度。

## 測試自動完成

配置設置後，測試自動完成體驗：

1. **在無痕窗口中打開您的商店首頁**
2. **點擊搜索框** 以聚焦它
3. **慢慢輸入一個常見產品名稱**（例如，"laptop"）
4. **觀察**：
   - 在您停止輸入後結果出現的速度（延遲是否正常？）
   - 顯示的資訊（根據設定顯示縮略圖、價格、SKU）
   - 結果是否相關（如果沒有，請檢查相關性權重）
5. **測試行動裝置** - 確保下拉選單在手機上易於點擊且易於閱讀

## 小技巧

- **為提高速度禁用產品描述** - 描述會顯著增加數據負載大小，但在自動完成上下文中價值不大
- **在大型目錄中永遠不要啟用庫存狀態** - 庫存聚合會導致自動完成性能下降
- **使用觸控目標在行動裝置上測試** - 自動完成結果必須在手機上容易點擊
- **每周監控響應時間** - 自動完成請求目標小於 200ms
- **如果速度慢，增加緩存 TTL** - 最簡單的性能優化
- **產品數量成本很高 - 非必要請禁用** - 每個分類/品牌數量會增加每次自動完成請求的 2 個查詢
