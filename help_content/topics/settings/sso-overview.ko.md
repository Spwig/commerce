---
title: 관리자 단일 로그인 (SSO)
---

단일 로그인 (SSO)은 직원들이 조직의 인증 제공자(Identity Provider)를 사용하여 관리자 패널에 로그인하도록 해줍니다. 이는 별도의 사용자 이름과 비밀번호를 사용하는 대신에 사용할 수 있습니다. Spwig은 OpenID Connect (OIDC) 프로토콜을 사용하는 모든 인증 제공자를 지원합니다. 이는 Microsoft Entra ID, Google Workspace, Okta, Auth0, Keycloak 및 기타 제공자도 포함합니다.

## 기업용 SSO란 무엇인가요?

기업용 SSO는 소셜 로그인(개인 Google 또는 Facebook 계정으로 로그인)과 다릅니다. 기업용 SSO를 사용하면 다음과 같은 이점이 있습니다:

- 직원들은 **조직의 인증 제공자**를 통해 인증합니다. 이는 이메일, 내부 도구 및 기타 비즈니스 애플리케이션에서 사용하는 동일한 시스템입니다.
- IT 팀은 중앙에서 액세스를 제어할 수 있습니다. 조직을 떠나는 직원의 경우, 인증 제공자에서 계정을 비활성화하면 즉시 Spwig 액세스가 취소됩니다.
- 인증 제공자가 다중 인증(MFA)을 강제로 수행하여 모든 애플리케이션에 대해 일관된 보안 정책을 제공합니다.
- 직원들은 Spwig에 대한 별도의 비밀번호를 기억할 필요가 없습니다.

## 작동 방식

SSO가 활성화되면 관리자 로그인 페이지에 **[Provider]로 로그인** 버튼이 표시됩니다. 인증 흐름은 다음과 같습니다:

1. 직원이 Spwig 로그인 페이지의 SSO 버튼을 클릭합니다.
2. 그들은 조직의 인증 제공자 로그인 페이지로 리디렉션됩니다 (예: Microsoft 로그인).
3. 인증 제공자와 인증을 수행합니다 (제공자가 요구하는 MFA 포함).
4. 인증 제공자는 보안 인가 코드와 함께 Spwig으로 다시 리디렉션됩니다.
5. Spwig은 코드를 사용하여 사용자 정보를 가져오고 세션을 생성합니다.
6. 직원은 관리자 대시보드에 완전히 인증된 상태로 도착합니다.

이것은 산업 표준 **OpenID Connect (OIDC)** 프로토콜을 사용하며, 거의 모든 기업 인증 제공자가 지원합니다.

## SSO 활성화

SSO는 두 곳에서 구성됩니다:

1. **사이트 설정 > 보안 탭** — SSO를 활성화하거나 비활성화하고 비밀번호 로그인의 가시성을 제어합니다.
2. **SSO 제공자 구성** — 조직의 인증 제공자의 OIDC 세부 정보를 입력합니다.

### 단계 1: 인증 제공자 구성

Spwig에서 SSO를 활성화하기 전에, Spwig을 조직의 인증 제공자에 애플리케이션으로 등록해야 합니다. 제공자별 가이드를 참조하십시오:

- **Microsoft Entra ID** — Microsoft Entra ID 설정 가이드를 참조하십시오.
- **Google Workspace** — Google Workspace 설정 가이드를 참조하십시오.
- **Okta** — Okta 설정 가이드를 참조하십시오.
- **기타 제공자** — OIDC 호환 제공자는 모두 작동합니다. 리디렉션 URI `https://your-store.com/oidc/callback/`을 사용하여 웹 애플리케이션을 등록하고, 제공자의 문서에서 OIDC 발견 URL, 클라이언트 ID 및 클라이언트 비밀을 확인하십시오.

### 단계 2: Spwig에서 SSO 제공자 구성

**SSO 제공자 구성** 페이지로 이동하십시오 (보안 탭에서 링크된 페이지 또는 관리자 사이드바에서 **Enterprise SSO > SSO 제공자 구성**으로 접근 가능). 다음을 입력하십시오:

1. **제공자 이름** — 로그인 버튼에 표시됩니다 (예: "Microsoft Entra ID")
2. **OIDC 발견 URL** — 제공자의 `.well-known/openid-configuration` URL입니다. **자동 발견**을 클릭하여 끝점 필드를 자동으로 채울 수 있습니다.
3. **클라이언트 ID** 및 **클라이언트 비밀** — 제공자의 애플리케이션 등록에서 가져옵니다.

클라이언트 비밀은 암호화되어 저장되며, 저장 후에는 표시되지 않습니다.

### 단계 3: 사이트 설정에서 SSO 활성화

**사이트 설정 > 보안** 탭으로 이동하고 **관리자 로그인용 SSO 활성화**를 선택합니다. SSO 버튼은 즉시 관리자 로그인 페이지에 나타납니다.

## SSO 설정

| 설정 | 설명 |
|---------|-------------|
| **관리자 로그인용 SSO 활성화** | 관리자 로그인 페이지에 SSO 버튼을 표시합니다. 비밀번호 로그인을 비활성화하지 않는 한 일반 비밀번호 로그인에 영향을 주지 않습니다. |
| **관리자 페이지에서 비밀번호 로그인 허용** | 비활성화되면 비밀번호 폼은 접기 가능한 토글 뒤에 숨겨집니다. 직원들은 기본적으로 SSO 버튼만 보입니다. 비밀번호 폼은 "로컬 계정으로 로그인"을 클릭하거나 로그인 URL에 `?password=1`을 추가하여 여전히 접근할 수 있습니다. |

### 로그인 페이지 행동

| SSO 활성화 | 비밀번호 로그인 | 결과 |
|-------------|---------------|--------|
| Off | On | 사용자 이름/비밀번호 폼만 있는 표준 로그인 페이지 |
| On | On | 상단에 SSO 버튼, "or" 구분선, 그 아래에 비밀번호 폼 |
| On | Off | SSO 버튼만. 비밀번호 폼은 "로컬 계정으로 로그인" 토글 뒤에 있습니다 |
| Off | Off | 불가능 — SSO가 비활성화되거나 설정되지 않은 경우 비밀번호 로그인은 자동으로 다시 활성화됩니다 |

## 사용자 일치

직원이 SSO를 통해 로그인할 때, Spwig는 **이메일 주소**로 기존 사용자 계정과 일치시킵니다(대소문자 구분 없음). 정체성 제공자의 클레임에서 온 이메일은 직원의 Spwig 계정에 있는 이메일과 일치해야 합니다.

일치하는 사용자가 없으면:

- **자동 생성 사용자 비활성화**(기본값) — 로그인은 거부됩니다. 먼저 Spwig에서 일치하는 이메일 주소로 직원 계정을 생성해야 합니다.
- **자동 생성 사용자 활성화** — 정체성 제공자의 클레임에서 이름과 이메일을 사용하여 새로운 사용자 계정이 자동으로 생성됩니다.

**직원에게 제한**(기본값으로 활성화됨) 설정은 추가적인 확인을 추가합니다: 사용자 계정이 존재하더라도 사용자가 직원 상태가 아니면 로그인은 거부됩니다. 이는 비직원 계정이 SSO를 통해 관리 패널에 접근하는 것을 방지합니다.

## 역할 매핑

정체성 제공자가 OIDC 클레임에 그룹 소속 정보를 전송하는 경우, Spwig는 그룹 소속에 따라 직원 및 슈퍼유저 상태를 자동으로 설정할 수 있습니다.

역할 매핑을 구성하려면:

1. SSO 제공자 구성에서 **그룹 클레임**(Groups Claim) 필드를 제공자가 사용하는 클레임 이름으로 설정합니다(기본값: `groups`)
2. **직원 그룹**(Staff Groups)에 쉼표로 구분된 그룹 이름 또는 ID를 입력합니다. 이 그룹에 속한 사용자는 직원 상태가 부여됩니다.
3. **슈퍼유저 그룹**(Superuser Groups)에 쉼표로 구분된 그룹 이름 또는 ID를 입력합니다. 이 그룹에 속한 사용자는 슈퍼유저 상태가 부여됩니다.

역할 매핑은 사용자가 SSO를 통해 로그인할 때마다 평가됩니다. 정체성 제공자에서 사용자가 그룹에서 제거되면 다음 SSO 로그인 시 해당 사용자의 직원 또는 슈퍼유저 상태가 업데이트됩니다.

**중요:** Microsoft Entra ID는 기본적으로 그룹 **오브젝트 ID**(UUID)를 전송하며, 그룹 이름은 전송하지 않습니다. 역할 매핑을 구성할 때 Azure 포털에서 오브젝트 ID를 복사합니다. Okta와 같은 다른 제공자는 일반적으로 그룹 이름을 전송합니다.

## 클레임 매핑

Spwig는 표준 OIDC 클레임에서 사용자 정보를 읽습니다. 기본값은 대부분의 제공자와 호환되지만, SSO 제공자 구성에서 클레임 필드 이름을 사용자 정의할 수 있습니다:

| 설정 | 기본값 | 설명 |
|---------|---------|-------------|
| **이메일 클레임** | `email` | 사용자의 이메일 주소를 포함하는 클레임 |
| **이름 클레임** | `given_name` | 사용자의 이름을 포함하는 클레임 |
| **성 클레임** | `family_name` | 사용자의 성을 포함하는 클레임 |
| **그룹 클레임** | `groups` | 그룹 소속을 포함하는 클레임(빈 값으로 설정하면 역할 매핑을 비활성화합니다) |

## MFA 행동

직원이 SSO를 통해 로그인할 때, Spwig의 내장 이중 인증(2FA) 요구사항은 자동으로 우회됩니다. 이는 정체성 제공자가 SSO 로그인 흐름의 일부로 MFA를 강제하는 책임이 있기 때문입니다.

조직에서 MFA가 필요한 경우, Spwig의 2FA 설정 대신 정체성 제공자의 조건부 액세스 정책에서 구성해야 합니다. 이는 모든 애플리케이션에 걸쳐 중앙 집중식 MFA 관리를 가능하게 합니다.

## 복구 액세스

정체성 제공자가 장애나 설정 오류를 겪는 경우에도 여전히 관리 로그인 폼에 액세스할 수 있습니다:

- **토글 클릭** — 비밀번호 로그인이 비활성화된 경우, 로그인 페이지에서 "로컬 계정으로 로그인"을 클릭하여 비밀번호 폼을 표시합니다
- **URL 파라미터** — 관리 로그인 URL에 `?password=1`을 추가합니다(예: `https://your-store.com/en/admin/login/?password=1`)하여 비밀번호 폼을 직접 표시합니다
- **비밀번호 로그인은 항상 사용 가능** — UI에서 숨겨진 경우에도 비밀번호 인증 백엔드는 여전히 활성화되어 있습니다. 표시 여부만 영향을 받습니다.

Spwig은 SSO가 활성화되고 올바르게 구성되지 않은 경우 비밀번호 로그인을 비활성화하는 것을 방지합니다. 즉, 실수로 자신을 밖으로 밀어내는 일이 발생하지 않습니다.

## 지원되는 제공자

Spwig은 OpenID Connect (OIDC) 프로토콜을 지원하는 모든 정체성 제공자와 함께 작동합니다. 자세한 설정 가이드는 다음과 같습니다:

- **Microsoft Entra ID** (이전 Azure Active Directory)
- **Google Workspace** (Google Cloud Identity)
- **Okta**

다른 OIDC 호환 제공자 (Auth0, Keycloak, OneLogin, Ping Identity, JumpCloud 등)의 경우 Spwig 구성 단계는 동일합니다. 제공자의 OIDC Discovery URL, 클라이언트 ID 및 클라이언트 비밀을 알아야 합니다. 제공자의 문서를 참조하여 웹 애플리케이션을 등록하고 이러한 자격 증명을 얻는 방법을 확인하세요. 사용해야 하는 리디렉션 URI는 항상 `https://your-store.com/oidc/callback/`입니다.

## 팁

- **비밀번호 로그인을 먼저 활성화하세요** — 비밀번호 로그인과 함께 SSO를 먼저 활성화하세요. 팀에서 SSO가 작동하는 것을 확인한 후 비밀번호 로그인을 선택적으로 비활성화할 수 있습니다.
- **익명 창에서 테스트하세요** — 현재 관리자 세션에 영향을 주지 않고 SSO를 테스트하려면 프라이빗/익명 브라우저 창을 사용하세요.
- **직원 계정을 먼저 생성하세요** — Auto-Create Users를 활성화하지 않는 한, 직원은 기존 Spwig 계정과 일치하는 이메일 주소가 있는 경우에만 SSO를 통해 로그인할 수 있습니다.
- **자동 탐색 버튼을 사용하세요** — 제공자의 OIDC Discovery URL을 입력하고 Auto-Discover를 클릭하여 모든 엔드포인트 필드를 자동으로 채우세요. 이는 엔드포인트를 수동으로 입력하는 것보다 더 빠르고 오류가 적습니다.
- **로컬 관리자 계정을 유지하세요** — 정체성 제공자 문제 발생 시 복구 옵션으로 항상 최소한 하나의 비밀번호가 있는 로컬 관리자 계정을 유지하세요.
- **클라이언트 비밀 만료를 모니터링하세요** — 일부 제공자 (특히 Microsoft Entra ID)는 만료 날짜가 있는 클라이언트 비밀을 발급합니다. 만료되기 전에 비밀을 회전시키는 일정 알림을 설정하세요.
