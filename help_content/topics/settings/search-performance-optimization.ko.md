검색 성능 최적화

검색 성능은 고객 경험과 전환율에 직접적인 영향을 미칩니다. 느린 검색은 고객을 실망시키고 이탈률을 증가시킵니다. 이 종합 가이드는 Spwig의 데이터베이스 네이티브 검색 시스템에서 일반적인 성능 병목 현상을 식별하고, 최적화 전략을 제공하며, 성능 목표를 설정합니다. 검색 응답 시간이 수용 가능한 임계값을 초과하거나 카탈로그 성장 계획이 있는 경우 이 가이드를 사용하세요.

목표 응답 시간: 자동 완성 <200ms, 전체 검색 <500ms. 아래의 최적화 확인 목록을 따르면 이러한 목표를 달성할 수 있습니다.

## 성능 지표 이해

**검색 > 검색 분석**에서 다음 지표를 모니터링하세요:

**응답 시간** - 검색 쿼리 실행에 걸리는 밀리초 (서버 측만, 네트워크 지연 시간 제외)

**캐시 히트율** - 캐시에서 제공된 검색 비율 (데이터베이스 대비)

**쿼리 수** - 검색당 데이터베이스 쿼리 수 (적을수록 좋음)

**데이터베이스 쿼리 시간** - 데이터베이스에서 소요된 시간 대 애플리케이션 코드

## 성능 목표

| 쿼리 유형 | 목표 | 수용 가능 | 최적화 필요 |
|------------|--------|------------|----------------------|
| 자동 완성 | <200ms | 200-300ms | >300ms 일관성 있게 |
| 전체 검색 | <500ms | 500-800ms | >800ms 일관성 있게 |
| 관리자 검색 | <1000ms | 1000-1500ms | >1500ms 일관성 있게 |

평균 응답 시간이 "최적화 필요" 임계값을 초과하면 아래 전략을 실행하세요.

## 성능 모니터링

**분석 대시보드 평균 응답 시간**

**검색 > 검색 분석**으로 이동하여 모든 검색에 대한 평균 응답 시간을 확인하세요. 이는 주요 성능 모니터링 지표입니다.

**조사 시점**: 자동 완성의 평균 응답 시간이 300ms 이상이거나, 전체 검색이 800ms 이상인 경우 여러 날 동안 일관되게 확인하세요.

**주간 모니터링**: 매주 월요일에 분석을 검토하여 성능 저하를 조기에 발견하세요.

## 알려진 성능 병목 현상

Spwig의 데이터베이스 네이티브 검색에는 피해야 할 여러 문서화된 병목 현상이 있습니다:

### CTR 계산 N+1 쿼리

**정의**: 분석 서비스의 클릭-through 비율 계산은 각 집계된 결과 항목에 대해 별도의 쿼리를 실행합니다.

**영향**: 많은 추적 쿼리가 있는 고 트래픽 스토어에 심각한 영향을 줍니다.

**코드 위치**: `search/services/analytics_service.py` - `get_click_through_rate()` 메서드

**해결 방법**: 프로덕션에서 CTR 계산을 호출하지 마세요. 이는 주로 관리자 분석 기능으로, 동기식이 아닌 비동기식으로 계산되어야 합니다.

### 재고 집계

**정의**: `with_stock_totals()`은 모든 창고별로 제품의 on_hand 수량을 계산합니다.

**영향**: 제품 수가 1,000개 이상인 카탈로그에 비용이 많이 들며, `in_stock` 필터가 사용되거나 자동 완성에서 재고 상태가 표시될 때 호출됩니다.

**트리거**: **검색 설정 > 자동 완성** - "재고 상태 표시" 옵션

**추천**: 대규모 카탈로그에서 자동 완성에 재고 상태를 활성화하지 마세요. 요청당 200-500ms 추가됩니다.

### 변형 조인

**정의**: SKU 검색은 변형 테이블에 JOIN을 실행하여 변형 SKU를 검색합니다.

**영향**: 제품당 많은 변형(10개 이상)이 있는 경우 2-3배 느립니다.

**해결 방법**: `.distinct()`를 사용하여 중복을 피하지만, 이는 오버헤드를 추가합니다. SKU 기능이 필요한 경우 비활성화하지 마세요.

### 자동 완성에서의 제품 수

**정의**: 카테고리/브랜드 자동 완성 결과는 제품 수를 표시합니다("전자기기 (234)")

**영향**: 각 콘텐츠 유형에 대해 수를 활성화하면 2개 추가 쿼리가 발생합니다. 쿼리에는 조인 및 집계가 포함됩니다.

**트리거**: **검색 설정 > 자동 완성** - "카테고리/브랜드 제품 수 표시"

**추천**: 제품 수를 비활성화하세요. 자동 완성 요청당 2-4개의 쿼리를 절약합니다. 자동 완성 최적화의 가장 큰 이점입니다.

### 문서 인덱싱

**정의**: 검색 쿼리 중 PDF/DOCX/XLSX 파일에서 텍스트 추출.

**영향**: 매우 비용이 많이 들며 (파일 I/O + 텍스트 추출). 동기식 블로킹 작업.

**트리거**: **검색 설정 > 깊은 인덱싱** - "문서 인덱싱"

**추천**: 성능 비용이 거의 대부분이므로 거의 활성화하지 마세요. 작은 디지털 제품 카탈로그 (<500 제품)에만 활성화하세요. 철저한 테스트 후에.

## 캐시 구성

캐시는 단일로 가장 효과적인 성능 최적화입니다.

**자동 완성 캐시** - 기본값: 60초
- **권장 범위**: 45-90초
- **높은 TTL (90-120초)**: 재고가 거의 변경되지 않는 경우 성능 향상
- **낮은 TTL (30-45초)**: 제품을 시간당 추가하는 경우 더 최신 결과

**결과 캐시** - 기본값: 300초 (5분)
- **권장 범위**: 180-600초
- **높은 TTL (600초/10분)**: 정적 카탈로그에 대해 성능 향상이 커짐
- **낮은 TTL (180초)**: 제품 데이터가 자주 업데이트되는 경우 더 최신

**최적화 전략**: 검색이 느릴 경우 기능을 비활성화하기 전에 캐시 TTL을 두 배로 늘리세요. 자동 완성 캐시를 60초 → 120초로 늘리면 데이터베이스 부하를 반으로 줄입니다.

## 자동 완성 최적화 확인 목록

자동 완성 설정에 다음 변경 사항을 적용하여 최대 성능을 달성하세요:

**1. 디바운스를 300-400ms로 증가**
- 위치: **검색 설정 > 자동 완성** - "디바운스 지연"
- 영향: 키스토로크 사이에 더 오래 기다리므로 API 호출이 줄어듭니다
- 트레이드오프: 약간 덜 반응성이 있음 (대부분의 사용자에게는 인지 불가능)

**2. 최대 결과 수를 8에서 5-6으로 줄이기**
- 위치: **검색 설정 > 자동 완성** - "유형별 최대 결과"
- 영향: 더 작은 결과 집합 = 더 빠른 쿼리 및 더 작은 JSON 페이로드
- 트레이드오프: 표시되는 선택지가 적음 (보통 충분함)

**3. 제품 수 비활성화 (가장 큰 이점)**
- 위치: **검색 설정 > 자동 완성** - "카테고리/브랜드 제품 수 표시" 체크 해제
- 영향: 자동 완성 요청당 2-4개의 쿼리를 절약합니다
- 트레이드오프: 드롭다운에 제품 수 없음 (필요한 경우 드물음)

**4. 재고 상태 비활성화**
- 위치: **검색 설정 > 자동 완성** - "재고 상태 표시" 체크 해제
- 영향: 비용이 많이 드는 재고 집계를 제거합니다
- 트레이드오프: 재고 배지 없음 (자동 완성 맥락에서는 비중요)

**5. 제품 설명 비활성화**
- 위치: **검색 설정 > 자동 완성** - "설명 표시" 체크 해제
- 영향: 텍스트 처리 및 페이로드 크기를 줄입니다
- 트레이드오프: 미리보기 텍스트가 줄어듭니다 (제품 이름이 일반적으로 충분함)

**6. 캐시 TTL을 90초로 증가**
- 위치: **검색 설정 > 캐싱** - "자동 완성 캐시 TTL"
- 영향: 더 많은 요청이 캐시에서 제공됩니다
- 트레이드오프: 결과가 최대 90초 동안 오래된 상태일 수 있음 (대부분의 스토어에 수용 가능)

**예상 개선**: 이 6가지 최적화를 모두 적용하면 자동 완성 응답 시간이 일반적으로 50-70% 감소합니다.

## 깊은 인덱싱 최적화

각 깊은 인덱싱 옵션은 오버헤드를 추가합니다. 카탈로그 크기에 따라 비활성화하세요:

| 카탈로그 크기 | 권장 깊은 인덱싱 |
|--------------|---------------------------|
| **<1,000 제품** | 모두 ON (최소 영향) |
| **1,000-10,000** | SKU, 속성, 커스텀 필드 ON 유지; 리뷰 비활성화 |
| **10,000-20,000** | SKU, 속성 ON 유지; 커스텀 필드, 리뷰 비활성화 |
| **20,000-50,000** | SKU만 ON 유지; 모든 다른 옵션 비활성화 |
| **>50,000** | SKU ON 유지; Elasticsearch 이전을 고려하세요 |

**문서 인덱싱**: 디지털 제품에 검색 가능한 문서가 있고 총 제품 수가 <500인 경우에만 활성화하세요.

## 콘텐츠 유형 최적화

**검색 설정 > 콘텐츠 유형**에서 사용하지 않는 콘텐츠 유형을 비활성화하세요:

- **블로그가 없음?** "블로그 게시물" 비활성화 - 쿼리 절약
- **브랜드 필터링이 필요 없음?** "브랜드" 비활성화 - 쿼리 절약
- **쇼핑 전용 스토어?** "카테고리" 및 "블로그 게시물" 비활성화

각 비활성화된 콘텐츠 유형은 모든 검색에서 데이터베이스 쿼리를 제거합니다.

## 데이터베이스 최적화

Spwig은 마이그레이션을 통해 필요한 인덱스를 생성합니다. 프로파일링 없이 추가 인덱스를 생성하지 마세요.

**PostgreSQL 유지보수** (PostgreSQL 사용 시):
- 주간 `VACUUM ANALYZE` 실행하여 쿼리 계획기 통계 업데이트
- 대규모 카탈로그는 월별 `VACUUM FULL` (다운타임 필요)

**데이터베이스 쿼리 시간 모니터링**: 개발 중에는 프로파일링 도구를 사용하여 느린 쿼리를 식별하세요. 대부분의 쿼리 최적화는 이미 구현되어 있습니다:
- `.select_related('brand', 'category')` 제품에 적용
- `.prefetch_related('images')` 썸네일에 적용
- `.distinct()` 변형 검색에 적용

## 퍼지 매칭 성능

Levenshtein 거리 계산은 계산적으로 비용이 많이 들며 (O(m*n) 복잡도):

**임계값 최적화**:
- **높은 임계값 (0.85 대 0.80)**: 더 빠르지만 오타를 덜 잡음
- **낮은 임계값 (0.75 대 0.80)**: 더 느리지만 더 관대함

**최대 편집 최적화**:
- **낮은 최대 편집 (1 대 2)**: 더 빠르지만 더 많은 오타를 놓침
- **높은 최대 편집 (2 대 3)**: 더 느리지만 더 많은 오타를 잡음

**라이브러리 성능**: Spwig은 `rapidfuzz`가 사용 가능할 경우 이를 사용합니다 (순수 파이썬 대비 10배 빠름). 설치를 확인하세요: `pip install rapidfuzz`

## 동의어 및 리디렉션 성능

**동의어 쿼리 확장**: 각 동의어는 검색 쿼리에 OR 절을 추가합니다. 항목당 최대 10-20개의 동의어로 제한하세요.

**정규 표현식 일치 유형**: 정규 표현식 리디렉션은 정확/포함/시작_위한 것보다 느립니다. 복잡한 패턴은 피하세요.

**추천**: 가능한 한 간단한 일치 유형을 사용하세요. 다른 일치 유형이 작동하지 않는 경우에만 정규 표현식을 사용하세요.

## 대규모 카탈로그 최적화 (>10,000 제품)

대규모 카탈로그에 대한 전략:

**1. 강한 캐싱**
- 자동 완성: 90-120초 TTL
- 결과: 600초 (10분) TTL
- 성능을 위해 오래된 결과 수용

**2. 최소한의 깊은 인덱싱**
- SKU만 (속성, 커스텀 필드, 리뷰 비활성화)
- 속성 포함/제외하여 성능 테스트

**3. 자동 완성 결과 감소**
- 유형당 최대 5개 결과 (기존 8개에서 감소)
- 쿼리 오버헤드 감소

**4. 모든 곳에서 재고 상태 비활성화**
- 자동 완성
- 검색 결과에 표시되는 경우

**5. 50,000 제품 이상 시 Elasticsearch 고려**
- 데이터베이스 네이티브 검색은 약 50,000 제품까지 적합
- 그 이상의 경우 Elasticsearch를 고려하세요:
  - 복잡한 페시티드 검색
  - 높은 동시 검색 부하 (>100 검색/초)
  - 최적화에도 불구하고 일관된 500ms 이상 응답 시간

## 다국어 성능

JSONField JSONB 인덱싱 (PostgreSQL)은 다국어를 효율적으로 만듭니다:

- **1-3 언어**: 최소 오버헤드 (5-10ms)
- **5+ 언어**: 쿼리 복잡성의 약간 증가 (20-40ms)
- **10+ 언어**: 눈에 띄는 오버헤드 (50-100ms)

언어 수에 따라 오버헤드가 선형적으로 증가합니다.

## 긴급 성능 수정

검색이 매우 느릴 경우 (응답 시간 >2초), 다음 즉시 수정 사항을 순서대로 적용하세요:

**즉시** (지금 적용):
1. 문서 인덱싱 비활성화
2. 자동 완성에서 제품 수 비활성화
3. 캐시 TTL을 자동 완성 120초 / 결과 600초로 증가

**빠른** (24시간 이내 적용):
4. 자동 완성에서 재고 상태 비활성화
5. 자동 완성 최대 결과 수를 5로 줄이기
6. 자동 완성에서 제품 설명 비활성화

**중간** (1주일 이내 적용):
7. 제품 수가 20,000 이상인 경우 리뷰 인덱싱 비활성화
8. 사용하지 않는 콘텐츠 유형 검토 및 비활성화
9. 디바운스를 400ms로 증가

**예상 개선**: 이 9가지 수정 사항은 일반적으로 대규모 카탈로그에서 응답 시간을 60-80% 감소시킵니다.

## 팁

- **주간 응답 시간 모니터링** - 성능 저하를 조기에 발견하세요
- **캐시 증가가 첫 번째 최적화** - 캐시 TTL을 두 배로 늘리는 것이 가장 쉬운 이점입니다
- **자동 완성의 제품 수 = 비용이 많이 드는** - 자동 완성 성능의 가장 큰 악당
- **문서 인덱싱은 거의 가치가 없음** - 성능 비용이 이점보다 거의 항상 더 큼
- **한 번에 하나의 변경 사항만 테스트** - 동시에 변경 사항을 적용하면 원인/효과를 식별할 수 없습니다
- **실제 데이터 볼륨으로 벤치마킹** - 프로덕션 크기의 카탈로그로 테스트
- **대규모 카탈로그에서 재고 집계는 성능을 해침** - 자동 완성에서 재고 표시를 피하세요
- **50,000 제품 이상 시 Elasticsearch 고려** - 데이터베이스 네이티브 검색에는 한계가 있습니다
