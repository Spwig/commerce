---
title: 'การตั้งค่า SSO: Okta'
---

คู่มือนี้จะพาคุณผ่านขั้นตอนการเชื่อมต่อ Spwig กับ Okta เพื่อให้สามารถเข้าสู่ระบบแบบ single sign-on สำหรับผู้ดูแลระบบ เมื่อตั้งค่าเสร็จสิ้น บุคลากรของคุณสามารถเข้าสู่ระบบแดชบอร์ดผู้ดูแลระบบ Spwig ด้วยบัญชี Okta ของพวกเขา

**หมายเหตุ:** Okta อาจอัปเดตอินเทอร์เฟซคอนโซลผู้ดูแลระบบของพวกเขาเป็นระยะ คำแนะนำนี้ถูกเขียนขึ้นโดยอ้างอิงจากคอนโซลผู้ดูแลระบบ Okta ณ ช่วงต้นปี 2026 หากมีขั้นตอนใดที่แตกต่างจากสิ่งที่คุณเห็น กรุณาอ้างอิงเอกสารทางการของ Okta ที่ [สร้างการผสาน OIDC app](https://developer.okta.com/docs/guides/sign-into-web-app-redirect/main/)

## ข้อกำหนดเบื้องต้น

- องค์กร Okta (ทุกระดับ — บัญชีผู้พัฒนาฟรีสามารถใช้สำหรับการทดสอบได้)
- บทบาท **Super Administrator** หรือ **Application Administrator** ใน Okta
- URL ร้านค้า Spwig ของคุณ (เช่น `https://your-store.com`)
- บุคลากรต้องมีที่อยู่อีเมลใน Spwig ที่ตรงกับบัญชี Okta ของพวกเขา

## ขั้นตอนที่ 1: สร้างแอปพลิเคชัน

1. เข้าสู่ระบบใน [Okta Admin Console](https://your-org-admin.okta.com)
2. ไปที่ **Applications > Applications**
3. คลิก **Create App Integration**
4. เลือก:

| Field | Value |
|-------|-------|
| **Sign-in method** | OIDC - OpenID Connect |
| **Application type** | Web Application |

5. คลิก **Next**

## ขั้นตอนที่ 2: ตั้งค่าแอปพลิเคชัน

กรอกค่าตั้งค่าแอปพลิเคชัน:

| Field | Value |
|-------|-------|
| **App integration name** | `Spwig Admin SSO` (หรือชื่อใดก็ตามที่คุณต้องการ) |
| **Grant type** | Authorization Code (ควรได้รับการเลือกโดยค่าเริ่มต้น) |
| **Sign-in redirect URIs** | `https://your-store.com/oidc/callback/` |
| **Sign-out redirect URIs** | `https://your-store.com/en/admin/login/` |
| **Controlled access** | เลือกตามความต้องการของคุณ (ดูข้อมูลเพิ่มเติมด้านล่าง) |

สำหรับ **Controlled access** เลือกหนึ่งในตัวเลือกต่อไปนี้:

- **Allow everyone in your organization to access** — ผู้ใช้ Okta ทุกคนสามารถเข้าสู่ระบบได้ (คุณยังสามารถควบคุมการเข้าถึง Spwig ด้วยการตั้งค่า Restrict to Staff)
- **Limit access to selected groups** — เฉพาะผู้ใช้ในกลุ่ม Okta ที่กำหนดเท่านั้นที่สามารถเข้าสู่ระบบได้
- **Skip group assignment for now** — คุณจะกำหนดผู้ใช้หรือกลุ่มด้วยตนเองในภายหลัง

คลิก **บันทึก**

**สำคัญ:** URI สำหรับการเปลี่ยนเส้นทางการเข้าสู่ระบบต้องตรงกับ `https://your-store.com/oidc/callback/` — รวมถึงเครื่องหมาย slash ที่อยู่ด้านท้าย

## ขั้นตอนที่ 3: รับ Client Credentials

หลังจากบันทึก แท็บ **General** ของแอปพลิเคชันจะแสดงคุณสมบัติของคุณ:

| Value | ที่ไหนสามารถหาได้ |
|-------|-----------------|
| **Client ID** | แท็บ General, หัวข้อ Client Credentials |
| **Client Secret** | แท็บ General, หัวข้อ Client Credentials (คลิกไอคอนตาเพื่อเปิดเผย) |

คัดลอกค่าทั้งสอง — คุณจะต้องใช้มันกับ Spwig

## ขั้นตอนที่ 4: สร้าง URL สำหรับการค้นพบ (Discovery URL)

URL สำหรับการค้นพบขึ้นอยู่กับองค์กร Okta และเซิร์ฟเวอร์การอนุญาตของคุณ:

**เซิร์ฟเวอร์การอนุญาตแบบเริ่มต้น (ที่พบบ่อยที่สุด):**
```
https://your-org.okta.com/.well-known/openid-configuration
```

**เซิร์ฟเวอร์การอนุญาตแบบกำหนดเอง (ถ้าได้ตั้งค่าไว้):**
```
https://your-org.okta.com/oauth2/{authorization-server-id}/.well-known/openid-configuration
```

แทนที่ `your-org.okta.com` ด้วยโดเมน Okta จริงของคุณ คุณสามารถหาโดเมน Okta ของคุณได้ในแถบ URL ของคอนโซลผู้ดูแลระบบ หรือภายใต้ **Settings > Account**

**เคล็ดลับ:** องค์กรส่วนใหญ่ใช้ Org Authorization Server (เซิร์ฟเวอร์การอนุญาตแบบเริ่มต้น) ใช้ URL เซิร์ฟเวอร์การอนุญาตแบบกำหนดเองเพียงเมื่อผู้ดูแลระบบ Okta ของคุณตั้งค่าไว้โดยเฉพาะ

## ขั้นตอนที่ 5: กำหนดผู้ใช้หรือกลุ่ม

หากคุณเลือก "ข้ามการกำหนดกลุ่ม" ในขั้นตอนที่ 2 คุณจะต้องกำหนดผู้ใช้ก่อนที่พวกเขาจะสามารถเข้าสู่ระบบได้:

1. ในแท็บ **Assignments** ของแอปพลิเคชัน คลิก **Assign**
2. เลือก **Assign to People** หรือ **Assign to Groups**
3. เลือกผู้ใช้หรือกลุ่มและคลิก **Assign**
4. คลิก **Done**

ผู้ใช้ที่ไม่ได้รับการกำหนดให้กับแอปพลิเคชันจะเห็นข้อผิดพลาดเมื่อพยายามเข้าสู่ระบบผ่าน SSO

## ขั้นตอนที่ 6: กำหนดค่า Claims ของกลุ่ม (ไม่จำเป็น)

หากคุณต้องการให้ Spwig ตั้งค่าสถานะพนักงานหรือผู้ใช้ระดับสูงโดยอัตโนมัติโดยใช้การเป็นสมาชิกกลุ่มใน Okta:

1.

ไปที่ **Security > API** ในคอนโซลผู้ดูแลระบบ
2.

เลือก **Authorization Server** ของคุณ (ใช้ "default" หากคุณยังไม่ได้สร้างเซิร์ฟเวอร์การอนุญาตแบบกำหนดเอง หรือ Org Authorization Server)
3.

ไปที่แท็บ **Claims**
4.



คลิก **เพิ่มการเรียกร้อง**
5.

ตั้งค่าการเรียกร้อง:

| Field | Value |
|-------|-------|
| **ชื่อ** | `groups` |
| **รวมในประเภทโทเคน** | ID Token, Always |
| **ประเภทค่า** | Groups |
| **ตัวกรอง** | Matches regex: `.*` (เพื่อรวมกลุ่มทั้งหมด) |
| **รวมใน** | Any scope (หรือ `openid` หากคุณต้องการจำกัดมัน) |

6. คลิก **สร้าง**

**เคล็ดลับ:** ต่างจาก Microsoft Entra ID ที่ส่ง Object IDs, Okta ส่ง **ชื่อกลุ่ม** ตามค่าเริ่มต้น ทำให้การจับคู่บทบาทเป็นเรื่องเข้าใจง่ายขึ้น — คุณสามารถใช้ชื่อแสดงของกลุ่ม Okta ของคุณโดยตรงในช่อง Staff Groups และ Superuser Groups ของ Spwig

### การกรองกลุ่ม

หากผู้ใช้ของคุณอยู่ในกลุ่ม Okta หลายกลุ่ม และคุณต้องการให้รวมเฉพาะบางกลุ่มในโทเคน:

- เปลี่ยนตัวกรองจาก `.*` เป็น regex ที่เฉพาะเจาะจงมากขึ้น เช่น `^Spwig.*` เพื่อรวมเฉพาะกลุ่มที่เริ่มด้วย "Spwig"
- หรือใช้ตัวกรอง **เริ่มด้วย**, **เท่ากับ**, หรือ **มีอยู่** แทน regex

## ขั้นตอนที่ 7: ตั้งค่าใน Spwig

1. ในแดชบอร์ดแอดมินของ Spwig ให้ไปที่ **Enterprise SSO > SSO Provider Configuration**
2. ตั้งค่า **Provider Name** เป็น `Okta`
3. ใส่ URL การค้นพบจากขั้นตอนที่ 4
4. คลิก **Auto-Discover** — นี่จะเติมฟิลด์ปลายทางทั้งหมดอัตโนมัติ
5. ใส่ **Client ID** จากขั้นตอนที่ 3
6. ใส่ **Client Secret** จากขั้นตอนที่ 3
7. หากคุณตั้งค่าการเรียกร้องกลุ่มในขั้นตอนที่ 6:
   - ตั้งค่า **Groups Claim** เป็น `groups`
   - ในช่อง **Staff Groups** ใส่ชื่อกลุ่ม Okta ของกลุ่มที่สมาชิกควรเป็นพนักงาน (แยกด้วยเครื่องหมายจุลภาค)
   - ในช่อง **Superuser Groups** ใส่ชื่อกลุ่ม Okta ของกลุ่มที่สมาชิกควรเป็นผู้ใช้ระดับซูเปอร์ยูเซอร์ (แยกด้วยเครื่องหมายจุลภาค)
8. คลิก **บันทึก**

## ขั้นตอนที่ 8: เปิดใช้งานและทดสอบ

1.

ไปที่ **Site Settings > Security** tab
2.

ตรวจสอบ **Enable SSO for admin login**
3.

คลิก **บันทึก**
4.

เปิดหน้าเพจล็อกอินแอดมินใน **หน้าต่างส่วนตัว/อินคอกนิโตร**
5.

คุณควรเห็นปุ่ม **Sign in with Okta**
6.

คลิกมัน — คุณควรจะถูกเปลี่ยนเส้นทางไปยังหน้าล็อกอินของ Okta
7.



เข้าสู่ระบบด้วยบัญชี Okta ที่ถูกกำหนดให้กับแอปพลิเคชัน และอีเมลตรงกับผู้ใช้พนักงานใน Spwig
8.

คุณควรจะถูกเปลี่ยนเส้นทางกลับไปยังแดชบอร์ดผู้ดูแลระบบ Spwig

## ปัญหาที่พบบ่อย

| ปัญหา | สาเหตุ | วิธีแก้ไข |
|---------|-------|----------|
| **URI การเปลี่ยนเส้นทางไม่ได้รับอนุญาต** | URI การเปลี่ยนเส้นทางไม่ตรงกับการตั้งค่าของแอปพลิเคชัน | ตรวจสอบให้ URI การเปลี่ยนเส้นทางเข้าสู่ระบบตรงกับ `https://your-store.com/oidc/callback/` อย่างแน่นอน พร้อมทั้งเครื่องหมาย slash ที่ปลายทาง |
| **ผู้ใช้ไม่ได้รับการกำหนดให้กับแอปพลิเคชัน** | ผู้ใช้ไม่ได้รับการกำหนดให้กับแอปพลิเคชัน Okta | กำหนดผู้ใช้หรือกลุ่มของผู้ใช้ให้กับแอปพลิเคชันในแท็บ Assignments |
| **การเข้าสู่ระบบสำเร็จที่ Okta แต่ล้มเหลวที่ Spwig** | ไม่มีผู้ใช้ที่ตรงกันใน Spwig | ตรวจสอบให้มีบัญชีพนักงานใน Spwig ที่มีอีเมลเดียวกัน ตรวจสอบการตั้งค่า Restrict to Staff |
| **กลุ่ม (Groups) ไม่มีข้อมูล** | ข้อมูลกลุ่ม (Groups) ไม่ได้กำหนดในเซิร์ฟเวอร์การยืนยันตัวตน | ทำตามขั้นตอนที่ 6 เพื่อเพิ่มข้อมูลกลุ่ม (Groups) ตรวจสอบให้แน่ใจว่าคุณกำลังเพิ่มมันในเซิร์ฟเวอร์การยืนยันตัวตนที่ถูกต้อง |
| **เซิร์ฟเวอร์การยืนยันตัวตนไม่ถูกต้อง** | URL การค้นพบใช้เซิร์ฟเวอร์การยืนยันตัวตนที่ต่างออกไปจากที่กำหนดข้อมูลกลุ่ม (Groups) | ตรวจสอบให้ URL การค้นพบตรงกับเซิร์ฟเวอร์การยืนยันตัวตนที่คุณกำหนดข้อมูลกลุ่ม (Groups) |
| **"Client ID ที่ให้มาไม่ถูกต้อง"** | Client ID ไม่ตรงกันหรือแอปพลิเคชันไม่ได้เปิดใช้งาน | ตรวจสอบให้ Client ID ถูกต้อง และสถานะของแอปพลิเคชันใน Okta เป็น Active |

## เคล็ดลับ

- **Okta ส่งชื่อกลุ่ม ไม่ใช่ ID** — ทำให้การจับคู่บทบาทเป็นเรื่องง่าย

กรอกชื่อแสดงของกลุ่มที่ถูกต้อง (เช่น `Spwig Admins`) ในช่อง Staff Groups หรือ Superuser Groups ของ Spwig
- **ใช้การกำหนดกลุ่มเพื่อควบคุมการเข้าถึง** — กำหนดกลุ่ม Okta ที่เฉพาะเจาะจงให้กับแอปพลิเคชัน Spwig แทนที่จะอนุญาตให้ผู้ใช้ทั้งหมดเข้าถึง

# การตั้งค่าการเข้าสู่ระบบ SSO สำหรับ Spwig

ด้วยวิธีนี้ บุคลากรที่ต้องการเท่านั้นจึงสามารถเข้าสู่ระบบได้
- **Okta client secrets ไม่หมดอายุโดยปริยาย** — แต่คุณสามารถหมุนเวียนพวกเขาได้ทุกเมื่อจากแท็บ General ของแอปพลิเคชัน เพื่อปฏิบัติตามแนวทางด้านความปลอดภัยที่ดีที่สุด
- **ทดสอบด้วยบัญชีที่ไม่ใช่ผู้ดูแลระบบ** — ใช้ผู้ใช้ Okta ทั่วไป (ไม่ใช่ผู้ดูแลระบบหลัก) ที่ถูกกำหนดให้กับแอปพลิเคชันเพื่อตรวจสอบว่า SSO ทำงานตามที่คาดไว้
- **MFA ใน Okta** — ตั้งค่านโยบายการเข้าสู่ระบบระดับโลกหรือนโยบายการยืนยันตัวตนของ Okta เพื่อต้องการ MFA

สิ่งนี้จะใช้กับการเข้าสู่ระบบ SSO ทั้งหมดใน Spwig โดยไม่จำเป็นต้องตั้งค่า MFA แยกใน Spwig
