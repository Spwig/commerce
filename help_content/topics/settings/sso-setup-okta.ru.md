Этот руководство поможет вам подключить Spwig к Okta для однократного входа в административную панель. После настройки ваш персонал сможет войти в административную панель Spwig, используя свой аккаунт Okta.

**Примечание:** Okta может обновлять интерфейс своего административного консоли со временем. Эти инструкции были написаны на основе административного консоли Okta на начало 2026 года. Если какие-либо шаги отличаются от того, что вы видите, обратитесь к официальной документации Okta о [создании интеграции приложения OIDC](https://developer.okta.com/docs/guides/sign-into-web-app-redirect/main/).

## Предварительные условия

- Организация Okta (любой уровень — бесплатные аккаунты разработчика подходят для тестирования)
- Роль **Super Administrator** или **Application Administrator** в Okta
- URL вашего магазина Spwig (например, `https://your-store.com`)
- У сотрудников должны быть адреса электронной почты в Spwig, совпадающие с их аккаунтами Okta

## Шаг 1: Создание приложения

1. Войдите в [Okta Admin Console](https://your-org-admin.okta.com)
2. Перейдите в **Applications > Applications**
3. Нажмите **Create App Integration**
4. Выберите:

| Поле | Значение |
|-------|-------|
| **Метод входа** | OIDC - OpenID Connect |
| **Тип приложения** | Web Application |

5. Нажмите **Next**

## Шаг 2: Настройка приложения

Заполните настройки приложения:

| Поле | Значение |
|-------|-------|
| **Имя интеграции приложения** | `Spwig Admin SSO` (или любое имя, которое вы предпочитаете) |
| **Тип разрешения** | Authorization Code (должен быть выбран по умолчанию) |
| **URL перенаправления входа** | `https://your-store.com/oidc/callback/` |
| **URL перенаправления выхода** | `https://your-store.com/en/admin/login/` |
| **Контролируемый доступ** | Выберите в зависимости от ваших потребностей (см. ниже) |

Для **Контролируемого доступа** выберите один из:

- **Разрешить всем в вашей организации доступ** — все пользователи Okta могут войти (вы все равно можете управлять доступом к Spwig с помощью настройки "Ограничить доступ для сотрудников\

Нажмите **Добавить утверждение**
5.

Настройте утверждение:

| Параметр | Значение |
|---------|---------|
| **Название** | `groups` |
| **Включить в тип токена** | ID Token, Always |
| **Тип значения** | Groups |
| **Фильтр** | Matches regex: `.*` (чтобы включить все группы) |
| **Включить в** | Any scope (или `openid`, если вы хотите ограничить это) |

6. Нажмите **Создать**

**Совет:** В отличие от Microsoft Entra ID, которая отправляет Object IDs, Okta по умолчанию отправляет **названия групп**. Это делает сопоставление ролей более интуитивным — вы можете напрямую использовать отображаемые названия групп Okta в полях Группы персонала и Группы суперпользователей в Spwig.

### Фильтрация групп

Если ваши пользователи принадлежат к множеству групп Okta, и вы хотите включить только определенные из них в токен:

- Измените фильтр с `.*` на более конкретное регулярное выражение, например, `^Spwig.*`, чтобы включить только группы, начинающиеся с "Spwig"
- Или используйте фильтры **Начинается с**, **Равно** или **Содержит** вместо регулярных выражений

## Шаг 7: Настройка в Spwig

1. В админ-панели Spwig перейдите в **Enterprise SSO > Настройка провайдера SSO**
2. Установите **Название провайдера** на `Okta`
3. Введите URL открытия из шага 4
4. Нажмите **Автоматическое обнаружение** — это заполнит все поля конечных точек автоматически
5. Введите **Client ID** из шага 3
6. Введите **Client Secret** из шага 3
7. Если вы настроили утверждения групп в шаге 6:
   - Установите **Groups Claim** на `groups`
   - В поле **Группы персонала** введите названия групп Okta, члены которых должны быть персоналом (разделенные запятыми)
   - В поле **Группы суперпользователей** введите названия групп Okta, члены которых должны быть суперпользователями (разделенные запятыми)
8. Нажмите **Сохранить**

## Шаг 8: Включение и тестирование

1. Перейдите в **Настройки сайта > вкладка Безопасность**
2. Поставьте галочку **Включить SSO для входа в админ-панель**
3. Нажмите **Сохранить**
4. Откройте страницу входа в админ-панель в **частном/инкогнито окне**
5. Вы должны увидеть кнопку **Вход через Okta**
6. Нажмите на нее — вас перенаправят на страницу входа Okta
7. Войдите с учетной записью Okta, которая назначена приложению и чей email совпадает с учетной записью персонала в Spwig
8. Вас должны перенаправить обратно на админ-панель Spwig

## Распространенные проблемы

| Проблема | Причина | Решение |
|---------|-------|----------|
| **Перенаправляющий URI не разрешен** | Перенаправляющий URI не совпадает с конфигурацией приложения | Проверьте, что URI перенаправления входа точно равен `https://your-store.com/oidc/callback/` с завершающим слешем |
| **Пользователь не назначен клиентскому приложению** | Пользователь не назначен приложению Okta | Назначьте пользователя или его группу приложению в вкладке Назначения |
| **Вход успешно проходит в Okta, но не проходит в Spwig** | Нет совпадающего пользователя в Spwig | Убедитесь, что в Spwig существует учетная запись персонала с таким же email. Проверьте настройку "Ограничить только персоналом". |
| **Утверждение групп пустое** | Утверждение групп не настроено на сервере авторизации | Следуйте шагу 6, чтобы добавить утверждение групп. Убедитесь, что вы добавляете его на правильный сервер авторизации. |
| **Неправильный сервер авторизации** | URL открытия использует другой сервер авторизации, чем тот, где настроено утверждение групп | Проверьте, совпадает ли URL открытия с сервером авторизации, где вы настроили утверждение групп |
| **"Предоставленный client_id недействителен"** | Client ID не совпадает или приложение неактивно | Проверьте, что Client ID правильный, и статус приложения в Okta — Active |

## Советы

- **Okta отправляет названия групп, а не ID** — это делает сопоставление ролей простым.

Введите точное отображаемое название группы (например, `Spwig Admins`) в полях Группы персонала или Группы суперпользователей в Spwig.
- **Используйте назначение групп для контроля доступа** — назначьте конкретные группы Okta приложению Spwig вместо разрешения доступа всем пользователям.

# Настройка SSO с Okta

Таким образом, только намеченный персонал может войти в систему.
- **Секреты клиентов Okta не истекают по умолчанию** — но вы можете их обновить в любое время на вкладке "Общее" приложения для соблюдения лучших практик безопасности.
- **Тестирование с непривилегированным аккаунтом** — используйте обычного пользователя Okta (не супер-администратора), назначенного приложению, чтобы проверить, работает ли SSO как ожидается.
- **MFA в Okta** — настройте глобальную политику сеанса Okta или политики аутентификации, чтобы требовать MFA.

Это будет применяться ко всем входам в Spwig через SSO без необходимости отдельной настройки MFA в Spwig.
