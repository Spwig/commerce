---
title: वेबहुक लॉग्स
---

वेबहुक लॉग्स सभी आने वाले वितरक वेबहुक अनुरोधों के लिए एक स्थायी एडिट ट्रेल प्रदान करते हैं - अनुरोध विधि, अंत मार्ग URL, हेडर, पेलोड, प्रोसेसिंग स्थिति (पेंडिंग/प्रोसेस्ड/फेल), और जवाब को रिकॉर्ड करते हैं। प्रोसेसिंग के पहले प्रत्येक वेबहुक को लॉग किया जाता है ताकि यदि प्रोसेसिंग विफल हो जाता है तो कोई घटना न खो जाए। लॉग्स वेबहुक एंटीग्रेशन समस्याओं के डीबगिंग, वितरक API विश्वसनीयता की निगरानी, और ग्राहक समर्थन के लिए डिलीवरी टाइमलाइन के पुनर्निर्माण के लिए उपयोगी हैं।

यह पढ़-लिख एडमिन पेज वेबहुक विफलताओं के डीबगिंग और वितरक एंटीग्रेशन स्वास्थ्य की पुष्टि के लिए मदद करता है।

## वेबहुक लॉग संरचना

प्रत्येक लॉग एंट्री रिकॉर्ड करता है:

**अनुरोध विवरण**:
- **प्रदाता कुंजी**: कौन सा वितरक वेबहुक भेजा (fedex, ups, dhl)
- **अंत मार्ग**: वेबहुक URL पथ (जैसे, `/webhooks/shipping/fedex/`)
- **विधि**: HTTP विधि (आमतौर पर POST)
- **हेडर**: अनुरोध हेडर (JSON)
- **पेलोड**: अनुरोध शरीर (JSON)

**प्रोसेसिंग**:
- **प्रोसेसिंग स्थिति**: पेंडिंग, प्रोसेस्ड, फेल
- **त्रुटि संदेश**: विफलता कारण (यदि स्थिति=फेल)
- **जवाब**: वितरक के लिए भेजा गया HTTP जवाब
- **जवाब स्थिति कोड**: 200, 400, 500, आदि

**टाइमस्टैंप्स**:
- **प्राप्त करने पर**: जब वेबहुक आया
- **प्रोसेस करने पर**: जब प्रोसेसिंग पूर्ण हुआ

---

## प्रोसेसिंग स्थिति मान

**पेंडिंग**: वेबहुक प्राप्त, प्रोसेसिंग के लिए इंतजार
- प्राप्ति के बाद क्षण असामान्य नहीं है
- यदि पेंडिंग में फंसा है, तो प्रोसेसिंग क्यू के बैकलॉग को इंगित करता है

**प्रोसेस्ड**: वेबहुक सफलता से प्रोसेस किया गया
- ट्रैकिंग ईवेंट बनाया गया
- ग्राहक सूचना भेजी गई (यदि लागू)
- वितरक को HTTP 200 जवाब भेजा गया

**फेल**: वेबहुक प्रोसेसिंग विफल
- त्रुटि_संदेश देखें कारण
- सामान्य कारण: अमान्य JSON, अज्ञात शिपमेंट, डुप्लिकेट ईवेंट

---

## वेबहुक फ्लो

**सामान्य कार्यप्रवाह**:
```
1. वितरक पैकेज स्कैन करता है
   ↓
2. वितरक Spwig वेबहुक अंत मार्ग पर POST भेजता है
   ↓
3. Spwig WebhookLog बनाता है (स्थिति=पेंडिंग)
   ↓
4. पृष्ठभूमि कार्यकर्ता वेबहुक प्रोसेस करता है
   ↓
5. JSON पेलोड पार्स करें
   ↓
6. ट्रैकिंग नंबर द्वारा तुलना करते हुए अनुरोध शिपमेंट खोजें
   ↓
7. ट्रैकिंग ईवेंट बनाएं
   ↓
8. WebhookLog अपडेट करें (स्थिति=प्रोसेस्ड)
   ↓
9. वितरक को HTTP 200 जवाब भेजें
```

**विफलता सीनारियो**:
- **अमान्य JSON**: वितरक द्वारा अमान्य डेटा भेजा गया → स्थिति=फेल, त्रुटि="JSON पार्स त्रुटि"
- **अज्ञात शिपमेंट**: ट्रैकिंग नंबर कोई शिपमेंट नहीं मिला → स्थिति=फेल, त्रुटि="शिपमेंट नहीं मिला"
- **डुप्लिकेट**: ईवेंट पहले से मौजूद है → स्थिति=फेल, त्रुटि="डुप्लिकेट ईवेंट"

---

## वेबहुक विफलता के डीबगिंग

**चरण-दर-चरण**:

**1. स्थिति=फेल के अनुसार छाने**
- शिपमेंट > वेबहुक लॉग्स जाएं
- छानें: प्रोसेसिंग स्थिति = "फेल"
- हालिया विफलताओं की समीक्षा करें

**2. त्रुटि संदेश देखें**
- लॉग एंट्री पर क्लिक करें
- त्रुटि_संदेश क्षेत्र पढ़ें
- सामान्य त्रुटियां:
  - "शिपमेंट नहीं मिला" → ट्रैकिंग नंबर असंगति
  - "JSON डिकोड त्रुटि" → वितरक द्वारा अमान्य JSON भेजा
  - "अपेक्षित क्षेत्र की कमी" → पेलोड में अपेक्षित डेटा कम

**3. पेलोड की जांच करें**
- क्रूर JSON पेलोड देखें
- संरचना अपेक्षित प्रारूप के साथ सुमेल चेक करें
- लापता क्षेत्र (ट्रैकिंग_आईडी, ईवेंट_प्रकार, आदि) की जांच करें

**4. शिपमेंट के अस्तित्व की पुष्टि करें**
- पेलोड से ट्रैकिंग नंबर निकालें
- ट्रैकिंग नंबर के लिए शिपमेंट खोजें
- शिपमेंट के अस्तित्व और सही वितरक के उपयोग की पुष्टि करें

**5. प्रदाता विनिर्माण की जांच करें**
- प्रदाता खाता सक्रिय है या नहीं
- वेबहुक अंत मार्ग URL सही है या नहीं
- प्रदाता API प्रमाणपत्र जांचें

**6. प्रोसेसिंग के पुनर्प्रयास** (यदि लागू)
- कुछ वेबहुक प्रोसेसर मानक रूप से पुनर्प्रयास के समर्थन करते हैं
- प्रमुख समस्या को पहले ठीक करें
- विफल वेबहुक के पुनर्प्रयास करें

---

## सामान्य वेबहुक समस्याएं

**समस्या 1: "शिपमेंट नहीं मिला"**

**कारण**: वेबहुक में ट्रैकिंग नंबर कोई शिपमेंट नहीं मिला
- शिपमेंट बनाते समय टाइपो
- अलग खाता के वेबहुक
- शिपमेंट वेबहुक प्राप्त होने से पहले ही डिलीट कर दिया गया

**समाधान**:
- ट्रैकिंग नंबर के लिखावट की जांच करें
- शिपमेंट वितरक के वेबहुक प्रदाता के साथ सुमेल की पुष्टि करें
- आवश्यकता पड़े तो शिपमेंट फिर से बनाएं

---

**समस्या 2: "JSON डिकोड त्रुटि"**

**कारण**: वितरक द्वारा अमान्य JSON भेजा गया
- दुर्लभ, आमतौर पर वितरक API बग
- करैक्टर एन्कोडिंग समस्याएं

**समाधान**:
- वितरक समर्थन के साथ क्रूर पेलोड भेजें
- हेडर के लिए करैक्टर एन्कोडिंग जांचें
- वितरक डैशबोर्ड में अंत मार्ग URL की पुष्टि करें

---

**समस्या 3: डुप्लिकेट वेबहुक**

**कारण**: वितरक एक ही ईवेंट कई बार भेजता है
- पुनर्प्रयास तकनीक (वितरक 200 जवाब प्राप्त नहीं करता)
- वितरक बग

**समाधान**:
- प्रणाली ऑटो डुप्लिकेट को अस्वीकृत करता है (सामान्य व्यवहार)
- जांचें कि जवाब_स्थिति_कोड 200 है
- यदि बनी रहती है, तो वितरक समर्थन से संपर्क करें

---

**समस्या 4: लापता वेबहुक**

**कारण**: अपेक्षित वेबहुक कभी नहीं प्राप्त हुआ
- वितरक ने भेजा नहीं (स्कैन छूट गया)
- वेबहुक अंत मार्ग वितरक डैशबोर्ड में गलत रूप से विनिर्मित
- फायरवॉल अनुरोधों को ब्लॉक करता है

**समाधान**:
- वितरक डैशबोर्ड वेबहुक विनिर्माण जांचें
- अंत मार्ग URL की पुष्टि करें कि यह सार्वजनिक और पहुँच योग्य है
- अंत मार्ग के साथ curl/Postman टेस्ट करें
- सर्वर फायरवॉल नियम जांचें

---

## वेबहुक अंत मार्ग विनिर्माण

**सामान्य वेबहुक URLs**:
```
FedEx: https://yourdomain.com/webhooks/shipping/fedex/
UPS: https://yourdomain.com/webhooks/shipping/ups/
DHL: https://yourdomain.com/webhooks/shipping/dhl/
```

**वितरक डैशबोर्ड सेटअप**:
1. वितरक विकास पोर्टल में लॉग इन करें
2. वेबहुक सेटिंग्स तक जाएं
3. Spwig वेबहुक URL दर्ज करें
4. सब्सक्राइब करने वाले ईवेंट्स का चयन करें (ट्रैकिंग अपडेट, डिलीवरी, अपवाद)
5. विनिर्माण को सहेजें
6. वितरक के परीक्षण उपकरण के साथ वेबहुक की जांच करें

**सुरक्षा**:
- वेबहुक HTTPS की आवश्यकता होती है (HTTP नहीं)
- कुछ वितरक अनुरोधों के हस्ताक्षर करते हैं (हस्ताक्षर जांचें)
- IP व्हाइटलिस्ट (यदि वितरक स्थायी IP की पेशकश करता है)

---

## वेबहुक स्वास्थ्य निगरानी

**महत्वपूर्ण मीट्रिक्स**:

**सफलता दर**:
```
सफलता दर = (प्रोसेस्ड / कुल) × 100%

लक्ष्य: >98%
```

**प्रोसेसिंग समय**:
```
अवधि = प्रोसेस करने पर - प्राप्त करने पर

लक्ष्य: <2 सेकंड
```

**विफलता पैटर्न**:
- अचानक विफलताओं का उछाल → वितरक API बदलाव या बंद होना
- समान "शिपमेंट नहीं मिला" → ट्रैकिंग नंबर सिंक्रोनाइजेशन समस्या
- सभी वेबहुक विफल → अंत मार्ग विनिर्माण समस्या

**निगरानी रणनीति**:
- दैनिक विफलता दर जांचें
- यदि विफलता दर >5% है, तो सचेत करें
- हफ्ते में त्रुटि संदेश जांचें
- वितरक स्थिति पृष्ठ के साथ तुलना करें

---

## वेबहुक रिटेंशन

**लॉग्स स्थायी हैं** - कभी भी स्वचालित रूप से डिलीट नहीं होते

**क्यों स्थायी**:
- एडिट अनुपालन
- ग्राहक समर्थन (डिलीवरी टाइमलाइन पुनर्निर्माण)
- विवाद समाधान
- वेबहुक डीबगिंग

**संग्रहण**: लॉग्स दक्षता से संग्रहित (संपीड़ित JSON)

---

## सुझाव

- **वेबहुक स्थायी एडिट लॉग हैं** - यहां तक कि यदि सफलता से प्रोसेस किया गया है, कभी भी डिलीट न करें
- **प्रतिदिन असफल वेबहुक जांचें** - एंटीग्रेशन समस्याओं को शुरुआत में पकड़ें
- **प्रोसेसिंग लैग की निगरानी करें** - लंबा विलंब प्रदर्शन समस्या को इंगित करता है
- **क्रूर पेलोड सहेजें** - वितरक API बदलाव के लिए आवश्यक है
- **अंत मार्ग विनिर्माण की जांच करें** - वितरक परीक्षण उपकरण के साथ स्थापना की पुष्टि करें
- **हस्ताक्षर वेबहुक सक्षम करें** - असली वितरक अनुरोधों की जांच करें
- **वितरक IP को व्हाइटलिस्ट करें** - यदि वितरक स्थायी IP सीमा पेश करता है
- **अभिकर्मक सेट करें** - जब विफलता दर अपने द्वारा निर्धारित थ्रेशोल्ड के ऊपर होता है तो सूचित करें
- **वितरक स्थिति के साथ तुलना करें** - वेबहुक अंतर वितरक बंद होना को इंगित कर सकता है
- **वितरक पेलोड प्रारूप दस्तावेजीकरण करें** - जब वितरक API अपग्रेड करता है तो मदद करता है
- **वेबहुक URLs को स्थायी रखें** - URLs के बदलाव के लिए वितरक डैशबोर्ड अपडेट की आवश्यकता होती है
