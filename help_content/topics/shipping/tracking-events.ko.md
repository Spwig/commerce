---
title: 추적 이벤트
---

추적 이벤트는 배송 주기 내에서 배송 상태의 체크포인트를 기록합니다. 각 이벤트는 상태(이동 중, 배송 중, 배송 완료), 타임스탬프, 위치, 설명, 원본 운송업체 데이터를 포함합니다. 이벤트는 운송업체 웹훅 알림을 통해 자동으로 생성되거나 판매자가 수동으로 생성할 수 있습니다. 고객은 자신의 계정 및 주문 확인 이메일에서 추적 이벤트 기록을 볼 수 있으며, 실시간 배송 가시성을 제공합니다.

이 관리 페이지는 감사 및 고객 지원을 위해 읽기 전용 이벤트 기록을 표시합니다.

## 추적 이벤트 구조

각 이벤트는 다음과 같은 정보를 포함합니다:

**상태 정보**:
- **상태**: in_transit, out_for_delivery, delivered, exception, failed, returned
- **설명**: 인간이 읽을 수 있는 상태(예: "패키지가 분류 시설에 도착함")
- **운송업체 상태 코드**: 원본 운송업체 상태(예: "DEP" for departed)

**위치 데이터**:
- **도시**: 이벤트 위치 도시
- **주**: 이벤트 위치 주/도
- **국가**: 이벤트 위치 국가
- **우편 번호**: 이벤트 위치 ZIP/우편 번호

**타임스탬프**:
- **발생 시간**: 이벤트가 실제로 발생한 시간(운송업체 시간)
- **생성 시간**: 이벤트가 Spwig에 기록된 시간(시스템 시간)

**메타데이터**:
- **원본 데이터**: 운송업체 API에서 전달된 전체 JSON 응답
- **배송**: 연결된 배송 ID

---

## 이벤트 상태 유형

**in_transit**: 패키지가 운송업체 네트워크를 통해 이동 중
- 예시: "설비 출발", "허브에 도착", "다음 설비로 이동 중"

**out_for_delivery**: 패키지가 배송 차량에 있음
- 예시: "배송 중", "배송 차량에 탑승"

**delivered**: 패키지가 성공적으로 배송됨
- 예시: "앞문에 배송됨", "수신처에 남김", "수령인에게 전달됨"

**exception**: 주의가 필요한 배송 문제
- 예시: "기상 지연", "주소 오류", "배송 시도 실패"

**failed**: 영구적으로 배송 실패
- 예시: "기재된 주소로 배송 불가", "수령인에 의해 거부됨"

**returned**: 패키지가 발신자에게 반환 중
- 예시: "반환 시작", "패키지 반환"

## 추적 이벤트 생성 방법

### 자동 (운송업체 웹훅)

**워크플로우**:
1. 운송업체가 패키지를 스캔함(출발, 도착, 배송)
2. 운송업체가 Spwig 웹훅 엔드포인트로 웹훅을 전송함
3. 웹훅이 WebhookLog 테이블에 기록됨
4. 시스템이 웹훅 페이로드를 파싱함
5. 추적 이벤트가 추출된 데이터와 함께 생성됨
6. 고객 이메일 알림이 전송됨(설정된 경우)

**이점**:
- 실시간 업데이트(폴링 필요 없음)
- 운송업체에서 제공하는 정확한 타임스탬프
- 전체 이벤트 기록이 자동으로 유지됨

### 수동 (판매자 입력)

**워크플로우**:
1. 배송 상세 정보로 이동
2. "추적 이벤트 추가"를 클릭
3. 드롭다운에서 상태를 선택
4. 설명을 입력
5. 선택 사항: 위치 데이터 입력
6. 발생 시간 타임스탬프 설정
7. 저장

**사용 사례**:
- 웹훅 지원이 없는 운송업체
- 수동 배송 수정
- 지역 배송(운송업체 외)
- 내부 상태 업데이트

## 이벤트 표시 순서

이벤트는 **역순 시간 순서**로 표시됩니다(최신 먼저):

**예시 표시**:
```
2026년 2월 13일 오후 10:30 - 배송 완료 (브루클린, NY)
2026년 2월 13일 오전 8:15 - 배송 중 (브루클린, NY)
2026년 2월 12일 오후 11:45 - 지역 시설 도착 (브루클린, NY)
2026년 2월 12일 오후 6:30 - 이동 중 (뉴어크, NJ)
2026년 2월 12일 오후 2:15 - 출발지 출발 (필라델피아, PA)
2026년 2월 12일 오전 9:00 - 수령 완료 (필라델피아, PA)
```

## 고객 가시성

고객에게 표시되는 추적 이벤트는 다음과 같습니다:

**주문 확인 이메일**:
- 최신 이벤트 상태
- 예상 배송 날짜
- 추적 링크

**고객 계정 > 주문 세부정보**:
- 전체 이벤트 타임라인
- 이벤트 설명
- 위치 기록
- 타임스탬프

**추적 페이지**(활성화된 경우):
- 전용 추적 URL
- 시각적 타임라인
- 운송업체 로고
- 배송 지도(위치 데이터가 있는 경우)

---

## 추적 이벤트 필터링

**유용한 필터**:
- **배송**: 특정 배송의 이벤트를 보기 위해
- **상태**: 이벤트 유형(delivered, in_transit 등)으로 필터
- **날짜 범위**: 시간대 내 이벤트
- **위치**: 특정 도시/주에 있는 이벤트

**사용 사례**:
- "오늘 배송 완료된 모든 배송 보기"
- "지난 주의 예외 모두 찾기"
- "현재 in_transit 중인 배송 추적"

---

## 원본 데이터 (디버깅)

**원본 데이터 필드**:
- 운송업체 API에서 전달된 전체 JSON 응답을 저장함
- 웹훅 문제 디버깅에 유용함
- 운송업체 특정 메타데이터 포함

**원본 데이터 예시**(FedEx):
```json
{
  "event_type": "OD",
  "event_description": "Out for delivery",
  "timestamp": "2026-02-13T08:15:00Z",
  "location": {
    "city": "Brooklyn",
    "state": "NY",
    "postal_code": "11201",
    "country": "US"
  },
  "delivery_signature": null,
  "estimated_delivery": "2026-02-13T17:00:00Z"
}
```

**원본 데이터 확인 시기**:
- 이벤트 설명이 불명확할 때
- 위치 데이터 누락 시
- 웹훅 처리 오류 시
- 운송업체 지원 요청 시

---

## 이벤트 타이밍

**발생 시간** vs **생성 시간**:

**발생 시간**: 운송업체 이벤트가 실제로 발생한 시간
- 예시: 패키지가 오전 10:30에 스캔됨

**생성 시간**: Spwig이 웹훅을 수신한 시간
- 예시: 웹훅이 오전 10:32에 수신됨(2분 지연)

**왜 다른가요?**:
- 네트워크 지연
- 운송업체 배치 처리
- 웹훅 재시도 지연

**고객 표시에는 발생 시간 사용** - 실제 배송 진행 상황을 더 정확하게 반영합니다.

---

## 팁

- **이벤트는 읽기 전용입니다** - 생성 후 편집 불가(감사 무결성)
- **세부 정보를 확인하려면 원본 데이터를 확인하세요** - 표시된 필드보다 더 많은 정보를 포함함
- **웹훅 지연 모니터링** - occurred_at과 created_at 사이의 큰 지연은 웹훅 문제를 나타냄
- **고객 지원에 사용** - 이벤트 타임라인은 배송 문제 진단에 도움됨
- **배송 패턴 추적** - 이벤트 타이밍을 분석하여 운송업체 성능 평가
- **알림 설정** - 주요 이벤트(out_for_delivery, delivered) 시 고객에게 자동 이메일 전송
- **이벤트를 삭제하지 마세요** - 전체 감사 기록 보존
- **웹훅 처리 오류 확인을 위해 WebhookLog 확인** - 누락된 이벤트는 웹훅 처리 오류를 나타낼 수 있음
- **위치 데이터는 운송업체에 따라 다름** - 일부 운송업체는 자세한 위치를 제공하고, 다른 운송업체는 최소한의 정보만 제공함
- **예외 이벤트는 주의가 필요함** - 배송 예외를 모니터링하고 후속 조치 필요
