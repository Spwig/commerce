---
title: เหตุการณ์การติดตาม
---

เหตุการณ์การติดตามบันทึกสถานะการส่งของที่สำคัญตลอดวงจรการจัดส่ง—แต่ละเหตุการณ์จะบันทึกสถานะ (กำลังส่ง, กำลังจะส่งของ, ได้รับสินค้า), เวลาที่เกิดเหตุการณ์, สถานที่, คำอธิบาย, และข้อมูลผู้ให้บริการต้นฉบับ เหตุการณ์ถูกสร้างโดยอัตโนมัติผ่านการแจ้งเตือน webhook ของผู้ให้บริการ หรือสร้างด้วยตนเองโดยผู้ค้า ลูกค้าจะเห็นประวัติเหตุการณ์การติดตามในบัญชีของตนเองและอีเมลยืนยันการสั่งซื้อ ซึ่งให้ความเห็นเชิงเวลาจริงเกี่ยวกับการจัดส่ง

หน้าผู้ดูแลระบบนี้แสดงประวัติเหตุการณ์แบบอ่านเพียงอย่างเดียวเพื่อการตรวจสอบและสนับสนุนลูกค้า

## โครงสร้างของเหตุการณ์การติดตาม

แต่ละเหตุการณ์ประกอบด้วย:

**ข้อมูลสถานะ**:
- **สถานะ**: in_transit, out_for_delivery, delivered, exception, failed, returned
- **คำอธิบาย**: สถานะที่มนุษย์สามารถอ่านได้ (เช่น "พัสดุมาถึงศูนย์จัดเรียง")
- **รหัสสถานะผู้ให้บริการ**: สถานะต้นฉบับของผู้ให้บริการ (เช่น "DEP" สำหรับออกจากศูนย์)

**ข้อมูลสถานที่**:
- **เมือง**: เมืองที่เกิดเหตุการณ์
- **รัฐ**: รัฐ/จังหวัดที่เกิดเหตุการณ์
- **ประเทศ**: ประเทศที่เกิดเหตุการณ์
- **รหัสไปรษณีย์**: รหัสไปรษณีย์/ZIP ที่เกิดเหตุการณ์

**เวลาที่เกิดเหตุการณ์**:
- **เกิดขึ้นเมื่อ**: เวลาที่เหตุการณ์เกิดขึ้นจริง (เวลาของผู้ให้บริการ)
- **สร้างเมื่อ**: เวลาที่เหตุการณ์ถูกบันทึกใน Spwig (เวลาของระบบ)

**ข้อมูลเพิ่มเติม**:
- **ข้อมูลดิบ**: ข้อมูล JSON ทั้งหมดจาก API ของผู้ให้บริการ
- **การส่งของ**: ID การส่งของที่เกี่ยวข้อง

---

## ประเภทของสถานะเหตุการณ์

**in_transit**: พัสดุกำลังเคลื่อนที่ผ่านเครือข่ายของผู้ให้บริการ
- ตัวอย่าง: "ออกจากศูนย์", "มาถึงศูนย์กลาง", "กำลังส่งไปยังศูนย์ถัดไป"

**out_for_delivery**: พัสดุอยู่ในยานพาหนะสำหรับจัดส่ง
- ตัวอย่าง: "กำลังจะส่งของ", "อยู่ในยานพาหนะ"

**delivered**: พัสดุถูกส่งมอบอย่างสำเร็จ
- ตัวอย่าง: "ส่งถึงประตูหน้าบ้าน", "ทิ้งไว้ที่แผนกต้อนรับ", "ส่งมอบให้ผู้รับ"

**exception**: ปัญหาการจัดส่งที่ต้องการความสนใจ
- ตัวอย่าง: "ความล่าช้าจากสภาพอากาศ", "ที่อยู่ไม่ถูกต้อง", "การส่งของล้มเหลว"

**failed**: การจัดส่งล้มเหลวอย่างถาวร
- ตัวอย่าง: "ไม่สามารถส่งถึงที่อยู่ที่ระบุได้", "ผู้รับปฏิเสธการรับของ"

**returned**: พัสดุกำลังคืนคืนให้ผู้ส่ง
- ตัวอย่าง: "เริ่มต้นกระบวนการคืนสินค้า", "พัสดุกำลังคืน"

## วิธีการสร้างเหตุการณ์การติดตาม

### อัตโนมัติ (การแจ้งเตือน webhook ของผู้ให้บริการ)

**กระบวนการทำงาน**:
1. ผู้ให้บริการสแกนพัสดุ (การออกเดินทาง, การมาถึง, การส่งมอบ)
2. ผู้ให้บริการส่ง webhook ไปยังจุดปลายทาง webhook ของ Spwig
3. webhook ถูกบันทึกในตาราง WebhookLog
4. ระบบวิเคราะห์ payload ของ webhook
5. สร้าง TrackingEvent ด้วยข้อมูลที่ได้รับ
6. ส่งการแจ้งเตือนผ่านอีเมลให้ลูกค้า (ถ้าตั้งค่าไว้)

**ประโยชน์**:
- อัปเดตแบบเรียลไทม์ (ไม่จำเป็นต้องมีการสอบถาม)
- เวลาที่ถูกต้องจากผู้ให้บริการ
- ประวัติเหตุการณ์ทั้งหมดถูกจัดการโดยอัตโนมัติ

### ด้วยตนเอง (การบันทึกของผู้ค้า)

**กระบวนการทำงาน**:
1. ไปที่รายละเอียดการส่งของ
2. คลิก "เพิ่มเหตุการณ์การติดตาม"
3. เลือกสถานะจาก dropdown
4. ป้อนคำอธิบาย
5. ตัวเลือก: ป้อนข้อมูลสถานที่
6. ตั้งค่า timestamp ที่เกิดขึ้นจริง
7. บันทึก

**กรณีการใช้งาน**:
- ผู้ให้บริการที่ไม่รองรับ webhook
- การแก้ไขการส่งของด้วยตนเอง
- การจัดส่งภายในพื้นที่ (ไม่ใช่ผู้ให้บริการ)
- การอัปเดตสถานะภายใน

## ลำดับการแสดงเหตุการณ์

เหตุการณ์จะแสดงในลำดับ **ย้อนหลังตามเวลา** (ใหม่สุดก่อน):

**ตัวอย่างการแสดงผล**:
```
13 ก.พ. 2026 10:30 น. - ได้รับสินค้า (Brooklyn, NY)
13 ก.พ. 2026 08:15 น. - กำลังจะส่งของ (Brooklyn, NY)
12 ก.พ. 2026 23:45 - มาถึงศูนย์ท้องถิ่น (Brooklyn, NY)
12 ก.พ. 2026 18:30 - กำลังส่ง (Newark, NJ)
12 ก.พ. 2026 14:15 - ออกจากจุดเริ่มต้น (Philadelphia, PA)
12 ก.พ. 2026 09:00 - ถูกสแกน (Philadelphia, PA)
```

## ความเป็นไปได้ในการมองเห็นของลูกค้า

เหตุการณ์การติดตามที่แสดงให้ลูกค้าเห็นใน:

**อีเมลยืนยันการสั่งซื้อ**:
- สถานะเหตุการณ์ล่าสุด
- วันที่คาดการณ์การจัดส่ง
- ลิงก์การติดตาม

**บัญชีลูกค้า > รายละเอียดคำสั่งซื้อ**:
- ไทม์ไลน์เหตุการณ์ทั้งหมด
- คำอธิบายเหตุการณ์
- ประวัติสถานที่
- เวลาที่เกิดเหตุการณ์

**หน้าติดตาม** (ถ้าเปิดใช้งาน):
- URL การติดตามเฉพาะ
- ไทม์ไลน์แบบภาพ
- โลโก้ผู้ให้บริการ
- แผนที่การจัดส่ง (ถ้ามีข้อมูลสถานที่)

## การกรองเหตุการณ์การติดตาม

**ตัวกรองที่มีประโยชน์**:
- **การส่งของ**: ดูเหตุการณ์สำหรับการส่งของเฉพาะ
- **สถานะ**: กรองตามประเภทเหตุการณ์ (ได้รับสินค้า, in_transit, ฯลฯ)
- **ช่วงเวลา**: เหตุการณ์ภายในช่วงเวลาที่กำหนด
- **สถานที่**: เหตุการณ์ในเมือง/รัฐเฉพาะ

**กรณีการใช้งาน**:
- "แสดงเหตุการณ์ที่ได้รับสินค้าทั้งหมดในวันนี้"
- "ค้นหาเหตุการณ์ที่เกิดขึ้นในสัปดาห์ที่แล้ว"
- "ติดตามการส่งของที่กำลังส่งอยู่"

---

## ข้อมูลดิบ (การตรวจสอบ)

**ฟิลด์ข้อมูลดิบ**:
- จัดเก็บการตอบสนองจาก API ของผู้ให้บริการทั้งหมดเป็น JSON
- มีประโยชน์ในการตรวจสอบปัญหา webhook
- ประกอบด้วยข้อมูลเฉพาะผู้ให้บริการ

**ตัวอย่างข้อมูลดิบ** (FedEx):
```json
{
  "event_type": "OD",
  "event_description": "Out for delivery",
  "timestamp": "2026-02-13T08:15:00Z",
  "location": {
    "city": "Brooklyn",
    "state": "NY",
    "postal_code": "11201",
    "country": "US"
  },
  "delivery_signature": null,
  "estimated_delivery": "2026-02-13T17:00:00Z"
}
```

**กรณีที่ควรตรวจสอบข้อมูลดิบ**:
- คำอธิบายเหตุการณ์ไม่ชัดเจน
- ขาดข้อมูลสถานที่
- เกิดข้อผิดพลาดการประมวลผล webhook
- การติดต่อผู้ให้บริการเพื่อแก้ไขปัญหา

## เวลาเหตุการณ์

**Occurred At** ต่างจาก **Created At**:

**Occurred At**: เวลาที่เหตุการณ์ของผู้ให้บริการเกิดขึ้นจริง
- ตัวอย่าง: พัสดุถูกสแกนเวลา 10:30 น.

**Created At**: เวลาที่ Spwig ได้รับ webhook
- ตัวอย่าง: ได้รับ webhook เวลา 10:32 น. (ความล่าช้า 2 นาที)

**เหตุใดจึงแตกต่างกัน?**:
- ความล่าช้าของเครือข่าย
- การประมวลผลแบบแบตช์ของผู้ให้บริการ
- ความล่าช้าจากการส่ง webhook อีกครั้ง

**ใช้ Occurred At สำหรับการแสดงผลลูกค้า** - สะท้อนความก้าวหน้าของการจัดส่งที่แม่นยำกว่า

## เคล็ดลับ

- **เหตุการณ์เป็นแบบอ่านเพียงอย่างเดียว** - ไม่สามารถแก้ไขหลังจากสร้าง (เพื่อรักษาความสมบูรณ์ของการตรวจสอบ)
- **ตรวจสอบข้อมูลดิบเพื่อดูรายละเอียด** - มีข้อมูลมากกว่าที่แสดงในฟิลด์
- **ตรวจสอบความล่าช้าของ webhook** - ความล่าช้าที่มากระหว่าง occurred_at และ created_at อาจบ่งบอกถึงปัญหา webhook
- **ใช้กับการสนับสนุนลูกค้า** - ไทม์ไลน์เหตุการณ์ช่วยวินิจฉัยปัญหาการจัดส่ง
- **ติดตามรูปแบบการจัดส่ง** - วิเคราะห์เวลาเหตุการณ์เพื่อประเมินประสิทธิภาพผู้ให้บริการ
- **ตั้งค่าการแจ้งเตือน** - อีเมลลูกค้าอัตโนมัติเมื่อเกิดเหตุการณ์สำคัญ (out_for_delivery, delivered)
- **อย่าลบเหตุการณ์** - รักษาประวัติการตรวจสอบทั้งหมด
- **ตรวจสอบ WebhookLog สำหรับข้อผิดพลาด** - เหตุการณ์หายไปอาจบ่งบอกถึงข้อผิดพลาดการประมวลผล webhook
- **ข้อมูลสถานที่แตกต่างกันตามผู้ให้บริการ** - ผู้ให้บริการบางรายให้ข้อมูลสถานที่ที่ละเอียด ผู้ให้บริการบางรายให้ข้อมูลน้อย
- **เหตุการณ์ข้อผิดพลาดต้องได้รับความสนใจ** - ตรวจสอบและติดตามปัญหาการจัดส่งที่เกิดข้อผิดพลาด
