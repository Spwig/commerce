---
title: บันทึกเวิร์กโฮก
---

บันทึกเวิร์กโฮกให้การติดตามการตรวจสอบอย่างถาวรของคำขอเวิร์กโฮกที่เข้ามาทั้งหมด—จับข้อมูลวิธีการร้องขอ จุดปลาย URL หัวข้อ ข้อมูล สถานะการประมวลผล (รอประมวลผล/ประมวลผลเสร็จ/ล้มเหลว) และการตอบกลับ ทุกเวิร์กโฮกจะถูกบันทึกก่อนการประมวลผลเพื่อให้มั่นใจว่าไม่มีเหตุการณ์ใดหายไปหากการประมวลผลล้มเหลว บันทึกช่วยในการแก้ไขปัญหาการบูรณาการเวิร์กโฮก ตรวจสอบความน่าเชื่อถือของ API ผู้ให้บริการ และสร้างไทม์ไลน์การจัดส่งเพื่อการสนับสนุนลูกค้า

หน้าผู้ดูแลระบบอ่านได้เพียงอย่างเดียวนี้ช่วยในการแก้ไขปัญหาการล้มเหลวของเวิร์กโฮกและตรวจสอบความสมบูรณ์ของระบบบูรณาการผู้ให้บริการ

## โครงสร้างบันทึกเวิร์กโฮก

แต่ละรายการบันทึกจะบันทึก:

**รายละเอียดคำขอ**:
- **คีย์ผู้ให้บริการ**: ผู้ให้บริการที่ส่งเวิร์กโฮก (fedex, ups, dhl)
- **จุดปลาย**: เส้นทาง URL เวิร์กโฮก (ตัวอย่างเช่น `/webhooks/shipping/fedex/`)
- **วิธี**: วิธี HTTP (มักเป็น POST)
- **หัวข้อ**: หัวข้อคำขอ (JSON)
- **ข้อมูล**: ร่างกายคำขอ (JSON)

**การประมวลผล**:
- **สถานะการประมวลผล**: รอประมวลผล, ประมวลผลเสร็จ, ล้มเหลว
- **ข้อความข้อผิดพลาด**: สาเหตุการล้มเหลว (หากสถานะ=ล้มเหลว)
- **การตอบกลับ**: การตอบกลับ HTTP ที่ส่งไปยังผู้ให้บริการ
- **รหัสสถานะการตอบกลับ**: 200, 400, 500, ฯลฯ

**เวลาที่บันทึก**:
- **ได้รับเมื่อ**: เวิร์กโฮกถูกส่งมาถึง
- **ประมวลผลเมื่อ**: เมื่อการประมวลผลเสร็จสิ้น

---

## ค่าสถานะการประมวลผล

**รอประมวลผล**: เวิร์กโฮกได้รับแล้ว กำลังรอการประมวลผล
- ปกติเป็นช่วงสั้นหลังจากได้รับ
- หากติดอยู่ในสถานะรอ แสดงถึงการค้างค้างในคิวการประมวลผล

**ประมวลผลเสร็จ**: เวิร์กโฮกได้รับการประมวลผลสำเร็จ
- สร้าง TrackingEvent
- ส่งการแจ้งเตือนลูกค้า (หากเกี่ยวข้อง)
- ส่งการตอบกลับ HTTP 200 ไปยังผู้ให้บริการ

**ล้มเหลว**: การประมวลผลเวิร์กโฮกล้มเหลว
- ตรวจสอบข้อความข้อผิดพลาดเพื่อดูสาเหตุ
- สาเหตุที่พบบ่อย: JSON ไม่ถูกต้อง ไม่มีการส่งของ เหตุการณ์ซ้ำ

---

## ลำดับการดำเนินการเวิร์กโฮก

**กระบวนการทำงานปกติ**:
```
1. ผู้ให้บริการสแกนแพ็กเกจ
   ↓
2. ผู้ให้บริการส่ง POST ไปยังจุดปลายเวิร์กโฮกของ Spwig
   ↓
3. Spwig สร้าง WebhookLog (status=pending)
   ↓
4. ผู้ทำงานในพื้นหลังประมวลผลเวิร์กโฮก
   ↓
5. วิเคราะห์ข้อมูล JSON
   ↓
6. ค้นหา Shipment ที่ตรงกัน (โดยใช้หมายเลขติดตาม)
   ↓
7. สร้าง TrackingEvent
   ↓
8. อัปเดต WebhookLog (status=processed)
   ↓
9. ส่งการตอบกลับ HTTP 200 ไปยังผู้ให้บริการ
```

**สถานการณ์ล้มเหลว**:
- **JSON ไม่ถูกต้อง**: ผู้ให้บริการส่งข้อมูลที่ผิดรูปแบบ → status=ล้มเหลว, ข้อความข้อผิดพลาด="ข้อผิดพลาดการวิเคราะห์ JSON"
- **ไม่มีการส่งของ**: หมายเลขติดตามไม่ตรงกับการส่งของใด → status=ล้มเหลว, ข้อความข้อผิดพลาด="ไม่พบการส่งของ"
- **ซ้ำ**: เหตุการณ์มีอยู่แล้ว → status=ล้มเหลว, ข้อความข้อผิดพลาด="เหตุการณ์ซ้ำ"

---

## การแก้ไขปัญหาการล้มเหลวของเวิร์กโฮก

**ขั้นตอนทีละขั้น**:

**1. กรองด้วย Status=ล้มเหลว**
- ไปที่ จัดส่ง > บันทึกเวิร์กโฮก
- กรอง: สถานะการประมวลผล = "ล้มเหลว"
- ตรวจสอบความล้มเหลวล่าสุด

**2. ตรวจสอบข้อความข้อผิดพลาด**
- คลิกรายการบันทึก
- อ่านฟิลด์ error_message
- ข้อผิดพลาดที่พบบ่อย:
  - "ไม่พบการส่งของ" → ไม่ตรงกับหมายเลขติดตาม
  - "ข้อผิดพลาดการแปลง JSON" → ผู้ให้บริการส่ง JSON ที่ไม่ถูกต้อง
  - "ขาดฟิลด์ที่จำเป็น" → ข้อมูลขาดฟิลด์ที่คาดไว้

**3. ตรวจสอบข้อมูล**
- ดูข้อมูล JSON ที่ไม่ผ่านการประมวลผล
- ตรวจสอบโครงสร้างตรงกับรูปแบบที่คาดไว้หรือไม่
- ตรวจสอบว่ามีฟิลด์ที่ขาดหายไปหรือไม่ (tracking_id, event_type, ฯลฯ)

**4. ตรวจสอบว่ามีการส่งของอยู่หรือไม่**
- ดึงหมายเลขติดตามจากข้อมูล
- ค้นหาการส่งของด้วยหมายเลขติดตาม
- ตรวจสอบว่าการส่งของมีอยู่และใช้ผู้ให้บริการที่ถูกต้อง

**5. ตรวจสอบการตั้งค่าผู้ให้บริการ**
- ตรวจสอบว่าบัญชีผู้ให้บริการยังเปิดใช้งานอยู่
- ยืนยันว่า URL จุดปลายเวิร์กโฮกถูกต้อง
- ทดสอบคุณสมบัติ API ผู้ให้บริการ

**6. ลองประมวลผลใหม่** (หากเกี่ยวข้อง)
- บางตัวประมวลผลเวิร์กโฮกสนับสนุนการลองอีกครั้งด้วยตนเอง
- แก้ไขปัญหาหลักก่อน
- ลองประมวลผลเวิร์กโฮกที่ล้มเหลวใหม่

---

## ปัญหาเวิร์กโฮกที่พบบ่อย

**ปัญหาที่ 1: "ไม่พบการส่งของ"**

**สาเหตุ**: หมายเลขติดตามในเวิร์กโฮกไม่ตรงกับการส่งของใด
- สะกดผิดเมื่อสร้างการส่งของ
- เวิร์กโฮกของบัญชีอื่น
- การส่งของถูกลบก่อนที่เวิร์กโฮกจะได้รับ

**วิธีแก้ไข**:
- ตรวจสอบการสะกดหมายเลขติดตาม
- ตรวจสอบว่าการส่งของผู้ให้บริการตรงกับผู้ให้บริการเวิร์กโฮก
- สร้างการส่งของใหม่หากจำเป็น

---

**ปัญหาที่ 2: "ข้อผิดพลาดการแปลง JSON"**

**สาเหตุ**: ผู้ให้บริการส่ง JSON ที่ผิดรูปแบบ
- เกิดขึ้นน้อย แต่เกี่ยวข้องกับข้อบกพร่องของ API ผู้ให้บริการ
- ปัญหาการเข้ารหัสตัวอักษร

**วิธีแก้ไข**:
- ติดต่อผู้ให้บริการสนับสนุนพร้อมข้อมูล JSON ดิบ
- ตรวจสอบหัวข้อสำหรับการเข้ารหัส charset
- ตรวจสอบ URL จุดปลายในแดชบอร์ดผู้ให้บริการ

---

**ปัญหาที่ 3: เวิร์กโฮกซ้ำ**

**สาเหตุ**: ผู้ให้บริการส่งเหตุการณ์เดียวกันหลายครั้ง
- ระบบมีระบบอัตโนมัติในการปฏิเสธการซ้ำ (เป็นพฤติกรรมปกติ)
- ตรวจสอบว่า response_status_code เป็น 200
- หากยังคงมีปัญหา ติดต่อผู้ให้บริการสนับสนุน

---

**ปัญหาที่ 4: ขาดเวิร์กโฮก**

**สาเหตุ**: เวิร์กโฮกที่คาดไว้ไม่เคยได้รับ
- ผู้ให้บริการไม่ได้ส่ง (การสแกนพลาด)
- URL จุดปลายเวิร์กโฮกตั้งค่าผิดในแดชบอร์ดผู้ให้บริการ
- กำแพงไฟร์วอลล์บล็อกคำขอ

**วิธีแก้ไข**:
- ตรวจสอบการตั้งค่าเวิร์กโฮกในแดชบอร์ดผู้ให้บริการ
- ตรวจสอบว่า URL จุดปลายเปิดใช้งานและเข้าถึงได้
- ทดสอบจุดปลายด้วย curl/Postman
- ตรวจสอบกฎไฟร์วอลล์เซิร์ฟเวอร์

---

## การตั้งค่า URL เวิร์กโฮก

**URL เวิร์กโฮกทั่วไป**:
```
FedEx: https://yourdomain.com/webhooks/shipping/fedex/
UPS: https://yourdomain.com/webhooks/shipping/ups/
DHL: https://yourdomain.com/webhooks/shipping/dhl/
```

**การตั้งค่าในแดชบอร์ดผู้ให้บริการ**:
1. ล็อกอินเข้าสู่พอร์ทัลผู้พัฒนาผู้ให้บริการ
2. ไปที่การตั้งค่าเวิร์กโฮก
3. ใส่ URL เวิร์กโฮกของ Spwig
4. เลือกเหตุการณ์ที่ต้องการติดตาม (อัปเดตการติดตาม, การจัดส่ง, ข้อผิดพลาด)
5. บันทึกการตั้งค่า
6. ทดสอบเวิร์กโฮกด้วยเครื่องมือทดสอบของผู้ให้บริการ

**ความปลอดภัย**:
- เวิร์กโฮกต้องใช้ HTTPS (ไม่ใช่ HTTP)
- บางผู้ให้บริการลงนามคำขอ (ตรวจสอบลายเซ็น)
- รายการ IP ที่อนุญาต (หากผู้ให้บริการให้ช่วง IP คงที่)

---

## การตรวจสอบสุขภาพเวิร์กโฮก

**ตัวชี้วัดหลัก**:

**อัตราความสำเร็จ**:
```
อัตราความสำเร็จ = (จำนวนที่ประมวลผลเสร็จ / จำนวนทั้งหมด) × 100%

เป้าหมาย: >98%
```

**เวลาการประมวลผล**:
```
เวลาเฉลี่ย = เวลาประมวลเสร็จ - เวลาได้รับ

เป้าหมาย: <2 วินาที
```

**รูปแบบการล้มเหลว**:
- การเพิ่มขึ้นของความล้มเหลวอย่างกะทันหัน → การเปลี่ยนแปลงหรือปัญหาของ API ผู้ให้บริการ
- การล้มเหลวอย่างต่อเนื่อง "ไม่พบการส่งของ" → ปัญหาการซิงค์หมายเลขติดตาม
- ทุกเวิร์กโฮกล้มเหลว → ปัญหาการตั้งค่าจุดปลาย

**กลยุทธ์การตรวจสอบ**:
- ตรวจสอบอัตราความล้มเหลวทุกวัน
- แจ้งเตือนหากอัตราความล้มเหลว >5%
- ตรวจสอบข้อความข้อผิดพลาดรายสัปดาห์
- เปรียบเทียบกับสถานะของผู้ให้บริการ

---

## การเก็บรักษาเวิร์กโฮก

**บันทึกเป็นถาวร** - ไม่เคยถูกลบอัตโนมัติ

**เหตุผลที่เป็นถาวร**:
- ปฏิบัติตามข้อกำหนดการตรวจสอบ
- สนับสนุนลูกค้า (สร้างไทม์ไลน์การจัดส่ง)
- แก้ไขข้อพิพาท
- การตรวจสอบเวิร์กโฮก

**การเก็บข้อมูล**: บันทึกเก็บอย่างมีประสิทธิภาพ (JSON ที่บีบอัด)

---

## เคล็ดลับ

- **เวิร์กโฮกเป็นบันทึกตรวจสอบถาวร** - ไม่เคยลบ แม้จะประมวลผลสำเร็จ
- **ตรวจสอบเวิร์กโฮกที่ล้มเหลวทุกวัน** - จับปัญหาการบูรณาการได้เร็วขึ้น
- **ตรวจสอบความล่าช้าในการประมวลผล** - เวลาแย่แสดงถึงปัญหาประสิทธิภาพ
- **บันทึกข้อมูลดิบ** - เป็นสิ่งสำคัญสำหรับการตรวจสอบการเปลี่ยนแปลง API ผู้ให้บริการ
- **ทดสอบการตั้งค่าจุดปลาย** - ใช้เครื่องมือทดสอบของผู้ให้บริการเพื่อยืนยันการตั้งค่า
- **เปิดใช้งานการลงนามเวิร์กโฮก** - ตรวจสอบว่าคำขอมาจากผู้ให้บริการจริง
- **เพิ่ม IP ของผู้ให้บริการ** - หากผู้ให้บริการให้ช่วง IP คงที่
- **ตั้งค่าการแจ้งเตือน** - แจ้งเตือนเมื่ออัตราความล้มเหลวเกินเกณฑ์
- **เปรียบเทียบกับสถานะผู้ให้บริการ** - ช่องว่างเวิร์กโฮกอาจบ่งบอกถึงปัญหาของผู้ให้บริการ
- **บันทึกรูปแบบข้อมูลของผู้ให้บริการ** - ช่วยเมื่อผู้ให้บริการอัปเดต API
- **รักษา URL เวิร์กโฮกให้คงที่** - การเปลี่ยน URL ต้องอัปเดตแดชบอร์ดผู้ให้บริการ
