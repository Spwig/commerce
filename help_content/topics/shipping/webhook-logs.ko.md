---
title: 웹후크 로그
---

웹후크 로그는 모든 입구 운송사 웹후크 요청에 대한 영구적인 감사 추적을 제공합니다. 요청 메서드, 엔드포인트 URL, 헤더, 페이로드, 처리 상태(대기/처리됨/실패), 응답을 기록합니다. 모든 웹후크는 처리 전에 로그로 기록되어 처리 실패 시 이벤트가 손실되지 않도록 합니다. 로그는 웹후크 통합 문제 디버깅, 운송사 API 신뢰성 모니터링, 고객 지원을 위한 배송 시간선 재구성에 도움이 됩니다.

이 읽기 전용 관리 페이지는 웹후크 실패를 해결하고 운송사 통합 상태를 확인하는 데 도움이 됩니다.

## 웹후크 로그 구조

각 로그 항목은 다음을 기록합니다:

**요청 세부 정보**:
- **프로바이더 키**: 웹후크를 보낸 운송사(fedex, ups, dhl)
- **엔드포인트**: 웹후크 URL 경로(예: `/webhooks/shipping/fedex/`)
- **메서드**: HTTP 메서드(일반적으로 POST)
- **헤더**: 요청 헤더(JSON)
- **페이로드**: 요청 본문(JSON)

**처리**:
- **처리 상태**: 대기, 처리됨, 실패
- **에러 메시지**: 실패 원인(상태=실패 시)
- **응답**: 운송사에게 보낸 HTTP 응답
- **응답 상태 코드**: 200, 400, 500 등

**타임스탬프**:
- **수신 시간**: 웹후크가 도착한 시간
- **처리 시간**: 처리가 완료된 시간

---

## 처리 상태 값

**대기**: 웹후크 수신, 처리 대기
- 수신 직후 일시적인 상태
- 처리 대기 상태에 멈춰 있다면 처리 대기열 백로그를 나타냅니다.

**처리됨**: 웹후크 성공적으로 처리됨
- TrackingEvent 생성됨
- 고객 알림 전송(적용 시)
- 운송사에게 HTTP 200 응답 전송

**실패**: 웹후크 처리 실패
- error_message를 확인하여 원인 파악
- 일반적인 원인: 유효하지 않은 JSON, 알 수 없는 배송, 중복된 이벤트

---

## 웹후크 흐름

**정상 작업 흐름**:
```
1. 운송사가 패키지 스캔
   ↓
2. 운송사가 Spwig 웹후크 엔드포인트로 POST 전송
   ↓
3. Spwig이 WebhookLog 생성 (상태=대기)
   ↓
4. 백그라운드 워커가 웹후크 처리
   ↓
5. JSON 페이로드 파싱
   ↓
6. 추적 번호로 일치하는 Shipment 찾기
   ↓
7. TrackingEvent 생성
   ↓
8. WebhookLog 업데이트 (상태=처리됨)
   ↓
9. 운송사에게 HTTP 200 응답 전송
```

**실패 시나리오**:
- **유효하지 않은 JSON**: 운송사가 잘못된 데이터 전송 → 상태=실패, 에러="JSON 파싱 오류"
- **알 수 없는 배송**: 추적 번호가 어떤 배송과도 일치하지 않음 → 상태=실패, 에러="배송 찾을 수 없음"
- **중복**: 이벤트가 이미 존재 → 상태=실패, 에러="중복된 이벤트"

---

## 웹후크 실패 디버깅

**단계별**:

**1. 상태=실패 필터링**
- Shipping > Webhook Logs로 이동
- 필터: 처리 상태 = "실패"
- 최근 실패 사항 검토

**2. 에러 메시지 확인**
- 로그 항목 클릭
- error_message 필드 읽기
- 일반적인 오류:
  - "배송 찾을 수 없음" → 추적 번호 불일치
  - "JSON 디코딩 오류" → 운송사가 유효하지 않은 JSON 전송
  - "필수 필드 누락" → 페이로드가 예상되는 데이터 누락

**3. 페이로드 검사**
- 원본 JSON 페이로드 보기
- 구조가 예상되는 형식과 일치하는지 확인
- 누락된 필드 확인(tracking_id, event_type 등)

**4. 배송 존재 여부 확인**
- 페이로드에서 추적 번호 추출
- 추적 번호로 배송 검색
- 배송이 존재하고 올바른 운송사 사용 여부 확인

**5. 프로바이더 구성 확인**
- 프로바이더 계정이 활성화되었는지 확인
- 웹후크 엔드포인트 URL이 올바른지 확인
- 프로바이더 API 인증 정보 확인

**6. 처리 재시도** (적용 시)
- 일부 웹후크 처리기에서 수동 재시도 지원
- 기본 문제 먼저 수정
- 실패한 웹후크 재시도

## 일반적인 웹후크 문제

**문제 1: "배송 찾을 수 없음"**

**원인**: 웹후크의 추적 번호가 어떤 배송과도 일치하지 않음
- 배송 생성 시 오타
- 다른 계정의 웹후크
- 웹후크 수신 전 배송 삭제

**해결 방법**:
- 추적 번호 철자가 정확한지 확인
- 배송의 운송사가 웹후크 프로바이더와 일치하는지 확인
- 필요시 배송 다시 생성

---

**문제 2: "JSON 디코딩 오류"**

**원인**: 운송사가 잘못된 JSON 전송
- 드물지만, 일반적으로 운송사 API 버그
- 문자 인코딩 문제

**해결 방법**:
- 원본 페이로드와 함께 운송사 지원 담당자 연락
- 헤더에서 charset 인코딩 확인
- 운송사 대시보드에서 엔드포인트 URL 확인

---

**문제 3: 중복된 웹후크**

**원인**: 운송사가 동일한 이벤트 여러 번 전송
- 재시도 로직(운송사가 200 응답을 받지 못함)
- 운송사 버그

**해결 방법**:
- 시스템이 자동으로 중복 거부(정상적인 동작)
- response_status_code가 200인지 확인
- 지속적으로 발생 시 운송사 지원 담당자 연락

---

**문제 4: 누락된 웹후크**

**원인**: 예상된 웹후크가 절대 수신되지 않음
- 운송사가 전송하지 않음(스캔 누락)
- 운송사 대시보드에서 웹후크 엔드포인트 잘못 설정
- 방화벽이 요청 차단

**해결 방법**:
- 운송사 대시보드 웹후크 설정 확인
- 엔드포인트 URL이 공개되고 접근 가능한지 확인
- curl/Postman으로 엔드포인트 테스트
- 서버 방화벽 규칙 확인

---

## 웹후크 엔드포인트 구성

**일반적인 웹후크 URL**:
```
FedEx: https://yourdomain.com/webhooks/shipping/fedex/
UPS: https://yourdomain.com/webhooks/shipping/ups/
DHL: https://yourdomain.com/webhooks/shipping/dhl/
```

**운송사 대시보드 설정**:
1. 운송사 개발자 포털에 로그인
2. 웹후크 설정으로 이동
3. Spwig 웹후크 URL 입력
4. 구독할 이벤트 선택(추적 업데이트, 배송, 예외)
5. 구성 저장
6. 운송사 테스트 도구로 웹후크 테스트

**보안**:
- 웹후크는 HTTPS(HTTP 아님) 필요
- 일부 운송사가 요청에 서명(서명 확인)
- IP 화이트리스트(운송사가 정적 IP 제공 시)

## 웹후크 건강 모니터링

**핵심 지표**:

**성공률**:
```
성공률 = (처리됨 / 총) × 100%

목표: >98%
```

**처리 시간**:
```
평균 시간 = 처리 시간 - 수신 시간

목표: <2초
```

**실패 패턴**:
- 갑작스럽게 실패 증가 → 운송사 API 변경 또는 장애
- "배송 찾을 수 없음" 지속 → 추적 번호 동기화 문제
- 모든 웹후크 실패 → 엔드포인트 구성 문제

**모니터링 전략**:
- 매일 실패율 확인
- 실패율 >5% 시 경고
- 매주 에러 메시지 검토
- 운송사 상태 페이지와 비교

---

## 웹후크 보존

**로그는 영구적** - 자동으로 삭제되지 않음

**영구성 이유**:
- 감사 준수
- 고객 지원(배송 시간선 재구성)
- 분쟁 해결
- 웹후크 디버깅

**저장**: 로그는 효율적으로 저장(압축된 JSON)

---

## 팁

- **웹후크는 영구 감사 로그** - 처리 성공 여부와 무관하게 삭제하지 마세요
- **매일 실패한 웹후크 확인** - 통합 문제를 조기에 발견하세요
- **처리 지연 모니터링** - 긴 지연은 성능 문제를 나타냅니다
- **원본 페이로드 저장** - 운송사 API 변경 시 디버깅에 필수적입니다
- **엔드포인트 구성 테스트** - 운송사 테스트 도구로 설정을 확인하세요
- **웹후크 서명 활성화** - 요청이 실제로 운송사에서 온 것인지 확인하세요
- **운송사 IP 화이트리스트** - 운송사가 정적 IP 범위를 제공하는 경우
- **경고 설정** - 실패율 임계값을 초과할 때 알림을 받으세요
- **운송사 상태와 비교** - 웹후크 간격이 운송사 장애를 나타낼 수 있습니다
- **운송사 페이로드 형식 문서화** - 운송사가 API를 업데이트할 때 도움이 됩니다
- **웹후크 URL을 안정적으로 유지** - URL 변경은 운송사 대시보드 업데이트가 필요합니다
