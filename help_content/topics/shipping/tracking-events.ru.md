---
title: Отслеживание событий
---

События отслеживания записывают контрольные точки статуса отправки на протяжении жизненного цикла доставки — каждое событие фиксирует статус (в пути, на пути к получателю, доставлено), временной штамп, местоположение, описание и сырые данные от перевозчика. События создаются автоматически через вебхуки перевозчика или вручную продавцами. Покупатели видят историю событий отслеживания в своем аккаунте и в письмах подтверждения заказа, обеспечивая прозрачность доставки в реальном времени.

Эта административная страница отображает только для чтения историю событий для целей аудита и поддержки клиентов.

## Структура события отслеживания

Каждое событие содержит:

**Информация о статусе**:
- **Статус**: in_transit, out_for_delivery, delivered, exception, failed, returned
- **Описание**: Человекочитаемый статус (например, "Пакет прибыл на сортировочную площадку")
- **Код статуса перевозчика**: Оригинальный статус перевозчика (например, "DEP" для ушедшего)

**Данные о местоположении**:
- **Город**: Город расположения события
- **Штат**: Штат/провинция расположения события
- **Страна**: Страна расположения события
- **Почтовый индекс**: Почтовый индекс/ZIP-код расположения события

**Временные метки**:
- **Произошло в**: Время, когда событие действительно произошло (время перевозчика)
- **Создано в**: Время, когда событие было записано в Spwig (время системы)

**Метаданные**:
- **Сырые данные**: Полный ответ в формате JSON от API перевозчика
- **Отправка**: Связанный идентификатор отправки

---

## Типы статусов событий

**in_transit**: Пакет перемещается по сети перевозчика
- Примеры: "Ушел с площадки", "Прибыл на хаб", "В пути к следующей площадке"

**out_for_delivery**: Пакет на транспортном средстве доставки
- Примеры: "На пути к получателю", "На транспортном средстве доставки"

**delivered**: Пакет успешно доставлен
- Примеры: "Доставлен на входную дверь", "Оставлен в приемной", "Передан получателю"

**exception**: Проблема с доставкой, требующая внимания
- Примеры: "Задержка из-за погоды", "Неправильный адрес", "Попытка доставки завершилась неудачей"

**failed**: Доставка завершилась неудачно
- Примеры: "Невозможно доставить по указанному адресу", "Отказ от получения получателем"

**returned**: Пакет возвращается отправителю
- Примеры: "Инициировано возвращение отправителю", "Пакет возвращается"

## Как создаются события отслеживания

### Автоматически (через вебхуки перевозчика)

**Рабочий процесс**:
1. Перевозчик сканирует пакет (отправка, прибытие, доставка)
2. Перевозчик отправляет вебхук в конечную точку вебхука Spwig
3. Вебхук записывается в таблицу WebhookLog
4. Система парсит полезную нагрузку вебхука
5. Создается событие отслеживания с извлеченными данными
6. Отправляется уведомление по электронной почте клиенту (если настроено)

**Преимущества**:
- Обновления в реальном времени (не требуется опрос)
- Точное время от перевозчика
- Полная история событий автоматически поддерживается

### Вручную (ввод продавцом)

**Рабочий процесс**:
1. Перейдите к деталям отправки
2. Нажмите "Добавить событие отслеживания"
3. Выберите статус из выпадающего списка
4. Введите описание
5. Необязательно: Введите данные о местоположении
6. Установите временной штамп occurred_at
7. Сохраните

**Сценарии использования**:
- Перевозчики без поддержки вебхуков
- Ручная корректировка отправки
- Локальная доставка (не через перевозчика)
- Внутренние обновления статуса

## Порядок отображения событий

События отображаются в **обратном хронологическом порядке** (сначала новые):

**Пример отображения**:
```
13 февраля 2026, 10:30 AM - Доставлено (Бруклин, Нью-Йорк)
13 февраля 2026, 08:15 AM - На пути к получателю (Бруклин, Нью-Йорк)
12 февраля 2026, 23:45 - Прибыл на локальную площадку (Бруклин, Нью-Йорк)
12 февраля 2026, 18:30 - В пути (Ньюарк, Нью-Джерси)
12 февраля 2026, 14:15 - Ушел из пункта отправления (Филадельфия, Пенсильвания)
12 февраля 2026, 09:00 - Получен (Филадельфия, Пенсильвания)
```

## Видимость событий для клиентов

События отслеживания отображаются клиентам в:

**Письме подтверждения заказа**:
- Последний статус события
- Предполагаемая дата доставки
- Ссылка для отслеживания

**Аккаунт клиента > Детали заказа**:
- Полный временной график событий
- Описания событий
- История местоположения
- Временные метки

**Страница отслеживания** (если включено):
- Дополнительная ссылка для отслеживания
- Визуальный временной график
- Логотип перевозчика
- Карта доставки (если доступны данные о местоположении)

## Фильтрация событий отслеживания

**Полезные фильтры**:
- **Отправка**: Просмотр событий для конкретной отправки
- **Статус**: Фильтрация по типу события (доставлено, in_transit и т.д.)
- **Диапазон дат**: События в определенном периоде
- **Местоположение**: События в определенном городе/штате

**Сценарии использования**:
- "Показать все доставленные отправки сегодня"
- "Найти все исключения за прошлую неделю"
- "Отслеживать отправки, которые в данный момент в пути"

## Сырые данные (отладка)

**Поле сырых данных**:
- Хранит полный ответ API перевозчика в формате JSON
- Полезно для отладки проблем с вебхуками
- Содержит метаданные, специфичные для перевозчика

**Пример сырых данных** (FedEx):
```json
{
  "event_type": "OD",
  "event_description": "На пути к получателю",
  "timestamp": "2026-02-13T08:15:00Z",
  "location": {
    "city": "Бруклин",
    "state": "NY",
    "postal_code": "11201",
    "country": "US"
  },
  "delivery_signature": null,
  "estimated_delivery": "2026-02-13T17:00:00Z"
}
```

**Когда проверять сырые данные**:
- Описание события неясно
- Отсутствуют данные о местоположении
- Ошибки обработки вебхуков
- Повышение уровня поддержки перевозчика

## Время события

**Произошло в** vs **Создано в**:

**Произошло в**: Когда событие перевозчика действительно произошло
- Пример: Пакет сканирован в 10:30 AM

**Создано в**: Когда Spwig получил вебхук
- Пример: Вебхук получен в 10:32 AM (задержка на 2 минуты)

**Почему они разные?**:
- Задержка сети
- Пакетная обработка перевозчика
- Задержки повторных попыток вебхуков

**Используйте "Произошло в" для отображения клиентам** — это более точное отражение реального прогресса доставки.

## Советы

- **События только для чтения** — нельзя редактировать после создания (целостность аудита)
- **Проверяйте сырые данные для получения подробной информации** — больше информации, чем отображаемые поля
- **Следите за задержкой вебхуков** — большое время между occurred_at и created_at указывает на проблемы с вебхуками
- **Используйте для поддержки клиентов** — временной график событий помогает диагностировать проблемы с доставкой
- **Отслеживайте паттерны доставки** — анализируйте время событий для оценки производительности перевозчика
- **Настройте уведомления** — автоматически отправляйте клиентам электронные письма при ключевых событиях (out_for_delivery, delivered)
- **Не удаляйте события** — сохраняйте полный аудит
- **Проверяйте WebhookLog на наличие сбоев** — отсутствующие события могут указывать на ошибки обработки вебхуков
- **Данные о местоположении различаются в зависимости от перевозчика** — некоторые перевозчики предоставляют подробные данные о местоположении, другие — минимальные
- **События-исключения требуют внимания** — отслеживайте и следите за исключениями в доставке
