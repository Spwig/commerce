---
title: Журналы веб-хука
---

Журналы веб-хука обеспечивают постоянный аудит-трек всех входящих запросов от перевозчиков — фиксируя метод запроса, URL-адрес конечной точки, заголовки, полезную нагрузку, статус обработки (ожидание/обработано/провалено) и ответ. Каждый веб-хук записывается перед обработкой, чтобы гарантировать, что никакие события не будут потеряны в случае сбоя обработки. Журналы позволяют отлаживать проблемы интеграции веб-хуков, отслеживать надежность API перевозчика и реконструировать временные линии доставки для поддержки клиентов.

Эта страница администратора только для чтения помогает диагностировать сбои веб-хуков и проверять здоровье интеграции с перевозчиками.

## Структура журнала веб-хука

Каждая запись журнала фиксирует:

**Детали запроса**:
- **Ключ провайдера**: Какой перевозчик отправил веб-хук (fedex, ups, dhl)
- **Конечная точка**: Путь URL веб-хука (например, `/webhooks/shipping/fedex/`)
- **Метод**: HTTP-метод (обычно POST)
- **Заголовки**: Заголовки запроса (JSON)
- **Полезная нагрузка**: Тело запроса (JSON)

**Обработка**:
- **Статус обработки**: ожидание, обработано, провалено
- **Сообщение об ошибке**: Причина провала (если статус=провалено)
- **Ответ**: HTTP-ответ, отправленный перевозчику
- **Код статуса ответа**: 200, 400, 500 и т.д.

**Временные метки**:
- **Получен в**: Когда прибыл веб-хук
- **Обработан в**: Когда завершилась обработка

---

## Значения статуса обработки

**ожидание**: Веб-хук получен, ожидает обработки
- Нормально на короткое время после получения
- Если застрял в состоянии ожидания, это указывает на задержку в очереди обработки

**обработано**: Веб-хук успешно обработан
- Создано событие отслеживания
- Отправлено уведомление клиенту (если применимо)
- Отправлен HTTP-ответ 200 перевозчику

**провалено**: Обработка веб-хука завершилась неудачно
- Проверьте поле error_message для причины
- Распространенные причины: Неверный JSON, неизвестная посылка, дублирующееся событие

---

## Поток веб-хука

**Нормальный поток**:
```
1. Перевозчик сканирует посылку
   ↓
2. Перевозчик отправляет POST-запрос в конечную точку веб-хука Spwig
   ↓
3. Spwig создает WebhookLog (status=pending)
   ↓
4. В фоновом режиме обрабатывается веб-хук
   ↓
5. Парсит полезную нагрузку JSON
   ↓
6. Находит соответствующую посылку (по номеру отслеживания)
   ↓
7. Создает событие отслеживания
   ↓
8. Обновляет WebhookLog (status=processed)
   ↓
9. Отправляет HTTP-ответ 200 перевозчику
```

**Сценарии сбоя**:
- **Неверный JSON**: Перевозчик отправил некорректные данные → status=провалено, error="Ошибка парсинга JSON"
- **Неизвестная посылка**: Номер отслеживания не совпадает с какой-либо посылкой → status=провалено, error="Посылка не найдена"
- **Дубликат**: Событие уже существует → status=провалено, error="Дублирующееся событие"

## Отладка сбоев веб-хуков

**Пошагово**:

**1. Отфильтровать по статусу=провалено**
- Перейдите в Доставка > Журналы веб-хуков
- Фильтр: Статус обработки = "провалено"
- Просмотрите недавние сбои

**2. Проверьте сообщение об ошибке**
- Нажмите на запись журнала
- Прочтите поле error_message
- Распространенные ошибки:
  - "Посылка не найдена" → Несоответствие номера отслеживания
  - "Ошибка декодирования JSON" → Перевозчик отправил неверный JSON
  - "Отсутствует обязательное поле" → Полезная нагрузка не содержит ожидаемых данных

**3. Проверьте полезную нагрузку**
- Просмотрите исходный JSON-данный
- Убедитесь, что структура соответствует ожидаемому формату
- Проверьте на наличие отсутствующих полей (tracking_id, event_type и т.д.)

**4. Проверьте, существует ли посылка**
- Извлеките номер отслеживания из полезной нагрузки
- Найдите посылку по номеру отслеживания
- Убедитесь, что посылка существует и использует правильного перевозчика

**5. Проверьте настройки провайдера**
- Убедитесь, что аккаунт провайдера активен
- Подтвердите, что URL-адрес конечной точки веб-хука правильный
- Проверьте учетные данные API провайдера

**6. Повторная обработка** (если применимо)
- Некоторые обработчики веб-хуков поддерживают ручную повторную попытку
- Сначала исправьте подлежащую проблему
- Повторно обработайте проваленный веб-хук

## Распространенные проблемы с веб-хуками

**Проблема 1: "Посылка не найдена"**

**Причина**: Номер отслеживания в веб-хуке не совпадает с какой-либо посылкой
- Опечатка при создании посылки
- Веб-хук для другого аккаунта
- Посылка была удалена до получения веб-хука

**Решение**:
- Проверьте правильность написания номера отслеживания
- Убедитесь, что перевозчик посылки совпадает с провайдером веб-хука
- Пересоздайте посылку при необходимости

---

**Проблема 2: "Ошибка декодирования JSON"**

**Причина**: Перевозчик отправил некорректный JSON
- Редко, обычно баг в API перевозчика
- Проблемы с кодировкой символов

**Решение**:
- Свяжитесь с поддержкой перевозчика, предоставив исходные данные
- Проверьте заголовки на наличие кодировки charset
- Убедитесь, что URL-адрес конечной точки указан правильно в панели перевозчика

---

**Проблема 3: Дублирующиеся веб-хуки**

**Причина**: Перевозчик отправляет одно и то же событие несколько раз
- Логика повторных попыток (перевозчик не получил ответ 200)
- Ошибка перевозчика

**Решение**:
- Система автоматически отклоняет дубликаты (нормальное поведение)
- Убедитесь, что response_status_code равен 200
- Если проблема сохраняется, свяжитесь с поддержкой перевозчика

---

**Проблема 4: Отсутствующие веб-хуки**

**Причина**: Ожидался веб-хук, но он никогда не был получен
- Перевозчик не отправил (пропущен скан)
- Конечная точка веб-хука настроена неправильно в панели перевозчика
- Огнестенок блокирует запросы

**Решение**:
- Проверьте настройки веб-хука в панели перевозчика
- Убедитесь, что URL-адрес конечной точки публичный и доступен
- Тестируйте конечную точку с помощью curl/Postman
- Проверьте правила брандмауэра сервера

## Настройка конечной точки веб-хука

**Типичные URL-адреса веб-хуков**:
```
FedEx: https://yourdomain.com/webhooks/shipping/fedex/
UPS: https://yourdomain.com/webhooks/shipping/ups/
DHL: https://yourdomain.com/webhooks/shipping/dhl/
```

**Настройка в панели перевозчика**:
1. Войдите в портал разработчика перевозчика
2. Перейдите в настройки веб-хуков
3. Введите URL-адрес веб-хука Spwig
4. Выберите события для подписки (обновления отслеживания, доставка, исключения)
5. Сохраните настройки
6. Проверьте веб-хук с помощью тестового инструмента перевозчика

**Безопасность**:
- Веб-хуки требуют HTTPS (не HTTP)
- Некоторые перевозчики подписывают запросы (проверьте подпись)
- Белый список IP-адресов (если перевозчик предоставляет статические IP-адреса)

## Мониторинг здоровья веб-хуков

**Ключевые метрики**:

**Доля успешных запросов**:
```
Доля успеха = (Обработано / Всего) × 100%

Цель: >98%
```

**Время обработки**:
```
Среднее время = Время обработки - Время получения

Цель: <2 секунд
```

**Паттерны сбоев**:
- Резкий скачок сбоев → Изменение или сбой API перевозчика
- Постоянные "посылка не найдена" → Проблема с синхронизацией номеров отслеживания
- Все веб-хуки провалены → Проблема с настройкой конечной точки

**Стратегия мониторинга**:
- Проверяйте дневную долю сбоев
- Оповещайте, если доля сбоев >5%
- Анализируйте сообщения об ошибках еженедельно
- Сравнивайте с панелью статуса перевозчика

---

## Хранение веб-хуков

**Журналы постоянны** - никогда не удаляются автоматически

**Почему постоянные**:
- Соответствие требованиям аудита
- Поддержка клиентов (восстановление временной линии доставки)
- Разрешение споров
- Отладка веб-хуков

**Хранение**: Журналы хранятся эффективно (сжатый JSON)

---

## Советы

- **Веб-хуки — это постоянный аудит-трек** - Никогда не удаляйте, даже если обработка прошла успешно
- **Ежедневно проверяйте проваленные веб-хуки** - Рано обнаружьте проблемы с интеграцией
- **Контролируйте задержку обработки** - Долгая задержка указывает на проблему производительности
- **Сохраняйте сырые данные** - Важно для отладки изменений в API перевозчика
- **Тестируйте настройки конечной точки** - Используйте инструменты тестирования перевозчика для проверки конфигурации
- **Включите подпись веб-хука** - Проверьте, действительно ли запросы исходят от перевозчика
- **Разрешите IP-адреса перевозчика** - Если перевозчик предоставляет статические диапазоны IP-адресов
- **Настройте оповещения** - Получайте уведомления, когда доля сбоев превышает порог
- **Сравнивайте с состоянием перевозчика** - Пробелы в веб-хуках могут указывать на сбой перевозчика
- **Документируйте форматы полезных данных перевозчика** - Помогает при обновлении API перевозчика
- **Сохраняйте URL-адреса веб-хуков стабильными** - Изменение URL-адресов требует обновления в панели перевозчика
