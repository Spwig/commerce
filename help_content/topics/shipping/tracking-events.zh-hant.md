---
title: 追蹤事件
---

追蹤事件記錄運送狀態的檢查點，貫穿整個配送週期—每個事件會捕捉狀態（運送中、即將派送、已送達）、時間戳、位置、描述和原始承運商數據。事件會透過承運商的 webhook 通知自動創建，或由商家手動創建。客戶可在其帳戶和訂單確認郵件中看到追蹤事件歷史，提供即時的配送可視性。

此管理頁面顯示僅供審計和客戶支援用途的只讀事件歷史。

## 追蹤事件結構

每個事件包含：

**狀態資訊**：
- **狀態**：in_transit, out_for_delivery, delivered, exception, failed, returned
- **描述**：可讀的狀態（例如："包裹已到達分類中心"）
- **承運商狀態碼**：原始承運商狀態（例如："DEP" 表示已出發）

**位置數據**：
- **城市**：事件發生的城市
- **州/省**：事件發生的州/省
- **國家**：事件發生的國家
- **郵遞區號**：事件發生的郵遞區號

**時間戳**：
- **發生時間**：事件實際發生的時間（承運商時間）
- **創建時間**：事件在 Spwig 中被記錄的時間（系統時間）

**元資料**：
- **原始數據**：來自承運商 API 的完整 JSON 回應
- **運送單**：關聯的運送單 ID

---

## 事件狀態類型

**in_transit**：包裹正在通過承運商網絡運送
- 範例："已離開設施", "已到達樞紐", "正在運送到下一個設施"

**out_for_delivery**：包裹正在派送車上
- 範例："即將派送", "已在派送車上"

**delivered**：包裹已成功送達
- 範例："已送達門口", "放在接待處", "交給收件人"

**exception**：需要處理的配送問題
- 範例："天氣延誤", "地址錯誤", "派送嘗試失敗"

**failed**：配送永久失敗
- 範例："按地址無法投遞", "收件人拒絕接收"

**returned**：包裹正在退回發件人
- 範例："已啟動退回", "包裹正在退回"

## 如何創建追蹤事件

### 自動（承運商 Webhook）

**流程**：
1. 承運商掃描包裹（出發、到達、派送）
2. 承運商將 Webhook 發送到 Spwig 的 Webhook 端點
3. Webhook 被記錄在 WebhookLog 表中
4. 系統解析 Webhook 載荷
5. 使用提取的數據創建 TrackingEvent
6. 如果已配置，發送客戶電子郵件通知

**優點**：
- 實時更新（無需輪詢）
- 來自承運商的準確時間戳
- 自動維護完整的事件歷史

### 手動（商家輸入）

**流程**：
1. 導航至運送明細
2. 點擊 "新增追蹤事件"
3. 從下拉選單中選擇狀態
4. 輸入描述
5. 可選：輸入位置數據
6. 設定 occurred_at 時間戳
7. 儲存

**使用場景**：
- 無 Webhook 支援的承運商
- 手動修正運送資料
- 當地配送（非承運商）
- 內部狀態更新

## 事件顯示順序

事件按 **反向時間順序**（最新優先）顯示：

**範例顯示**：
```
2026年2月13日 上午10:30 - 已送達（布魯克林, NY）
2026年2月13日 上午8:15 - 即將派送（布魯克林, NY）
2026年2月12日 晚上11:45 - 已到達當地設施（布魯克林, NY）
2026年2月12日 下午6:30 - 運送中（紐瓦克, NJ）
2026年2月12日 下午2:15 - 已離開原點（費城, PA）
2026年2月12日 上午9:00 - 已取件（費城, PA）
```

## 客戶可見性

客戶可在以下地方看到追蹤事件：

**訂單確認郵件**：
- 最新事件狀態
- 預估送達日期
- 追蹤連結

**客戶帳戶 > 訂單明細**：
- 完整事件時間軸
- 事件描述
- 位置歷史
- 時間戳

**追蹤頁面**（如已啟用）：
- 專屬追蹤 URL
- 視覺化時間軸
- 承運商標誌
- 配送地圖（如位置數據可用）

## 過濾追蹤事件

**有用的過濾器**：
- **運送單**：查看特定運送單的事件
- **狀態**：按事件類型過濾（已送達、in_transit 等）
- **日期範圍**：指定時間範圍內的事件
- **位置**：特定城市/州的事件

**使用場景**：
- "顯示今天所有已送達的運送單"
- "找出上週所有例外事件"
- "追蹤目前處於 in_transit 的運送單"

## 原始數據（除錯）

**原始數據欄位**：
- 儲存完整的承運商 API 回應為 JSON 格式
- 用於除錯 Webhook 問題
- 包含承運商特定的元資料

**原始數據範例**（FedEx）：
```json
{
  "event_type": "OD",
  "event_description": "Out for delivery",
  "timestamp": "2026-02-13T08:15:00Z",
  "location": {
    "city": "Brooklyn",
    "state": "NY",
    "postal_code": "11201",
    "country": "US"
  },
  "delivery_signature": null,
  "estimated_delivery": "2026-02-13T17:00:00Z"
}
```

**什麼時候檢查原始數據**：
- 事件描述不清晰
- 缺少位置數據
- Webhook 處理錯誤
- 需要聯絡承運商支援

---

## 事件時間

**Occurred At** 與 **Created At**：

**Occurred At**：承運商事件實際發生的時間
- 範例：包裹在上午10:30被掃描

**Created At**：Spwig 收到 Webhook 的時間
- 範例：Webhook 在上午10:32收到（2分鐘延遲）

**為何不同**：
- 網路延遲
- 承運商批次處理
- Webhook 重試延遲

**用 Occurred At 顯示給客戶** - 更準確反映實際配送進度。

## 小技巧

- **事件為只讀** - 創建後無法編輯（審計完整性）
- **檢查原始數據以獲取細節** - 比顯示欄位包含更多資訊
- **監控 Webhook 延遲** - occurred_at 和 created_at 之間的大幅延遲表示 Webhook 問題
- **用於客戶支援** - 事件時間軸有助於診斷配送問題
- **追蹤配送模式** - 分析事件時間以評估承運商表現
- **設定通知** - 在關鍵事件（out_for_delivery, delivered）自動郵件通知客戶
- **不要刪除事件** - 保留完整的審計軌跡
- **檢查 WebhookLog 以查找失敗** - 缺少事件可能表示 Webhook 處理錯誤
- **位置數據因承運商而異** - 有些承運商提供詳細位置，有些僅提供基本資料
- **例外事件需要關注** - 監控並跟進配送例外情況
