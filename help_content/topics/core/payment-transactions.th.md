---
title: การทำธุรกรรมการชำระเงิน
---

การทำธุรกรรมการชำระเงินคือบันทึกทั้งหมดของเหตุการณ์การชำระเงินทุกครั้งที่ผ่านการประมวลผลในร้านค้าของคุณ — การเรียกเก็บเงิน การคืนเงิน การอนุมัติ และอื่นๆ หัวข้อนี้ยังรวมบันทึก webhook จากผู้ให้บริการชำระเงินของคุณและเจตนาการชำระเงินที่สร้างขึ้นระหว่างขั้นตอนการชำระเงินด้วย

## การทำธุรกรรมการชำระเงิน

ไปที่ **การชำระเงิน > การทำธุรกรรมการชำระเงิน** เพื่อดูการทำธุรกรรมทั้งหมดที่ร้านค้าของคุณได้ประมวลผล

### ประเภทการทำธุรกรรม

| ประเภท | หมายถึงอะไร |
|------|--------------|
| **การเรียกเก็บเงิน** | การชำระเงินทันที — เงินจะถูกเก็บทันทีที่การทำธุรกรรม |
| **การอนุมัติ** | เงินถูกเก็บไว้ในบัตรของลูกค้าแต่ยังไม่ได้ถูกเก็บ |
| **การจับเงิน** | เก็บเงินจากข้ออนุมัติที่ผ่านมา |
| **การยกเลิก** | ยกเลิกการอนุมัติก่อนที่จะถูกจับเงิน |
| **การคืนเงิน** | คืนเงินให้กับลูกค้า |

### สถานะการทำธุรกรรม

| สถานะ | หมายถึงอะไร |
|--------|--------------|
| **อยู่ระหว่างดำเนินการ** | การทำธุรกรรมได้เริ่มต้นแล้วแต่ยังไม่ได้รับการประมวลผล |
| **กำลังดำเนินการ** | กำลังถูกประมวลผลโดยผู้ให้บริการชำระเงิน |
| **ได้รับการอนุมัติ** | เงินถูกเก็บไว้ — รอการจับเงิน |
| **สำเร็จ** | การชำระเงินประสบความสำเร็จ |
| **ล้มเหลว** | การชำระเงินถูกปฏิเสธหรือเกิดข้อผิดพลาด |
| **ถูกยกเลิก** | การอนุมัติถูกยกเลิกก่อนที่จะถูกจับเงิน |
| **คืนเงินแล้ว** | ได้ทำการคืนเงินเต็มจำนวนแล้ว |
| **คืนเงินบางส่วน** | ส่วนหนึ่งของเงินได้ถูกคืนแล้ว |

### สิ่งที่คุณสามารถเห็นในบันทึกการทำธุรกรรม

# ข้อมูลการทำธุรกรรม

แต่ละธุรกรรมแสดง:
- **ID ธุรกรรม** — อ้างอิงภายในของ Spwig
- **ID ธุรกรรมของผู้ให้บริการ** — อ้างอิงจากผู้ให้บริการชำระเงินของคุณ (เช่น ID การเรียกเก็บเงินจาก Stripe)
- **จำนวนเงิน** — จำนวนเงินและสกุลเงินของธุรกรรม
- **สถานะ** และ **ประเภท**
- **อีเมลลูกค้า** และ **ชื่อลูกค้า**
- **วิธีการชำระเงิน** — ประเภท (บัตรเครดิต, การโอนเงินผ่านธนาคาร, ฯลฯ) และเลขสุดท้าย 4 หลัก
- **คำสั่งซื้อ** — คำสั่งซื้อที่ธุรกรรมนี้อยู่ภายใต้
- **บัญชีผู้ให้บริการ** — ผู้ให้บริการชำระเงินที่ดำเนินการ
- **การตอบกลับจากผู้ให้บริการ** — การตอบกลับทางเทคนิคจากผู้ให้บริการชำระเงิน
- **ข้อความข้อผิดพลาด** — หากธุรกรรมล้มเหลว สาเหตุที่ผู้ให้บริการให้มา
- เวลาที่สร้าง แก้ไขล่าสุด และเสร็จสิ้น

### การกรองธุรกรรม

ใช้ตัวกรองในแดชบอร์ดเพื่อจำกัดธุรกรรมตาม:
- สถานะ (เช่น แสดงเฉพาะธุรกรรมที่ล้มเหลว)
- ประเภท (เช่น แสดงเฉพาะการคืนเงิน)
- บัญชีผู้ให้บริการ
- ช่วงวันที่

มีประโยชน์สำหรับการตรวจสอบยอดขายประจำวันหรือการตรวจสอบประวัติการชำระเงินของลูกค้ารายใดรายหนึ่ง

### ธุรกรรมสามารถคืนเงินได้เมื่อใด?

ธุรกรรมสามารถคืนเงินได้เมื่อ:
- สถานะของธุรกรรมคือ **เสร็จสิ้น**
- ประเภทของธุรกรรมคือ **Charge** หรือ **Capture**

เพื่อออกใบคืนเงิน ให้ใช้ **การคืนเงิน** จากหน้ารายละเอียดคำสั่งซื้อ คืนเงินที่ดำเนินการผ่านคำสั่งซื้อจะสร้างบันทึกธุรกรรมใหม่ประเภท **Refund**

### ขั้นตอนการอนุมัติและการจับเงิน

บางวิธีการชำระเงิน (และบางผู้ให้บริการชำระเงิน) สนับสนุนการอนุมัติและการจับเงินแยกกัน ซึ่งมีประโยชน์หากคุณต้องการตรวจสอบการชำระเงินก่อนส่งสินค้า:

1. **การอนุมัติ** — เงินจะถูกเก็บไว้ในบัตรของลูกค้า (สถานะ: `Authorized`)
2. **การจับเงิน** — ถูกเปิดใช้งานเมื่อคำสั่งซื้อถูกส่งหรือถูกดำเนินการ
3. หากไม่ได้จับเงินภายในช่วงเวลาการอนุมัติ การเก็บเงินจะ **หมดอายุ** โดยอัตโนมัติ

ฟิลด์ **หมดอายุที่** บนธุรกรรมแสดงว่าการอนุมัติจะหมดอายุเมื่อใด

## Webhook การชำระเงิน

ผู้ให้บริการชำระเงินจะส่งเหตุการณ์ webhook เพื่อแจ้งให้ร้านค้าของคุณทราบเกี่ยวกับการเปลี่ยนแปลงสถานะการชำระเงิน — ตัวอย่างเช่น เมื่อการชำระเงินสำเร็จ ล้มเหลว หรือมีข้อพิพาท


Spwig บันทึกข้อมูลทั้งหมดที่ได้รับจาก webhook

ไปที่ **Payments > Payment Webhooks** เพื่อดูบันทึก

### ข้อมูลที่บันทึกใน webhook

| Field | Description |
|-------|-------------|
| **Provider** | ผู้ให้บริการชำระเงินที่ส่ง webhook |
| **Event ID** | ตัวระบุเหตุการณ์ที่เป็นเอกลักษณ์ของผู้ให้บริการ |
| **Event Type** | ประเภทของเหตุการณ์ (เช่น `payment_intent.succeeded`, `charge.refunded`) |
| **Processed** | Spwig ได้ดำเนินการกับ webhook นี้แล้วหรือไม่ |
| **Signature Verified** | ลายเซ็นความปลอดภัยของ webhook ถูกตรวจสอบแล้วหรือไม่ |
| **Payload** | ข้อมูลทั้งหมดที่ผู้ให้บริการส่งมา |
| **Processing Result** | สิ่งที่ Spwig ได้ดำเนินการตอบสนอง |
| **Processing Error** | ข้อผิดพลาดใด ๆ ที่เกิดขึ้นระหว่างการประมวลผล |
| **Received At** | เวลาที่ webhook ถูกส่งมา |

### การใช้บันทึก webhook สำหรับการแก้ไขปัญหา

หากการชำระเงินดูเหมือนติดอยู่ หรือสถานะคำสั่งซื้อไม่ได้อัปเดตหลังจากการชำระเงิน:

1. ไปที่ **Payments > Payment Webhooks**
2. กรองโดยใช้ผู้ให้บริการและค้นหาเหตุการณ์ล่าสุด
3. ตรวจสอบคอลัมน์ **Processed** — webhook ที่ยังไม่ได้รับการประมวลผลอาจบ่งบอกถึงปัญหาในการส่ง
4. ตรวจสอบ **Signature Verified** — ลายเซ็นที่ล้มเหลวอาจหมายความว่าคุณตั้งค่า webhook secret ผิดพลาด
5. ตรวจสอบ **Processing Error** เพื่อดูข้อความข้อผิดพลาดใด ๆ

การส่งเหตุการณ์ซ้ำ ๆ จะถูกจัดการโดยอัตโนมัติ — การรวม `Event ID` และผู้ให้บริการนั้นเป็นเอกลักษณ์ ดังนั้น webhook เดียวกันจึงไม่สามารถประมวลผลซ้ำได้

## Payment intents

Payment intent ติดตามวงจรชีวิตของการชำระเงินในขั้นตอนการเช็คอินตั้งแต่ลูกค้าเริ่มกระบวนการชำระเงินจนถึงผลลัพธ์สุดท้าย Payment intents จะถูกสร้างโดยอัตโนมัติเมื่อลูกค้าไปถึงขั้นตอนการชำระเงินในขั้นตอนการเช็คอิน

ไปที่ **Payments > Payment Intents** เพื่อดูรายการ

### สถานะของ Payment intent

| สถานะ | ความหมาย |
|--------|---------|
| **สร้างขึ้นแล้ว** | ได้รับการสร้างแล้ว รอการระบุวิธีการชำระเงิน |
| **ต้องการวิธีการชำระเงิน** | รอให้ลูกค้ากรอกข้อมูลบัตร |
| **ต้องการยืนยัน** | ได้กรอกข้อมูลการชำระเงินแล้ว รอการยืนยัน |
| **ต้องดำเนินการ** | ลูกค้าต้องดำเนินการบางอย่าง (เช่น การยืนยันความปลอดภัย 3D Secure) |
| **กำลังประมวลผล** | การชำระเงินกำลังถูกประมวลผล |
| **สำเร็จ** | การชำระเงินสำเร็จ |
| **ยกเลิก** | การชำระเงินถูกยกเลิกหรือถูกทิ้งไว้ |
| **ล้มเหลว** | การพยายามชำระเงินล้มเหลว |

### การไหลของ Payment intent ไปยัง Order

1. ลูกค้าไปถึงขั้นตอนการชำระเงินที่เคาน์เตอร์ → Spwig สร้าง **Payment Intent** และ **Order** (ยังไม่ชำระเงิน) ร่าง
2. ลูกค้ากรอกข้อมูลการชำระเงินและยืนยัน
3. ผู้ให้บริการชำระเงินประมวลผลการชำระเงิน
4. เมื่อสำเร็จ ระบบจะอัปเดต Order เป็น **Paid** และ Payment Intent จะเปลี่ยนเป็น **Succeeded**
5. บันทึก **Payment Transaction** จะถูกสร้างขึ้นพร้อมรายละเอียดการคิดค่าใช้จ่ายสุดท้าย

Payment intent เชื่อมโยงระหว่างเซสชันการเช็กเอาท์ บัญชีผู้ให้บริการ และ Order — ให้คุณมีภาพรวมที่สมบูรณ์ของเส้นทางการเช็กเอาท์ของลูกค้า

### การใช้ Payment intents สำหรับการสนับสนุน

หากลูกค้ารายงานว่าพวกเขาได้ชำระเงินแล้วแต่ Order ยังแสดงว่าไม่ได้ชำระเงิน:

1. ค้นหา Order ของลูกค้าใน **Orders**
2. ไปที่ **Payments > Payment Intents** และค้นหา intents ที่เชื่อมโยงกับ Order นั้น
3. ตรวจสอบสถานะของ intent — หากเป็น **Succeeded** ให้ตรวจสอบ transaction ที่เชื่อมโยง
4. หาก intent เป็น **Requires Action** ลูกค้าอาจยังไม่ได้ทำขั้นตอนการยืนยันความปลอดภัย 3D Secure
5. หาก intent เป็น **Failed** รายละเอียดข้อผิดพลาดจะอธิบายว่าทำไมการชำระเงินถึงถูกปฏิเสธ

## เคล็ดลับ

- ตรวจสอบรายการการชำระเงินที่ล้มเหลวทุกวัน — รูปแบบของความล้มเหลว (เช่น วิธีการชำระเงินเฉพาะหรือประเทศ) อาจบ่งบอกถึงปัญหาการตั้งค่าหรือการพยายามฉ้อโกง
- บันทึก Webhook มีคุณค่ามากเมื่อตรวจสอบความไม่สอดคล้องของการชำระเงิน

# หากคำสั่งซื้อถูกชำระเงินแต่ยังไม่ได้ยืนยัน การบันทึก webhook จะช่วยบอกคุณว่าเกิดอะไรขึ้น
- การยึดเงินแบบอุปถัมภ์จะหมดอายุโดยอัตโนมัติ — หากคุณใช้รูปแบบ authorize-then-capture โปรดตรวจสอบให้แน่ใจว่ากระบวนการจัดส่งของคุณสามารถเก็บเงินได้ก่อนที่ช่วงเวลาการหมดอายุจะปิด (โดยทั่วไปคือ 7 วันสำหรับผู้ให้บริการส่วนใหญ่)
- ฟิลด์ **Provider Response** บนธุรกรรมมีข้อมูลดิบจากผู้ให้บริการชำระเงิน

แบ่งปันข้อมูลนี้กับทีมสนับสนุนของผู้ให้บริการหากคุณต้องการความช่วยเหลือในการแก้ไขปัญาธุรกรรมเฉพาะ
- การตรวจสอบลายเซ็นล้มเหลวใน webhook ควรได้รับการตรวจสอบทันที — อาจบ่งบอกถึงการตั้งค่า webhook secret ผิดพลาด หรือการพยายามส่งเหตุการณ์ webhook ปลอมไปยังร้านค้าของคุณ
