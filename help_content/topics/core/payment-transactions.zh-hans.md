---
title: 支付交易
---

支付交易是您商店处理的每一次支付事件的完整记录 —— 收款、退款、授权等。此部分还包括来自您的支付提供商的 Webhook 日志以及结账过程中创建的支付意图。

## 支付交易

导航到 **支付 > 支付交易** 以查看您商店处理过的每笔交易。

### 交易类型

| 类型 | 含义 |
|------|--------------|
| **收款** | 立即支付 —— 资金在交易时被收取 |
| **授权** | 资金被保留在客户的银行卡上但尚未收取 |
| **捕获** | 从之前的授权中收取资金 |
| **作废** | 在捕获之前取消授权 |
| **退款** | 将款项退还给客户 |

### 交易状态

| 状态 | 含义 |
|--------|--------------|
| **待处理** | 交易已启动但尚未处理 |
| **处理中** | 正在由支付提供商处理 |
| **已授权** | 资金已被保留 —— 等待捕获 |
| **已完成** | 支付成功 |
| **失败** | 支付被拒绝或发生错误 |
| **已作废** | 在捕获之前取消了授权 |
| **已退款** | 已发出全额退款 |
| **部分退款** | 部分款项已退还 |

### 交易记录中您可以查看的内容

# 交易详情

每笔交易显示以下信息：
- **交易 ID** — Spwig 的内部参考
- **支付提供商交易 ID** — 来自您的支付提供商的参考（例如，Stripe 收单 ID）
- **金额** — 交易金额和货币
- **状态** 和 **类型**
- **客户电子邮件** 和 **客户姓名**
- **支付方式** — 类型（信用卡、银行转账等）和最后四位数字
- **订单** — 该交易所属的订单
- **支付提供商账户** — 处理该交易的支付提供商
- **支付提供商响应** — 支付提供商提供的原始技术响应
- **错误信息** — 如果交易失败，提供商提供的失败原因
- 创建、最后更新和完成的时间戳

### 过滤交易

使用管理员过滤器按以下条件缩小交易范围：
- 状态（例如，仅显示失败的交易）
- 类型（例如，仅显示退款）
- 支付提供商账户
- 日期范围

这对于日常对账或调查特定客户的付款历史非常有用。

### 何时可以退款交易？

当以下条件满足时，可以对交易进行退款：
- 其状态为 **已完成**
- 其类型为 **Charge** 或 **Capture**

要发起退款，请使用订单详情页面上的 **退款** 操作。通过订单处理的退款会创建一个类型为 **Refund** 的新交易记录。

### 授权和捕获流程

某些支付方式（以及某些支付提供商）支持单独的授权和捕获。如果您希望在发货前验证付款，这会很有用：

1. **授权** — 客户的银行卡上保留资金（状态：`Authorized`）
2. **捕获** — 在订单发货或完成时触发
3. 如果在授权窗口内未进行捕获，保留的额度将 **自动过期**

交易上的 **Expires At** 字段显示授权将过期的时间。

## 支付 Webhook

支付提供商通过 Webhook 事件通知您的商店有关付款状态的变化 — 例如，当付款成功、失败或发生争议时。


Spwig 记录所有传入的网络钩子。

导航到 **支付 > 支付网络钩子** 以查看日志。

### 网络钩子记录显示的内容

| 字段 | 描述 |
|-------|-------------|
| **提供商** | 哪个支付提供商发送了网络钩子 |
| **事件 ID** | 提供商的唯一事件标识符 |
| **事件类型** | 事件类型（例如，`payment_intent.succeeded`，`charge.refunded`） |
| **已处理** | Spwig 是否已对此网络钩子采取了行动 |
| **签名验证** | 网络钩子的安全签名是否有效 |
| **有效负载** | 提供商发送的完整数据 |
| **处理结果** | Spwig 的响应操作 |
| **处理错误** | 处理过程中出现的任何错误 |
| **接收时间** | 网络钩子到达的时间 |

### 使用网络钩子日志进行故障排除

如果支付似乎卡住或支付后订单状态未更新：

1. 导航到 **支付 > 支付网络钩子**
2. 按提供商筛选并查找最近的事件
3. 检查 **已处理** 列 — 未处理的网络钩子可能表明交付存在问题
4. 检查 **签名验证** — 失败的签名可能意味着您的网络钩子密钥配置不正确
5. 查看 **处理错误** 以获取任何错误信息

重复事件会自动处理 — `事件 ID` 和提供商的组合是唯一的，因此相同的网络钩子不能被处理两次。

## 支付意图

支付意图跟踪从客户开始支付流程到最终结果的结账支付生命周期。当客户到达结账支付步骤时，支付意图会自动创建。

导航到 **支付 > 支付意图** 以查看列表。

### 支付意图状态

| 状态 | 含义 |
|--------|---------|
| **已创建** | 支付意图已创建，等待支付方式 |
| **需要支付方式** | 等待客户输入他们的信用卡信息 |
| **需要确认** | 已输入支付详情，等待确认 |
| **需要操作** | 客户需要完成一个操作（例如，3D Secure 验证） |
| **处理中** | 支付正在处理 |
| **成功** | 支付已成功完成 |
| **已取消** | 支付被放弃或取消 |
| **失败** | 支付尝试失败 |

### 支付意图到订单的流程

1. 客户到达结账支付步骤 → Spwig 创建一个 **支付意图** 和一个草稿 **订单**（未支付）
2. 客户输入支付详情并确认
3. 支付提供商处理支付
4. 成功时，订单更新为 **已支付**，支付意图移动到 **成功**
5. 创建一个带有最终收费详情的 **支付交易** 记录

支付意图将结账会话、提供商账户和订单联系在一起 —— 给您一个客户结账旅程的完整画面。

### 使用支付意图进行支持

如果客户报告他们已支付但订单显示为未支付：

1. 在 **订单** 中找到客户的订单
2. 转到 **支付 > 支付意图** 并搜索与该订单关联的意图
3. 检查意图状态 —— 如果是 **成功**，检查关联的交易
4. 如果意图是 **需要操作**，客户可能未完成 3D Secure 验证
5. 如果意图是 **失败**，错误详情会解释支付为何被拒绝

## 小贴士

- 每天检查失败的交易 —— 失败模式（例如，特定的支付方式或国家）可能表明配置问题或欺诈尝试。
- 在调查支付差异时，Webhook 日志非常有价值。

# 如果订单已支付但未确认，webhook 日志通常会告诉你哪里出了问题。
- 授权保留会自动过期 —— 如果你使用的是先授权后捕获的方式，请确保在授权保留期限结束前（通常大多数提供商为7天）完成资金捕获。
- 交易中的 **Provider Response** 字段包含来自支付提供商的原始数据。

如果你需要帮助解决特定交易问题，请将此信息与提供商的支持团队分享。
- webhook 的签名验证失败应立即调查 —— 这可能表明 webhook 密钥配置错误，或有人试图向你的商店发送欺诈性的 webhook 事件。
