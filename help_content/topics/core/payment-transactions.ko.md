---
title: 결제 거래
---

결제 거래는 상점에서 처리된 모든 결제 이벤트의 전체 기록입니다. 이 섹션에는 결제 제공업체에서 전송된 웹후크 로그와 체크아웃 시 생성된 결제 의도도 포함됩니다.

## 결제 거래

**결제 > 결제 거래**로 이동하여 상점이 처리한 모든 거래를 확인할 수 있습니다.

### 거래 유형

| 유형 | 의미 |
|------|--------------|
| **Charge** | 즉시 결제 — 거래 시점에 자금이 수집됩니다 |
| **Authorize** | 고객의 카드에 자금이 잠금되어 있지만 아직 수집되지 않았습니다 |
| **Capture** | 이전 승인에서 자금을 수집합니다 |
| **Void** | 수집 전 승인을 취소합니다 |
| **Refund** | 고객에게 결제를 환불합니다 |

### 거래 상태

| 상태 | 의미 |
|--------|--------------|
| **Pending** | 거래가 시작되었지만 아직 처리되지 않았습니다 |
| **Processing** | 결제 제공업체에서 처리 중입니다 |
| **Authorized** | 자금이 잠금되어 있으며, 수집을 기다리고 있습니다 |
| **Completed** | 결제가 성공했습니다 |
| **Failed** | 결제가 거절되었거나 오류가 발생했습니다 |
| **Voided** | 수집 전 승인이 취소되었습니다 |
| **Refunded** | 전체 환불이 처리되었습니다 |
| **Partially Refunded** | 결제의 일부가 환불되었습니다 |

### 거래 기록에서 확인할 수 있는 내용

# 거래 정보

각 거래는 다음을 표시합니다:
- **거래 ID** — Spwig의 내부 참조
- **결제 제공업체 거래 ID** — 결제 제공업체에서 제공하는 참조 (예: Stripe 결제 ID)
- **금액** — 거래 금액 및 통화
- **상태** 및 **유형**
- **고객 이메일** 및 **고객 이름**
- **결제 수단** — 유형 (신용카드, 계좌 이체 등) 및 마지막 4자리
- **주문** — 이 거래에 속한 주문
- **결제 제공업체 계정** — 이 거래를 처리한 결제 제공업체
- **결제 제공업체 응답** — 결제 제공업체에서 제공한 원본 기술 응답
- **에러 메시지** — 거래가 실패한 경우 제공업체에서 제공한 원인
- 생성, 마지막 업데이트 및 완료 시간의 타임스탬프

### 거래 필터링

관리자 필터를 사용하여 다음으로 거래를 좁혀보세요:
- 상태 (예: 오직 실패한 거래만 표시)
- 유형 (예: 오직 환불만 표시)
- 결제 제공업체 계정
- 날짜 범위

이 기능은 하루 종료 시 대조 또는 특정 고객의 결제 내역을 조사하는 데 유용합니다.

### 언제 거래를 환불할 수 있나요?

거래를 환불할 수 있는 경우:
- 상태가 **완료됨**인 경우
- 유형이 **Charge** 또는 **Capture**인 경우

환불을 발행하려면 주문 상세 페이지에서 **환불** 액션을 사용하세요. 주문을 통해 처리된 환불은 유형이 **Refund**인 새로운 거래 기록을 생성합니다.

### 승인 및 캡처 흐름

일부 결제 수단 (및 일부 결제 제공업체)은 별도의 승인 및 캡처를 지원합니다. 이는 배송 전 결제를 확인하려는 경우에 유용합니다:

1. **승인** — 고객의 카드에 자금이 잠금됩니다 (상태: `Authorized`)
2. **캡처** — 주문이 배송되거나 완료될 때 트리거됩니다
3. 승인 윈도우 내에서 캡처되지 않은 경우 잠금이 **자동으로 만료**됩니다

거래의 **만료 시간** 필드는 승인의 만료 시점을 보여줍니다.

## 결제 웹훅

결제 제공업체는 결제 상태 변경을 통보하기 위해 웹훅 이벤트를 보내는 것으로, 예를 들어 결제가 성공하거나 실패하거나 분쟁이 발생할 때입니다.

Spwig은 모든 들어오는 웹후크를 기록합니다.

**결제 > 결제 웹후크**로 이동하여 로그를 확인하세요.

### 웹후크 기록이 표시하는 내용

| 필드 | 설명 |
|-------|-------------|
| **공급업체** | 웹후크를 보낸 결제 공급업체 |
| **이벤트 ID** | 공급업체의 고유 이벤트 식별자 |
| **이벤트 유형** | 이벤트의 유형 (예: `payment_intent.succeeded`, `charge.refunded`) |
| **처리됨** | Spwig이 이 웹후크에 대해 작업했는지 여부 |
| **서명 확인됨** | 웹후크의 보안 서명이 유효했는지 여부 |
| **페이로드** | 공급업체가 보낸 전체 데이터 |
| **처리 결과** | Spwig이 이에 대해 수행한 작업 |
| **처리 오류** | 처리 중 발생한 오류 |
| **수신 시간** | 웹후크가 도착한 시간 |

### 웹후크 로그를 사용한 문제 해결

결제가 멈춰 있는 경우 또는 결제 후 주문 상태가 업데이트되지 않은 경우:

1. **결제 > 결제 웹후크**로 이동하세요
2. 공급업체로 필터링하고 최근 이벤트를 확인하세요
3. **처리됨** 열을 확인하세요 — 처리되지 않은 웹후크는 전달 문제를 나타낼 수 있습니다
4. **서명 확인됨**을 확인하세요 — 실패한 서명은 웹후크 비밀이 잘못 설정되었음을 의미할 수 있습니다
5. **처리 오류**를 확인하여 오류 메시지를 확인하세요

중복된 이벤트는 자동으로 처리됩니다 — `이벤트 ID`와 공급업체 조합은 고유하므로 동일한 웹후크는 두 번 처리될 수 없습니다.

## 결제 인텐트

결제 인텐트는 고객이 결제 프로세스를 시작하는 순간부터 최종 결과까지 체크아웃 결제의 수명 주기를 추적합니다. 고객이 체크아웃에서 결제 단계에 도달하면 결제 인텐트가 자동으로 생성됩니다.

**결제 > 결제 인텐트**로 이동하여 목록을 확인하세요.

### 결제 인텐트 상태

| 상태 | 의미 |
|--------|---------|
| **생성됨** | 결제 의도가 생성되었으며, 결제 수단 대기 중 |
| **결제 수단 필요** | 고객이 카드 정보를 입력할 때까지 대기 중 |
| **확인 필요** | 결제 정보가 입력되었으며, 확인 대기 중 |
| **작업 필요** | 고객이 작업(예: 3D Secure 인증)을 완료해야 함 |
| **처리 중** | 결제가 처리 중 |
| **성공** | 결제가 성공적으로 완료됨 |
| **취소됨** | 결제가 포기되거나 취소됨 |
| **실패** | 결제 시도 실패 |

### 결제 의도에서 주문 흐름

1. 고객이 결제 단계에 도달하면 → Spwig은 **결제 의도**와 미결 **주문**(미결제)을 생성합니다
2. 고객이 결제 정보를 입력하고 확인합니다
3. 결제 제공업체가 결제를 처리합니다
4. 성공 시, 주문은 **결제 완료**로 업데이트되고 결제 의도는 **성공** 상태로 이동합니다
5. 최종 결제 세부 정보를 포함한 **결제 거래** 기록이 생성됩니다

결제 의도는 체크아웃 세션, 제공업체 계정 및 주문을 연결합니다 — 고객의 체크아웃 여정에 대한 전체적인 그림을 제공합니다.

### 지원을 위한 결제 의도 사용

고객이 결제를 완료했지만 주문이 미결제 상태로 표시되는 경우:

1. **주문**에서 고객의 주문을 찾습니다
2. **결제 > 결제 의도**로 이동하여 해당 주문과 연결된 의도를 검색합니다
3. 의도 상태를 확인합니다 — **성공**인 경우 연결된 거래를 확인합니다
4. 의도가 **작업 필요**인 경우, 고객이 3D Secure 인증을 완료하지 않았을 수 있습니다
5. 의도가 **실패**인 경우, 결제가 거부된 이유를 설명하는 오류 세부 정보를 확인합니다

## 팁

- 매일 실패한 거래를 검토하세요 — 실패 패턴(예: 특정 결제 수단 또는 국가)은 구성 문제 또는 사기 시도를 나타낼 수 있습니다.
- 결제 불일치를 조사할 때 웹후크 로그는 매우 유용합니다.

# 주문이 지불되었지만 확인되지 않은 경우
웹훅 로그는 일반적으로 문제가 무엇인지 알려줍니다.
- 인증 잔금은 자동으로 만료됩니다 — authorize-then-capture를 사용하는 경우, 제공업체의 만료 기간이 끝나기 전에 자금을 확보하는 확보 프로세스를 수행해야 합니다(대부분의 제공업체는 일반적으로 7일입니다).
- 거래의 **Provider Response** 필드에는 결제 제공업체에서 전달된 원본 데이터가 포함되어 있습니다.

구체적인 거래 문제를 해결하는 데 도움이 필요하면 해당 내용을 제공업체의 지원 팀과 공유하세요.
- 웹훅에서 서명 검증 실패는 즉시 조사해야 합니다 — 이는 웹훅 시크릿이 잘못 설정되었거나, 웹훅 이벤트를 가짜로 보내려는 시도를 나타낼 수 있습니다.
