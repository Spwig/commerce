---
title: 付款交易
---

付款交易是您商店處理過的每一次付款事件的完整記錄 — 收款、退款、授權等。此部分還包括來自付款提供商的 Webhook 日誌以及結帳過程中創建的付款意圖。

## 付款交易

導航至 **付款 > 付款交易** 以查看您商店處理過的每一筆交易。

### 交易類型

| 類型 | 含義 |
|------|--------------|
| **Charge** | 即時付款 — 資金在交易時即被收取 |
| **Authorize** | 資金已暫存於客戶的信用卡中，但尚未收取 |
| **Capture** | 從先前的授權中收取資金 |
| **Void** | 在資金被收取前取消授權 |
| **Refund** | 將付款退回給客戶 |

### 交易狀態

| 狀態 | 含義 |
|--------|--------------|
| **Pending** | 交易已啟動但尚未處理 |
| **Processing** | 正在由付款提供商處理 |
| **Authorized** | 資金已暫存 — 等待收取 |
| **Completed** | 付款成功 |
| **Failed** | 付款被拒絕或發生錯誤 |
| **Voided** | 在資金被收取前取消授權 |
| **Refunded** | 已發出完整退款 |
| **Partially Refunded** | 部分付款已退回 |

### 在交易記錄中您可以看到


# 每筆交易顯示的資訊

- **交易編號** — Spwig 內部的參考編號
- **支付提供商交易編號** — 來自您的支付提供商的參考編號（例如 Stripe 收款編號）
- **金額** — 交易金額和貨幣
- **狀態** 和 **類型**
- **客戶電子郵件** 和 **客戶姓名**
- **支付方式** — 類型（信用卡、銀行轉帳等）和最後四位數字
- **訂單** — 這筆交易所屬的訂單
- **支付提供商帳號** — 哪個支付提供商處理了這筆交易
- **提供商回應** — 支付提供商提供的原始技術回應
- **錯誤訊息** — 如果交易失敗，提供商提供的錯誤原因
- 建立、最後更新和完成的時間戳

### 過濾交易

使用管理員過濾器來縮小交易範圍：
- 狀態（例如，只顯示失敗的交易）
- 類型（例如，只顯示退款）
- 支付提供商帳號
- 日期範圍

這對於當天結帳或調查特定客戶的付款歷史非常有用。

### 什麼時候可以退款？

交易可以在以下情況下退款：
- 其狀態為 **已完成**
- 其類型為 **Charge** 或 **Capture**

要發起退款，請使用 **退款** 這個動作，從訂單詳情頁面進行。透過訂單處理的退款會產生一個類型為 **Refund** 的新交易記錄。

### 授權與捕獲流程

某些支付方式（以及某些支付提供商）支援分離的授權和捕獲。如果您想在發貨前驗證付款，這會很有用：

1. **授權** — 資金會暫存在客戶的信用卡上（狀態：`Authorized`）
2. **捕獲** — 當訂單發貨或完成時觸發
3. 如果在授權窗口內未進行捕獲，暫存將 **自動過期**

交易上的 **Expires At** 欄位顯示授權將會過期的時間。

## 支付 Webhook

支付提供商會透過 Webhook 事件通知您的商店有關支付狀態的變更 — 例如，當支付成功、失敗或提出爭議時。


Spwig 會記錄所有收到的 Webhook。

導覽至 **付款 > 支付 Webhook** 以查看日誌。

### Webhook 記錄顯示的內容

| 欄位 | 說明 |
|-------|-------------|
| **提供者** | 哪個支付提供者發送了 Webhook |
| **事件 ID** | 提供者的唯一事件識別碼 |
| **事件類型** | 事件類型（例如，`payment_intent.succeeded`，`charge.refunded`） |
| **已處理** | Spwig 是否已對此 Webhook 作出反應 |
| **簽名驗證** | Webhook 的安全簽名是否有效 |
| **負載** | 提供者發送的完整資料 |
| **處理結果** | Spwig 的回應動作 |
| **處理錯誤** | 處理過程中發生的任何錯誤 |
| **收到時間** | Webhook 到達的時間 |

### 使用 Webhook 日誌進行疑難排解

如果付款似乎卡住或付款後訂單狀態未更新：

1. 導覽至 **付款 > 支付 Webhook**
2. 依提供者篩選並查找最近的事件
3. 檢查 **已處理** 欄位 — 未處理的 Webhook 可能表示傳遞問題
4. 檢查 **簽名驗證** — 簽名失敗可能表示您的 Webhook 秘密設定錯誤
5. 查看 **處理錯誤** 以確認任何錯誤訊息

重複事件會自動處理 — `事件 ID` 和提供者的組合是唯一的，因此相同的 Webhook 無法處理兩次。

## 支付意圖

支付意圖追蹤從客戶開始支付流程到最終結果的結帳支付生命週期。當客戶在結帳時到達支付步驟時，支付意圖會自動建立。

導覽至 **付款 > 支付意圖** 以查看清單。

### 支付意圖狀態

| 狀態 | 含義 |
|--------|---------|
| **已建立** | 意圖已建立，等待支付方式 |
| **需要支付方式** | 等待客戶輸入他們的信用卡詳細資料 |
| **需要確認** | 已輸入支付詳情，等待確認 |
| **需要操作** | 顧客需要完成一個操作（例如 3D Secure 認證） |
| **處理中** | 支付正在處理中 |
| **成功** | 支付已完成 |
| **已取消** | 支付被放棄或取消 |
| **失敗** | 支付嘗試失敗 |

### 支付意圖到訂單流程

1. 顧客到達結帳支付步驟 → Spwig 建立一個 **支付意圖** 和一個草稿 **訂單**（未付款）
2. 顧客輸入支付詳情並確認
3. 支付提供商處理支付
4. 成功時，訂單會更新為 **已付款**，而支付意圖會移至 **成功**
5. 會建立一個 **支付交易** 記錄，包含最終收費詳情

支付意圖將結帳會話、提供商帳戶和訂單連結在一起 — 給您顧客結帳旅程的完整畫面。

### 使用支付意圖進行支援

如果顧客報告他們已付款但訂單顯示為未付款：

1. 在 **訂單** 中找到顧客的訂單
2. 轉到 **支付 > 支付意圖**，並搜尋與該訂單連結的意圖
3. 檢查意圖狀態 — 如果是 **成功**，請檢查連結的交易
4. 如果意圖是 **需要操作**，顧客可能尚未完成 3D Secure 認證
5. 如果意圖是 **失敗**，錯誤詳情會說明支付被拒絕的原因

## 小提示

- 每天審查失敗的交易 — 失敗模式（例如特定支付方式或國家）可能表明配置問題或詐騙嘗試。
- 當調查支付差異時，Webhook 日誌非常有用。

# 如果訂單已付款但未確認，webhook 日誌通常會告訴你出了什麼問題。
- 授权保留會自動過期 —— 如果你使用授權後捕獲的方式，請確保在授權保留期限結束前（通常為大多數提供商的 7 天）完成履行流程以捕獲資金。
- 交易上的 **Provider Response** 欄位包含來自支付提供商的原始數據。

如果你需要幫助解決特定交易問題，請與提供商的支援團隊分享這些資訊。
- webhook 的簽名驗證失敗應立即進行調查 —— 這可能表示 webhook 秘密配置錯誤，或有人試圖向您的商店發送 fraudulent 的 webhook 事件。
