---
title: 支払い取引
---

支払い取引は、ストアを通じて処理されたすべての支払いイベントの完全な記録です — 支払い、返金、承認、その他。このセクションには、支払いプロバイダーからのウェブフックログと、チェックアウト時に作成された支払いインテントも含まれます。

# 取引の表示内容

各取引では以下が表示されます：
- **取引ID** — Spwigの内部参照
- **プロバイダ取引ID** — お支払いプロバイダからの参照（例：StripeのチャージID）
- **金額** — 取引金額と通貨
- **ステータス** および **タイプ**
- **顧客のメールアドレス** および **顧客名**
- **支払い方法** — タイプ（クレジットカード、銀行振込など）および最後の4桁
- **注文** — この取引が所属する注文
- **プロバイダアカウント** — どの支払いプロバイダが処理したか
- **プロバイダの応答** — 支払いプロバイダからの原始的な技術的な応答
- **エラーメッセージ** — 取引が失敗した場合、プロバイダが提供した理由
- 作成、最終更新、完了のタイムスタンプ

### 取引のフィルタリング

管理画面のフィルタを使用して、以下で取引を絞り込むことができます：
- ステータス（例：失敗した取引のみを表示）
- タイプ（例：返金のみを表示）
- プロバイダアカウント
- 日付範囲

これは、日の終わりの照合や特定の顧客の支払い履歴を調査する際に役立ちます。

### いつ取引を返金できますか？

取引を返金できるのは以下の時です：
- ステータスが **Completed**（完了）
- タイプが **Charge**（チャージ）または **Capture**（キャプチャ）

返金を発行するには、**Refund**（返金）アクションを注文詳細ページから使用してください。注文を通じて処理された返金は、**Refund**（返金）タイプの新しい取引レコードを作成します。

### 承認とキャプチャのフロー

一部の支払い方法（および一部の支払いプロバイダ）は、承認とキャプチャを分離してサポートしています。これは、出荷前に支払いを確認したい場合に役立ちます：

1. **承認** — 顧客のカードから資金が保留されます（ステータス：`Authorized`）
2. **キャプチャ** — 注文が発送されるか、または満たされるときにトリガーされます
3. 承認期間内にキャプチャされなかった場合、保留は自動的に **期限切れ** になります

取引の **Expires At**（有効期限）フィールドは、承認が期限切れになる時間を表示します。

## 支払いウェブフック

支払いプロバイダは、支払いのステータス変更を通知するために、ウェブフックイベントをストアに送信します。例えば、支払いが成功した、失敗した、または紛争が発生した場合などです。

Spwigはすべての入力されたウェブフックをログに記録します。

**支払い > 支払いウェブフック**に移動してログを確認してください。

### ウェブフックの記録が表示する内容

| フィールド | 説明 |
|-------|-------------|
| **プロバイダ** | ウェブフックを送信した支払いプロバイダ |
| **イベントID** | プロバイダのユニークなイベント識別子 |
| **イベントタイプ** | イベントの種類（例：`payment_intent.succeeded`, `charge.refunded`） |
| **処理済み** | Spwigがこのウェブフックに対してアクションを実行したかどうか |
| **署名検証済み** | ウェブフックのセキュリティ署名が有効かどうか |
| **ペイロード** | プロバイダから送信された完全なデータ |
| **処理結果** | Spwigが応答として行った内容 |
| **処理エラー** | 処理中に発生したエラー |
| **受信日時** | ウェブフックが到着した時刻 |

### ウェブフックログを使用したトラブルシューティング

支払いが停止しているように見える、または支払い後に注文ステータスが更新されない場合:

1. **支払い > 支払いウェブフック**に移動します
2. プロバイダでフィルタリングし、最近のイベントを確認します
3. **処理済み**列を確認してください — 処理されていないウェブフックは、配信の問題を示している可能性があります
4. **署名検証済み**を確認してください — 署名が失敗した場合、ウェブフックシークレットが誤って設定されている可能性があります
5. **処理エラー**を確認して、エラーメッセージを確認してください

重複したイベントは自動的に処理されます — `イベントID`とプロバイダの組み合わせはユニークであり、同じウェブフックは2回処理されません。

## 支払いインテント

支払いインテントは、顧客が支払いプロセスを開始した瞬間から最終的な結果まで、チェックアウトの支払いライフサイクルを追跡します。支払いインテントは、顧客がチェックアウトで支払いステップに到達したときに自動的に作成されます。

**支払い > 支払いインテント**に移動してリストを表示してください。

### 支払いインテントのステータス

| ステータス | 意味 |
|--------|---------|
| **作成済み** | 支払い方法の待機中にインテントが作成されました |
| **支払い方法が必要** | 顧客がカード情報を入力するのを待機しています |
| **確認が必要** | 支払い情報が入力され、確認を待機しています |
| **アクションが必要** | 顧客がアクション（例：3D Secure認証）を完了する必要があります |
| **処理中** | 支払いが処理されています |
| **成功** | 支払いが正常に完了しました |
| **キャンセル済み** | 支払いが放棄またはキャンセルされました |
| **失敗** | 支払いの試行に失敗しました |

### 支払いインテントから注文へのフロー

1. 顧客がチェックアウトの支払いステップに到達 → Spwig は **Payment Intent** と未払いのドラフト **Order** を作成します
2. 顧客が支払い情報を入力し、確認します
3. 支払いプロバイダーが支払いを処理します
4. 成功した場合、Order は **Paid** に更新され、Payment Intent は **Succeeded** に移行します
5. 最終的な請求詳細を含む **Payment Transaction** レコードが作成されます

支払いインテントはチェックアウトセッション、プロバイダーのアカウント、および注文をリンクし、顧客のチェックアウトジャーニーの完全な概要を提供します。

### 支払いインテントを使用したサポート

顧客が支払い済みであると報告したが、注文が未払いのままの場合は:

1. **Orders** で顧客の注文を検索します
2. **Payments > Payment Intents** に移動し、その注文に関連付けられたインテントを検索します
3. インテントのステータスを確認します — **Succeeded** の場合は、関連付けられたトランザクションを確認します
4. インテントが **Requires Action** の場合は、顧客が3D Secure認証を完了していない可能性があります
5. インテントが **Failed** の場合は、支払いが拒否された理由を説明するエラーデータを確認します

## Tips

- 毎日失敗したトランザクションを確認してください — 失敗のパターン（例：特定の支払い方法または国）は、設定の問題または詐欺の試みを示している可能性があります。
- 支払いの不一致を調査する際、Webhook ログは非常に価値があります。

# 注文が支払い済みだが確認されていない場合
 webhook ログは通常、何が問題になっているかを教えてくれます。
- 承認保留は自動的に期限切れになります — authorize-then-capture を使用している場合、履行プロセスで保留金額を支払い前に期限切れになる前に確定させる必要があります（多くのプロバイダーでは通常7日です）。
- トランザクションの **Provider Response** フィールドには、支払いプロバイダーからの生のデータが含まれています。

特定のトランザクションの問題を解決するために、この情報をプロバイダーのサポートチームと共有してください。
- webhook での署名検証の失敗はすぐに調査する必要があります — これは webhook シークレットの設定ミスや、不正な webhook イベントを店舗に送信しようとする試みを示している可能性があります。
