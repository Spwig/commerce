# 웹후크 전송 로그

매번 가게가 웹후크를 전송하려 할 때마다 전송 로그 항목이 생성됩니다. 이러한 로그는 정확히 무엇이 전송되었는지, 성공했는지, 재시도 시도 중에 무슨 일이 발생했는지를 확인할 수 있게 합니다. 이 가이드는 전송 로그를 읽는 방법과 전송이 실패할 때 문제를 디버깅하는 방법을 설명합니다.

## 전송 로그 보기

**통합 > 웹후크 전송**으로 이동하여 모든 엔드포인트에 걸쳐 모든 웹후크 전송 시도의 전체 기록을 볼 수 있습니다.

![웹후크 전송 로그](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

리스트는 각 전송의 엔드포인트 이름, 이벤트 유형, 상태, HTTP 응답 코드, 응답 시간, 시도 횟수를 보여줍니다.

전송 로그는 읽기 전용입니다 — 이벤트가 발생할 때 자동으로 생성되며 편집할 수 없습니다.

## 전송 상태

각 전송은 다음 상태 중 하나를 가집니다:

| 상태 | 의미 |
|------|------|
| **대기 중** | 전송이 대기 중이며 아직 시도되지 않았음 |
| **성공** | 수신 서버가 HTTP 2xx 상태 코드로 응답 — 전송 확인 |
| **실패** | 모든 전송 시도가 완료됨 — 전송은 다시 시도되지 않음 |
| **재시도 중** | 최근 시도가 실패했지만 시스템은 예약된 재시도 시간에 다시 시도함 |
| **샌드박스 차단** | 전송이 엔드포인트 URL이 현재 환경에서 접근 불가능하다는 이유로 차단됨 |

수신 서버가 HTTP 2xx 응답 코드(200, 201, 202 등)를 반환하면 전송이 성공으로 간주됩니다. 다른 응답 — 3xx 리디렉션 또는 4xx/5xx 오류 포함 —은 실패로 간주됩니다.

## 전송 필터링

오른쪽의 필터 패널을 사용하여 목록을 좁힐 수 있습니다:

- **상태** — 실패, 재시도, 성공한 전송만 보기
- **이벤트 유형** — 특정 이벤트에 대한 모든 전송 보기 (예: 모든 `order.created` 전송)
- **엔드포인트** — 특정 엔드포인트에 대한 전송 보기
- **생성 시간** — 날짜 범위로 필터링


검색 바를 사용하여 이벤트 유형 또는 엔드포인트 이름으로 검색하거나, 특정한 전송을 ID로 찾을 수 있습니다.

## 전송 세부 정보 보기

임의의 전송을 클릭하여 전체 세부 정보를 확인할 수 있습니다. 전송 기록은 읽기 전용입니다.

### 요약

- **ID** — 이 전송 시도에 대한 고유 식별자
- **엔드포인트** — 이 전송이 전송된 웹후크 엔드포인트 (엔드포인트 기록으로 이동)
- **이벤트 유형** — 이 전송을 트리거한 이벤트 (예: `order.paid`)
- **상태** — 현재 전송 상태

### Payload

**Payload** 섹션은 엔드포인트로 전송된 정확한 JSON 데이터를 보여줍니다. 이는 이벤트 유형, 타임스탬프 및 전체 이벤트 데이터를 포함합니다. 이 섹션을 사용하여 수신 서버가 올바른 데이터 구조를 받고 있는지 확인할 수 있습니다.

### 응답

**응답** 섹션은 서버가 반환한 내용을 보여줍니다:

- **응답 상태 코드** — 서버에서 반환한 HTTP 상태 코드. 색상 표시: 2xx(성공)은 녹색, 4xx(클라이언트 오류)는 노란색, 5xx(서버 오류)는 빨간색.
- **응답 시간** — 서버가 응답하는 데 걸린 시간(밀리초 단위). 색상 표시: 500ms 미만은 녹색, 2초 이내는 노란색, 2초 이상은 빨간색.
- **응답 본문** — 서버 응답 본문(1,000자까지 잘려 표시됨). 이는 웹후크가 서버에 거부된 이유를 파악하는 데 도움이 됩니다.
- **응답 헤더** — 서버에서 반환한 헤더.

### 오류 세부 정보

전송이 실패한 경우 **오류 세부 정보** 섹션은 오류 메시지를 보여줍니다. 예: `Connection refused`, `Timeout after 30s`, 또는 서버에서 반환한 HTTP 오류.

### 재시도 정보

- **시도 횟수** — 시도된 전송 횟수(처음 시도 포함)
- **다음 재시도 시점** — 다음 재시도가 시도될 시간(전송 상태가 **재시도 중**인 경우에만 표시됨)

재시도는 지수 백오프 스케줄을 따릅니다. 즉, 서버가 일시적으로 사용 불가능한 경우를 방지하기 위해 재시도 간격이 증가합니다.

최대 5회 재시도(기본값)로, 재시도 일정은 수시간에 걸쳐 펼쳐집니다.

## 수동으로 실패한 전송 재시도

자동 일정 대신 즉시 전송을 재시도하고 싶다면:

1. 재시도하고자 하는 전송 옆에 있는 체크박스를 선택합니다
2. **Action** 드롭다운에서 **Retry selected deliveries**를 선택합니다
3. **Go**를 클릭합니다

**Success** 상태가 아닌 전송만 재시도 대기열에 추가됩니다. 성공한 전송은 건너뜁니다.

이 기능은 수신 서버의 문제를 수정하고, 대기 없이 실패한 이벤트를 다시 처리하고자 할 때 유용합니다.

## 일반적인 실패 진단

### HTTP 4xx 응답 코드

서버에서 4xx 응답을 받는 경우 일반적으로 요청에 문제가 있다는 의미입니다 — 인증 실패, 엔드포인트 URL 변경, 또는 서버가 페이로드 형식을 거부한 경우입니다. 확인해야 할 사항:

- 엔드포인트 URL이 올바른가요?
- 서버가 HMAC 서명을 올바르게 검증하고 있나요? 불일치는 많은 서버에서 401 또는 403을 반환하게 합니다.
- 페이로드 구조가 변경되었나요? 전송 로그에서 페이로드를 확인하여 서버가 예상하는 내용과 비교해 보세요.

### HTTP 5xx 응답 코드

5xx 응답은 웹훅을 처리하는 중 서버에 내부 오류가 발생했음을 의미합니다. 서버 자체의 오류 로그를 확인하여 문제를 진단하세요.

### Connection refused / Timeout

이러한 오류는 Spwig가 서버에 도달하지 못했음을 의미합니다:

- 서버가 실행되고 공개적으로 접근 가능한가요?
- URL이 올바른가요(포함된 프로토콜 — http 또는 https도 확인하세요)?
- 방화벽이 들어오는 요청을 차단하고 있나요?
- 서버의 응답 시간이 설정된 타임아웃을 초과했나요? 그렇다면 엔드포인트에서 **Timeout** 설정을 증가시키거나, 서버의 웹훅 핸들러를 최적화하여 빠르게 응답하도록 하세요(이상적으로 5초 이내).

### Sandbox Blocked

로컬호스트 URL 또는 내부 네트워크 주소로 전송이 차단됩니다. 웹훅 엔드포인트는 반드시 공개적으로 접근 가능해야 합니다. 개발 중에는 ngrok과 같은 도구를 사용하여 로컬 서버를 공개적으로 노출시킬 수 있습니다.

## 팁

모든 마크다운 포맷, 이미지 경로, 코드 블록 및 기술 용어를 유지하세요.

- **실패**한 배송을 신속히 처리하세요 — 이벤트 데이터는 여전히 payload에 포함되어 있으며, 문제가 해결되면 수동으로 한 번 다시 시도할 수 있습니다.
- 특정 엔드포인트에 대해 많은 **재시도** 배송이 나타난다면, 해당 엔드포인트 기록을 열고 **Health** 섹션을 확인하세요 — 엔드포인트가 자동으로 비활성화될 준비가 되어 있을 수 있습니다.
- 응답 시간이 중요합니다: webhook 핸들러를 빠르게 응답하도록 구성하고(몇 초 이내), payload를 백그라운드에서 비동기적으로 처리하세요.

느린 핸들러는 로직이 올바르더라도 타임아웃 실패를 유발할 수 있습니다.
- **이벤트 유형** 필터를 사용하여 특정 이벤트 유형의 배송 기록을 확인하고, 통합이 올바른 이벤트를 수신하는지 조사할 때 사용하세요.
- 배송 로그는 시간이 지남에 따라 누적됩니다.

날짜 필터를 사용하여 최근 배송에 집중하고, 오래된 기록을 훑어보는 것을 피하세요.
