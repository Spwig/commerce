# ウェブフック配信ログ

ストアがウェブフックを送信しようとするたびに、配信ログが作成されます。これらのログを使用して、実際に何が送信されたか、成功したかどうか、再試行の試み中に何が起きたかを確認できます。このガイドでは、配信ログをどのように読み取り、配信が失敗した場合の問題をデバッグする方法を説明します。

## 配信ログの表示

**Integrations > Webhook Deliveries** に移動して、すべてのエンドポイントにわたるすべてのウェブフック配信試行の完全な履歴を確認できます。

![Webhook delivery logs](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

このリストには、各配信のエンドポイント名、イベントタイプ、ステータス、HTTP応答コード、応答時間、および試行回数が表示されます。

配信ログは読み取り専用です — イベントが発生したときに自動的に作成され、編集することはできません。

## 配信ステータス

各配信には次のステータスのいずれかが設定されます:

| ステータス | 意味 |
|-----------|------|
| **Pending** | 配信はキューにあり、まだ試行されていません |
| **Success** | 受信サーバーがHTTP 2xxステータスコードで応答 — 配信が確認済み |
| **Failed** | すべての配信試行が完了 — 再試行は行われません |
| **Retrying** | 最新の試行が失敗しましたが、システムはスケジュールされた再試行時間に再試行します |
| **Sandbox Blocked** | 配信がブロックされました。エンドポイントURLが現在の環境でアクセスできません |

受信サーバーが任意のHTTP 2xx応答コード（200、201、202など）を返すと、配信は成功とみなされます。他の応答 — 3xxリダイレクトや4xx/5xxエラーなど — はすべて失敗とみなされます。

## 配信のフィルタリング

右側のフィルタパネルを使用してリストを絞り込みます:

- **Status** — 失敗、再試行、または成功したみの配信のみを表示
- **Event Type** — 特定のイベント（例: すべての `order.created` 配信）のすべての配信を表示
- **Endpoint** — 特定のエンドポイントの配信を表示
- **Created At** — 日付範囲でフィルタリング

検索バーを使用して、イベントタイプやエンドポイント名で検索したり、特定の配信をIDで検索したりできます。

## 配信の詳細を表示する

任意の配信をクリックしてその完全な詳細を確認できます。配信履歴は読み取り専用です。

### 概要

- **ID** — この配信試行のユニークな識別子
- **エンドポイント** — この配信が送信された webhook エンドポイント（エンドポイントレコードへのリンク）
- **イベントタイプ** — この配信をトリガーしたイベント（例: `order.paid`）
- **ステータス** — 現在の配信ステータス

### ペイロード

**ペイロード**セクションは、あなたのエンドポイントに送信された正確な JSON データを表示します。これはイベントタイプ、タイムスタンプ、および完全なイベントデータを含みます。これにより、あなたの受信サーバーが正しいデータ構造を受け取っているかを確認できます。

### レスポンス

**レスポンス**セクションは、あなたのサーバーが返した内容を表示します:

- **レスポンスステータスコード** — あなたのサーバーから返された HTTP ステータスコード。色分け: 2xx（成功）は緑色、4xx（クライアントエラー）は黄色、5xx（サーバーエラー）は赤色。
- **レスポンスタイム** — あなたのサーバーが応答に要した時間（ミリ秒単位）。色分け: 500ms未満は緑色、2秒未満は黄色、2秒以上は赤色。
- **レスポンスボディ** — あなたのサーバーの応答ボディ（1,000文字まで切り捨て）。これは、あなたのサーバーが webhook を拒否した理由を特定するのに役立ちます。
- **レスポンスヘッダー** — あなたのサーバーから返されたヘッダー。

### エラー詳細

配信が失敗した場合、**エラー詳細**セクションはエラーメッセージを表示します — 例: `Connection refused`、`Timeout after 30s`、またはあなたのサーバーからの HTTP エラー。

### 再試行情報

- **試行回数** — これまでに実施された配信試行回数（最初の試行を含む）
- **次回の再試行時刻** — 次の再試行が実施される予定の時刻（**再試行中**ステータスの配信のみ表示）

再試行は指数関数的なバックオフスケジュールに従います — 再試行の間隔は、一時的に利用不可なサーバーを過負荷にしないように、各試行ごとに増加します。

最大で5回の再試行（デフォルト）があり、再試行スケジュールは数時間にわたります。

## 手動で配信を再試行する

自動スケジュールを待たずに配信をすぐに再試行したい場合は:

1. 再試行したい配信の横のチェックボックスを選びます
2. **Action** ドロップダウンから **Retry selected deliveries** を選択します
3. **Go** をクリックします

**Success** ステータスでない配信のみが再試行キューに追加されます。成功した配信はスキップされます。

これは、受信サーバーの問題を修正し、待たずに失敗したイベントを再処理したいときに役立ちます。

## 一般的な失敗の診断

### HTTP 4xx レスポンスコード

サーバーからの4xxレスポンスは、通常、リクエストに問題があることを意味します — 認証に失敗、エンドポイントURLが変更された、またはサーバーがペイロード形式を拒否した。確認してください:

- エンドポイントURLは正しいですか？
- サーバーがHMAC署名を正しく検証していますか？不一致は多くのサーバーで401または403を返す原因になります。
- ペイロード構造が変更されましたか？配信ログのペイロードを、サーバーが期待するものと照合してください。

### HTTP 5xx レスポンスコード

5xxレスポンスは、webhookの処理中にサーバーで内部エラーが発生したことを意味します。サーバー自身のエラーログを確認して問題を診断してください。

### 接続拒否 / タイムアウト

これらのエラーは、Spwigがサーバーに全く到達できなかったことを意味します:

- サーバーは稼働中で、公開アクセス可能ですか？
- URLは正しいですか（httpまたはhttpsの正しいプロトコルを含んでいますか）？
- ファイアウォールが入力リクエストをブロックしていますか？
- サーバーの応答時間が設定されたタイムアウトを超えていませんか？もしそうであれば、エンドポイントの**Timeout**設定を増やすか、webhookハンドラーを最適化して、迅速に応答するようにしてください（理想的には5秒以内）。

### Sandbox Blocked

localhost URLや内部ネットワークアドレスへの配信はブロックされています。webhookエンドポイントは必ず公開アクセス可能である必要があります。開発中にローカルサーバーを公開するため、ngrokなどのツールを使用してください。

## Tips
Preserve all markdown formatting, image paths, code blocks, and technical terms.

- **失敗**した配達を迅速に対処してください — イベントデータはまだペイロードに含まれており、問題が解決した後で手動で再試行できます。
- あるエンドポイントで多くの**再試行**配達が見られる場合は、そのエンドポイントのレコードを開き、**Health**セクションを確認してください — エンドポイントが自動的に無効になる可能性があります。
- レスポンス時間は重要です: webhookハンドラを速やかに応答するように設定し（数秒以内）、ペイロードの処理はバックグラウンドで非同期に行うようにしてください。

ハンドラが遅いと、論理が正しい場合でもタイムアウトエラーが発生します。
- **Event Type**フィルタを使用して、特定のイベントタイプの配達履歴を確認し、統合が正しいイベントを受け取っているかどうかを調査してください。
- 配達ログは時間が経つにつれて蓄積されます。

日付フィルタを使用して、最近の配達に焦点を当て、古い履歴を閲覧する必要がないようにしてください。
