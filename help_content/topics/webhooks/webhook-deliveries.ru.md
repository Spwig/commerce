---
title: Журналы доставки вебхуков
---

Каждый раз, когда ваш магазин пытается отправить вебхук, создается запись в журнале доставки. Эти журналы позволяют увидеть, что именно было отправлено, успешно ли это произошло и что произошло во время любых попыток повторной доставки. Это руководство объясняет, как читать журналы доставки и отлаживать проблемы, когда доставки не удается выполнить.

## Просмотр журналов доставки

Перейдите в **Интеграции > Доставки вебхуков**, чтобы увидеть полный исторический список всех попыток доставки вебхуков по всем вашим конечным точкам.

![Журналы доставки вебхуков](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

Список показывает имя конечной точки, тип события, статус, код HTTP-ответа, время ответа и количество попыток для каждой доставки.

Журналы доставки только для чтения — они создаются автоматически при возникновении событий и не могут быть изменены.

## Статусы доставки

Каждая доставка имеет один из следующих статусов:

| Статус | Что это означает |
|--------|---------------|
| **Ожидание** | Доставка находится в очереди и еще не была попытана |
| **Успешно** | Сервер-получатель ответил кодом HTTP 2xx — доставка подтверждена |
| **Неудачно** | Все попытки доставки исчерпаны — доставка не будет повторена |
| **Повторная попытка** | Последняя попытка завершилась неудачно, но система попытается снова в запланированное время повторной доставки |
| **Заблокировано в песочнике** | Доставка была заблокирована, потому что URL-адрес конечной точки недоступен в текущей среде |

Доставка считается успешной, если сервер-получатель возвращает любой код ответа HTTP 2xx (200, 201, 202 и т.д.). Любой другой ответ — включая перенаправления 3xx или ошибки 4xx/5xx — считается неудачей.

## Фильтрация доставок

Используйте панель фильтрации справа, чтобы сузить список:

- **Статус** — Просматривать только неудачные, повторные или успешные доставки
- **Тип события** — Просмотреть все доставки для конкретного события (например, все доставки `order.created`)
- **Конечная точка** — Просмотреть доставки для конкретной конечной точки
- **Создано в** — Фильтрация по диапазону дат

Используйте поле поиска, чтобы искать по типу события или названию конечной точки, или чтобы найти конкретную доставку по ее ID.

## Чтение деталей доставки

Нажмите на любую доставку, чтобы увидеть ее полные детали. Записи доставки только для чтения.

### Обзор

- **ID** — Уникальный идентификатор этой попытки доставки
- **Конечная точка** — Какая конечная точка вебхука была использована (ссылка на запись конечной точки)
- **Тип события** — Какое событие вызвало эту доставку (например, `order.paid`)
- **Статус** — Текущий статус доставки

### Полезная нагрузка

Раздел **Полезная нагрузка** показывает точные данные в формате JSON, которые были отправлены на вашу конечную точку. Это включает в себя тип события, временной штамп и полные данные события. Используйте это, чтобы проверить, получает ли ваш сервер правильную структуру данных.

### Ответ

Раздел **Ответ** показывает, что ваш сервер ответил:

- **Код статуса ответа** — Код HTTP-статуса, возвращенный вашим сервером. Цветовая кодировка: зеленый для 2xx (успех), желтый для 4xx (ошибка клиента), красный для 5xx (ошибка сервера).
- **Время ответа** — Сколько времени ваш сервер потратил на ответ в миллисекундах. Цветовая кодировка: зеленый при менее чем 500 мс, желтый до 2 секунд, красный при более чем 2 секундах.
- **Тело ответа** — Тело ответа вашего сервера (ограничено до 1000 символов). Это может помочь определить, почему ваш сервер отклонил вебхук.
- **Заголовки ответа** — Заголовки, возвращенные вашим сервером.

### Детали ошибки

Если доставка не удалась, раздел **Детали ошибки** показывает сообщение об ошибке — например, `Connection refused`, `Timeout after 30s` или HTTP-ошибка с вашего сервера.

### Информация о повторной попытке

- **Количество попыток** — Сколько попыток доставки было выполнено (включая первую попытку)
- **Следующая попытка в** — Когда будет выполнена следующая попытка (отображается только для доставок со статусом **Повторная попытка**)

Повторные попытки следуют расписанию экспоненциального отката — интервал между повторными попытками увеличивается с каждой попыткой, чтобы избежать перегрузки сервера, который временно недоступен. С максимальным количеством 5 повторных попыток (по умолчанию), расписание повторных попыток охватывает несколько часов.

## Повторная попытка доставки вручную

Если вы хотите сразу повторить доставку, не дожидаясь автоматического расписания:

1. Выберите чекбоксы рядом с доставками, которые хотите повторить
2. В раскрывающемся списке **Действие** выберите **Повторить выбранные доставки**
3. Нажмите **Go**

Только доставки, которые не находятся в статусе **Успешно**, будут добавлены в очередь для повторной попытки. Успешные доставки пропускаются.

Это полезно, когда вы исправили проблему на своем принимающем сервере и хотите повторно обработать неудачные события без ожидания.

## Диагностика распространенных сбоев

### Коды ответа HTTP 4xx

Ответ 4xx от вашего сервера обычно означает, что есть проблема с запросом — аутентификация не удалась, изменился URL-адрес конечной точки, или ваш сервер отклонил формат полезных данных. Проверьте:

- Верен ли URL-адрес конечной точки?
- Правильно ли ваш сервер проверяет подпись HMAC? Несоответствие может вызвать у многих серверов возврат кодов 401 или 403.
- Изменилась ли структура полезных данных? Сравните полезные данные в журнале доставки с тем, что ожидает ваш сервер.

### Коды ответа HTTP 5xx

Ответ 5xx означает, что ваш сервер столкнулся с внутренней ошибкой при обработке веб-хука. Проверьте собственные журналы ошибок сервера, чтобы диагностировать проблему.

### Отказ соединения / Тайм-аут

Эти ошибки означают, что Spwig вообще не смог достичь вашего сервера:

- Работает ли сервер и доступен ли он публично?
- Верен ли URL (включая правильный протокол — http или https)?
- Не блокирует ли огонек входящие запросы?
- Превышает ли время ответа сервера настроенный тайм-аут? Если да, увеличьте настройку **Тайм-аут** для конечной точки или оптимизируйте обработчик веб-хука вашего сервера, чтобы он быстро отвечал (идеально в течение 5 секунд).

### Заблокирована песочница

Доставки блокируются для URL-адресов localhost или внутренних сетевых адресов. Конечные точки веб-хука должны быть публично доступны. Используйте инструмент, например ngrok, во время разработки, чтобы сделать локальный сервер публично доступным.

## Советы

- Сразу же обращайте внимание на доставки со статусом **Неудачно** — данные события все еще в полезной нагрузке, и вы можете вручную повторить попытку после устранения проблемы.
- Если вы видите много доставок со статусом **Повторная попытка** для одной конечной точки, откройте запись конечной точки и проверьте раздел **Состояние** — конечная точка может быть автоматически отключена.
- Важно время ответа: настройте обработчик веб-хука так, чтобы он быстро отвечал (в течение нескольких секунд), и обрабатывал полезные данные асинхронно в фоновом режиме. Медленный обработчик вызывает сбои из-за тайм-аута, даже если ваша логика верна.
- Используйте фильтр **Тип события**, чтобы проверить историю доставок для конкретного типа события при проверке, получает ли ваша интеграция правильные события.
- Журналы доставок накапливаются со временем. Используйте фильтр по дате, чтобы сосредоточиться на недавних доставках и избежать перебора старых записей.
