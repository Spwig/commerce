Webhooks ช่วยให้ร้านค้าของคุณแจ้งระบบภายนอกอัตโนมัติ — เช่น ระบบจัดการสต็อก, ERP, บริการจัดส่ง, หรือแอปพลิเคชันที่กำหนดเอง — เมื่อเกิดเหตุการณ์ใดๆ ในร้านค้าของคุณ แทนที่ระบบเหล่านั้นจะถามซ้ำๆ ว่า "มีการเปลี่ยนแปลงอะไรบ้างหรือไม่?" ร้านค้าของคุณจะส่งการแจ้งเตือนทันทีที่เกิดเหตุการณ์

## ฟังก์ชันของ Webhooks

เมื่อเกิดเหตุการณ์ในร้านค้าของคุณ (มีการสั่งซื้อ, ได้รับการชำระเงิน, สินค้าหมดสต็อก) Spwig จะส่งคำขอ HTTP POST พร้อมข้อมูลเหตุการณ์ไปยัง URL ที่คุณตั้งค่า ระบบปลายทางสามารถดำเนินการกับข้อมูลนั้นได้ทันที — เช่น อัปเดตสต็อก, กระตุ้นการสร้างป้ายจัดส่ง, หรือส่งการแจ้งเตือนที่กำหนดเอง

การใช้ Webhooks ที่พบบ่อยมีดังนี้:

- ส 同步订单与履行合作伙伴
- 当库存变化时更新ERP中的库存
- 触发订单状态更改的短信或推送通知
- 在数据仓库中记录事件以供报告
- 连接到自动化工具如Zapier或Make

## 查看和管理端点

导航到 **集成 > Webhooks** 以查看所有配置的Webhook端点。

![Webhook端点列表](/static/core/admin/img/help/webhooks-overview/endpoint-list.webp)

列表显示每个端点的名称、URL、活动状态、它订阅的事件数量、其健康状态以及它最后一次接收交付的时间。

### 健康指标

**健康** 列可一目了然地显示每个端点的性能情况：

- **健康** — 所有最近的交付都已成功
- **降级** — 有一些最近的失败，但端点仍然处于活动状态
- **不健康/已禁用** — 在连续太多失败（默认为10次）后，端点会自动禁用。您必须在底层问题解决后手动重新启用它。

## 创建Webhook端点

点击 **+ 添加Webhook端点** 以打开设置向导。向导将引导您完成四个步骤。

### 第1步：基本信息


- **ชื่อ** — ป้ายกำกับที่เป็นมิตรเพื่อระบุจุดปลายทางนี้ (เช่น `Order Fulfilment Service` หรือ `Inventory Sync`)
- **URL** — URL ทั้งหมดของเซิร์ฟเวอร์ที่จะรับคำขอ POST จาก webhook

ต้องสามารถเข้าถึงได้จากภายนอก (ไม่ใช่ URL ที่ localhost)
- **คำอธิบาย** — หมายเหตุเพิ่มเติมเกี่ยวกับวัตถุประสงค์ของจุดปลายทางนี้
- **เปิดใช้งาน** — ระบุว่าจุดปลายทางนี้ควรรับการส่งข้อมูลหรือไม่

ถอดการเลือกเพื่อหยุดชั่วคราวโดยไม่ต้องลบจุดปลายทาง

### ขั้นตอนที่ 2: การสมัครรับการแจ้งเหตุ (Event subscriptions)

เลือกว่าเหตุการณ์ใดควรเป็นตัวกระตุ้นให้ส่งข้อมูลไปยังจุดปลายทางนี้ เหตุการณ์ถูกจัดกลุ่มตามประเภท:

#### เหตุการณ์เกี่ยวกับคำสั่งซื้อ

| เหตุการณ์ | เมื่อเกิดเหตุการณ์ |
|-------|---------------|
| `order.created` | มีการสร้างคำสั่งซื้อใหม่ |
| `order.paid` | การชำระเงินสำหรับคำสั่งซื้อได้รับการยืนยัน |
| `order.cancelled` | คำสั่งซื้อถูกยกเลิก |
| `order.fulfilled` | ทุกสินค้าในคำสั่งซื้อถูกส่งออกไปแล้ว |
| `order.partially_fulfilled` | สินค้าบางส่วนในคำสั่งซื้อถูกส่งออกไปแล้ว |
| `order.status_changed` | สถานะของคำสั่งซื้อเปลี่ยนไป |
| `order.note_added` | มีการเพิ่มหมายเหตุในคำสั่งซื้อ |

#### เหตุการณ์เกี่ยวกับการชำระเงิน

| เหตุการณ์ | เมื่อเกิดเหตุการณ์ |
|-------|---------------|
| `payment.received` | มีการรับชำระเงิน |
| `payment.failed` | การพยายามชำระเงินล้มเหลว |
| `payment.pending` | การชำระเงินกำลังรอการยืนยัน |

#### เหตุการณ์เกี่ยวกับการจัดส่ง

| เหตุการณ์ | เมื่อเกิดเหตุการณ์ |
|-------|---------------|
| `shipment.created` | มีการสร้างการจัดส่ง |
| `shipment.shipped` | การจัดส่งถูกส่งออกไป |
| `shipment.delivered` | การจัดส่งถูกส่งถึงผู้รับ |
| `shipment.returned` | การจัดส่งถูกคืน |
| `shipment.tracking_updated` | ข้อมูลการติดตามถูกอัปเดต |

#### เหตุการณ์เกี่ยวกับสต็อกสินค้า

| เหตุการณ์ | เมื่อเกิดเหตุการณ์ |
|-------|---------------|
| `inventory.low_stock` | สต็อกต่ำกว่าเกณฑ์ |
| `inventory.out_of_stock` | สินค้าหมดสต็อก |
| `inventory.restocked` | สินค้าถูกเติมสต็อก |
| `inventory.adjusted` | สต็อกถูกปรับเปลี่ยนโดยตรง |

#### เหตุการณ์เกี่ยวกับสินค้า

product.created", "product.updated", "product.deleted", "product.published", "product.unpublished

#### ข้อความเหตุการณ์ลูกค้า

`customer.created`, `customer.updated`, `customer.deleted`

#### ข้อความเหตุการณ์การสมัครสมาชิก

`subscription.created`, `subscription.activated`, `subscription.renewed`, `subscription.cancelled`, `subscription.expired`, `subscription.paused`, `subscription.resumed`, `subscription.payment_failed`

#### ข้อความเหตุการณ์อื่นๆ

`refund.created`, `refund.completed`, `refund.failed`, `cart.abandoned`, `cart.recovered`, `translation.job_completed`, `translation.job_failed`

เพื่อรับข้อความเหตุการณ์ทั้งหมด ให้สมัครรับข่าวสารจาก `*` (wildcard) ซึ่งมีประโยชน์สำหรับปลายทางบันทึกข้อมูลทั่วไป แต่จะสร้างการจราจรมากขึ้น — สมัครรับเฉพาะเหตุการณ์ที่คุณต้องการจริงสำหรับการผสานระบบในสภาพแวดล้อมการผลิต

### ขั้นตอนที่ 3: การตั้งค่า

- **จำนวนการลองส่งสูงสุด** — จำนวนครั้งที่ Spwig ควรลองส่งข้อความเหตุการณ์ที่ล้มเหลว ก่อนที่จะยอมแพ้ (ค่าเริ่มต้น: 5) แต่ละครั้งที่ลองส่งจะใช้ช่วงเวลาที่เพิ่มขึ้นแบบทวีคูณ
- **เวลาหมดอายุ (วินาที)** — ระยะเวลาที่ควรรอให้เซิร์ฟเวอร์ปลายทางตอบกลับก่อนที่จะถือว่าการส่งข้อความเหตุการณ์ล้มเหลว (ค่าเริ่มต้น: 30 วินาที) เพิ่มค่านี้เฉพาะเมื่อเซิร์ฟเวอร์ของคุณมีความล่าช้าเป็นที่ทราบกัน

### ขั้นตอนที่ 4: ความปลอดภัย

ทุกปลายทาง webhook จะได้รับ **คีย์ลับสำหรับการลงนาม** ที่สร้างโดยอัตโนมัติ — คีย์สุ่ม 64 ตัวอักษร Spwig ใช้คีย์นี้เพื่อลงนามข้อความ webhook ทุกข้อด้วยลายเซ็น HMAC-SHA256

ลายเซ็นจะถูกใส่ในหัวข้อคำขอ `X-Webhook-Signature` เซิร์ฟเวอร์ปลายทางของคุณควรตรวจสอบลายเซ็นนี้เพื่อยืนยันว่าคำขอมาจากร้านคุณจริงๆ และไม่ถูกแก้ไข

คีย์ลับจะแสดงในรูปแบบที่ถูกบล็อกในแดชบอร์ด ใช้ API ของ Spwig เพื่อดูหรือหมุนเวียนคีย์ลับ หากคุณสงสัยว่าคีย์ลับอาจถูกเปิดเผย ให้หมุนเวียนคีย์ลับทันที

## การเปิดใช้งานและปิดใช้งานปลายทาง

เพื่อเปิดหรือปิดใช้งานปลายทางหนึ่งหรือหลายตัวโดยไม่ต้องเปิดแต่ละตัว:

1.

เลือกกล่องถูกต้องที่อยู่ข้างๆ ปลายทางที่คุณต้องการเปลี่ยนแปลง
2.



ใช้ **Action** dropdown เพื่อเลือก **Enable selected endpoints** หรือ **Disable selected endpoints**
3.

คลิก **Go**

เพื่อเปิดใช้งาน endpoint ที่ถูกปิดใช้งานโดยอัตโนมัติเนื่องจากเกิดข้อผิดพลาด ให้เลือก endpoint นั้นและใช้การกระทำ **Reset failure count** จากนั้นเปิดใช้งานอีกครั้ง แก้ไขปัญหาที่ทำให้เกิดข้อผิดพลาดก่อน ไม่เช่นนั้น endpoint จะถูกปิดใช้งานอีกครั้งเร็ว ๆ นี้

## Tips

- สมัครรับเฉพาะเหตุการณ์ที่คุณต้องการจริง ๆ — เหตุการณ์ที่ไม่จำเป็นจะสร้างเสียงรบกวนในบันทึกของคุณและเพิ่มภาระในการส่งข้อมูล
- ตรวจสอบ signature ของ webhook บนเซิร์ฟเวอร์ปลายทางของคุณก่อนประมวลผล payload อย่างแน่ใจ ซึ่งจะช่วยป้องกันคุณจากคำขอที่ถูกปลอมแปลง
- ใช้ฟิลด์ **Description** เพื่อบันทึกว่าระบบหรือการผสานข้อมูลใดที่ endpoint นี้เชื่อมต่อ ซึ่งจะช่วยได้มากเมื่อแก้ไขปัญหาในอนาคต
- ตั้งค่า **Timeout** ให้สูงกว่าเวลาตอบสนองโดยเฉลี่ยของเซิร์ฟเวอร์ของคุณเล็กน้อย เวลาหมดอายุ 10–15 วินาทีนั้นเพียงพอสำหรับการผสานข้อมูลส่วนใหญ่
- หาก endpoint กลายเป็น **Unhealthy** ให้ตรวจสอบบันทึกการส่งข้อมูลก่อน (ดูที่ **Webhook Deliveries**) เพื่อทำความเข้าใจรูปแบบข้อผิดพลาดก่อนเปิดใช้งานอีกครั้ง
- สำหรับการทดสอบ ให้กำหนด webhook ไปยังเครื่องมือเช่น [webhook.site](https://webhook.site) เพื่อตรวจสอบ payload ที่ไม่ผ่านการประมวลผลโดยไม่จำเป็นต้องมีเซิร์ฟเวอร์จริง
