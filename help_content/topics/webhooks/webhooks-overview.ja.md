---
title: ウェブフック概要
---

ウェブフックは、店舗で何かが起きたときに、在庫管理ツール、ERP、配送サービス、またはカスタムアプリケーションなどの外部システムに自動的に通知を送信できる仕組みです。これらのシステムが繰り返し「何か変更がありましたか？」と尋ねる代わりに、店舗はイベントが発生した瞬間に通知を送信します。

## ウェブフックの役割

店舗内でイベントが発生した場合（注文が確定、支払いが受け取られ、商品が在庫切れになるなど）、Spwigは、あなたが設定したURLにイベントデータを含むHTTP POSTリクエストを送信します。受信側のシステムは、そのデータをすぐに処理できます（例：在庫を更新、配送ラベルを発行、カスタム通知を送信など）。

ウェブフックの一般的な用途には以下が含まれます：

- フルフィルメントパートナーとリアルタイムで注文を同期
- 在庫が変更されたときにERPで在庫を更新
- 注文状態の変更時にSMSまたはプッシュ通知をトリガー
- レポート用にデータウェアハウスにイベントを記録
- ZapierやMakeなどの自動化ツールと接続

## エンドポイントの表示と管理

**Integrations > Webhooks** に移動して、設定されたすべてのウェブフックエンドポイントを確認してください。

![ウェブフックエンドポイント一覧](/static/core/admin/img/help/webhooks-overview/endpoint-list.webp)

この一覧には、各エンドポイントの名前、URL、アクティブ状態、サブスクライブしているイベント数、健康状態、最後に配信を受け取った時間などが表示されます。

### 健康状態のインジケーター

**Health**（健康状態）列は、各エンドポイントの性能を一目で確認できるようにしています：

- **Healthy**（健全） — 最近のすべての配信が成功しました
- **Degraded**（劣化） — 最近の一部の失敗がありましたが、エンドポイントはまだアクティブです
- **Unhealthy / Disabled**（不健全/無効） — 連続して多くの失敗（デフォルトで10回）が発生したため、エンドポイントが自動的に無効になりました。根本的な問題が解決した後、手動で再び有効にする必要があります。

## ウェブフックエンドポイントの作成

**+ Add Webhook Endpoint** をクリックして、設定ウィザードを開きます。ウィザードは4つのステップをガイドします。

### ステップ1: 基本情報

- **名前** — このエンドポイントを識別するフレンドリーなラベル（例: `Order Fulfilment Service` または `Inventory Sync`）。
- **URL** — このウェブフックのPOSTリクエストを受信するサーバーの完全なURL。

これはパブリックにアクセス可能である必要があります（localhostのURLではありません）。
- **説明** — このエンドポイントの用途に関するオプションのノート。
- **有効** — このエンドポイントが配信を受けるべきかどうか。

チェックを外すと、エンドポイントを削除せずに一時的に停止できます。

### ステップ2: イベントサブスクリプション

このエンドポイントに配信をトリガーするべきイベントを選択してください。イベントはカテゴリごとにグループ化されています:

#### 注文イベント

| イベント | トリガーされるタイミング |
|-------|---------------|
| `order.created` | 新しい注文が行われた時 |
| `order.paid` | 注文の支払いが確認された時 |
| `order.cancelled` | 注文がキャンセルされた時 |
| `order.fulfilled` | 注文内のすべての商品が発送された時 |
| `order.partially_fulfilled` | 注文内の一部の商品が発送された時 |
| `order.status_changed` | 注文のステータスが変更された時 |
| `order.note_added` | 注文にノートが追加された時 |

#### 支払いイベント

| イベント | トリガーされるタイミング |
|-------|---------------|
| `payment.received` | 支払いが受け取られた時 |
| `payment.failed` | 支払いの試行が失敗した時 |
| `payment.pending` | 支払いが確認待ちになっている時 |

#### 配送イベント

| イベント | トリガーされるタイミング |
|-------|---------------|
| `shipment.created` | 配送が作成された時 |
| `shipment.shipped` | 配送が発送された時 |
| `shipment.delivered` | 配送が届けられた時 |
| `shipment.returned` | 配送が返却された時 |
| `shipment.tracking_updated` | 跟踪情報が更新された時 |

#### 在庫イベント

| イベント | トリガーされるタイミング |
|-------|---------------|
| `inventory.low_stock` | 在庫がしきい値を下回った時 |
| `inventory.out_of_stock` | 商品が在庫切れになった時 |
| `inventory.restocked` | 商品が再在庫された時 |
| `inventory.adjusted` | 在庫が手動で調整された時 |

#### 商品イベント

`product.created`, `product.updated`, `product.deleted`, `product.published`, `product.unpublished`

#### 顧客イベント

`customer.created`, `customer.updated`, `customer.deleted`

#### サブスクリプションイベント

`subscription.created`, `subscription.activated`, `subscription.renewed`, `subscription.cancelled`, `subscription.expired`, `subscription.paused`, `subscription.resumed`, `subscription.payment_failed`

#### その他のイベント

`refund.created`, `refund.completed`, `refund.failed`, `cart.abandoned`, `cart.recovered`, `translation.job_completed`, `translation.job_failed`

すべてのイベントを受信するには、`*`（ワイルドカード）にサブスクライブしてください。これは汎用のログ記録エンドポイントに便利ですが、トラフィックが増える可能性があります — 本番環境での統合に必要なイベントだけにサブスクライブしてください。

### ステップ3: 設定

- **最大リトライ回数** — Spwigが配信に失敗した場合に、諦めるまでに何回リトライするか（デフォルト: 5）。各リトライは指数関数的なバックオフ間隔で行われます。
- **タイムアウト（秒）** — 受信サーバーが応答するのを待つ時間（デフォルト: 30秒）。サーバーが遅いことがわかっている場合にのみこの値を増やすことをお勧めします。

### ステップ4: セキュリティ

すべてのウェブフックエンドポイントには、自動的に生成された**署名シークレット**が割り当てられます — 64文字のランダムなキーです。Spwigはこのシークレットを使用して、すべてのウェブフックペイロードにHMAC-SHA256署名で署名します。

署名は`X-Webhook-Signature`リクエストヘッダーに含まれます。受信サーバーはこの署名を検証し、リクエストが実際にあなたのストアから送信され、改ざんされていないことを確認する必要があります。

シークレットは管理画面でマスクされて表示されます。シークレットを表示またはローテートするには、Spwig APIを使用してください。シークレットが漏洩している可能性がある場合は、すぐにローテートしてください。

## エンドポイントの有効化と無効化

1.

変更したいエンドポイントの横にあるチェックボックスを選択
2.



選択したエンドポイントを**有効にする**または**無効にする**ために、**Action**ドロップダウンを使用してください

クリックして**Go**

失敗によって自動的に無効になったエンドポイントを再び有効にするには、それを選択し、**失敗回数をリセット**アクションを使用した後、再度有効にしてください。失敗の原因を最初に修正してください。それ以外の場合は、すぐに再度無効になる可能性があります。

## ヒント

- 実際に必要なイベントだけを購読してください。不要なイベントはログにノイズを出し、配信負荷を増加させます。

- ペイロードを処理する前に、受信サーバーでウェブフックの署名を常に確認してください。これは、偽装されたリクエストから保護します。

- このエンドポイントがどのシステムまたは統合に接続しているかを記録するために、**Description**フィールドを使用してください。これは、数カ月後にトラブルシューティングするときに役立ちます。

- サーバーの典型的な応答時間をわずかに上回る**Timeout**を設定してください。10〜15秒のタイムアウトは、ほとんどの統合に十分です。

- エンドポイントが**Unhealthy**になると、再び有効にする前に、まず配信ログを確認してください（**Webhook Deliveries**を参照）。失敗のパターンを理解してください。

- テストのためには、[webhook.site](https://webhook.site)などのツールにウェブフックを指すことで、ライブサーバーを必要とせずに生のペイロードを確認できます。
