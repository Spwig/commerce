---
title: บันทึกการส่ง Webhook
---

ทุกครั้งที่ร้านค้าของคุณพยายามส่ง Webhook จะมีการสร้างบันทึกการส่ง (delivery log) ขึ้นมา บันทึกเหล่านี้ช่วยให้คุณเห็นได้อย่างชัดเจนว่าส่งข้อมูลอะไรไป ว่าการส่งนั้นสำเร็จหรือไม่ และเกิดอะไรขึ้นระหว่างการพยายามส่งซ้ำ คู่มือนี้อธิบายวิธีการอ่านบันทึกการส่งและแก้ไขปัญหาเมื่อการส่งล้มเหลว

## การดูบันทึกการส่ง

ไปที่ **Integrations > Webhook Deliveries** เพื่อดูประวัติการส่ง Webhook ทั้งหมดที่เกิดขึ้นกับทุกปลายทาง (endpoints) ของคุณ

![บันทึกการส่ง Webhook](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

รายการนี้แสดงชื่อปลายทาง (endpoint) ประเภทเหตุการณ์ (event type) สถานะ รหัสคำตอบ HTTP เวลาในการตอบกลับ และจำนวนครั้งที่พยายามส่ง

บันทึกการส่งนั้นเป็นแบบอ่านเพียงอย่างเดียว — ถูกสร้างขึ้นโดยอัตโนมัติเมื่อเหตุการณ์เกิดขึ้น และไม่สามารถแก้ไขได้

## สถานะการส่ง

แต่ละการส่งมีสถานะหนึ่งในนี้:

| สถานะ | หมายถึงอะไร |
|--------|---------------|
| **Pending** | การส่งยังอยู่ในคิว และยังไม่ได้พยายามส่ง |
| **Success** | เซิร์ฟเวอร์ปลายทางตอบกลับด้วยรหัส HTTP 2xx — การส่งได้รับการยืนยัน |
| **Failed** | ทุกครั้งที่พยายามส่งได้หมด — การส่งจะไม่ถูกลองอีกครั้ง |
| **Retrying** | ครั้งล่าสุดล้มเหลว แต่ระบบจะลองอีกครั้งในเวลาที่กำหนดสำหรับการลองอีกครั้ง |
| **Sandbox Blocked** | การส่งถูกบล็อกเนื่องจาก URL ปลายทางไม่สามารถเข้าถึงได้ในสภาพแวดล้อมปัจจุบัน |

การส่งถือว่าสำเร็จเมื่อเซิร์ฟเวอร์ปลายทางตอบกลับด้วยรหัส HTTP 2xx ใด ๆ (200, 201, 202, ฯลฯ) ผลตอบกลับอื่น ๆ — รวมถึงการเปลี่ยนเส้นทาง 3xx หรือข้อผิดพลาด 4xx/5xx — จะถูกพิจารณาว่าเป็นความล้มเหลว

## การกรองการส่ง

ใช้แผงกรองด้านขวาเพื่อจำกัดรายการ:

- **Status** — ดูเฉพาะการส่งที่ล้มเหลว การส่งที่กำลังลองอีกครั้ง หรือการส่งที่สำเร็จ
- **Event Type** — ดูการส่งทั้งหมดสำหรับเหตุการณ์เฉพาะ (เช่น ทั้งหมด `order.created`)
- **Endpoint** — ดูการส่งสำหรับปลายทางเฉพาะ
- **Created At** — กรองตามช่วงวันที่

ใช้แถบค้นหาเพื่อค้นหาตามประเภทเหตุการณ์หรือชื่อปลายทาง หรือเพื่อค้นหาการส่งข้อมูลเฉพาะตาม ID ของมัน

## การดูรายละเอียดการส่งข้อมูล

คลิกที่การส่งข้อมูลใดๆ เพื่อดูรายละเอียดทั้งหมด การบันทึกการส่งข้อมูลเป็นเพียงการอ่านเท่านั้น

### สรุป

- **ID** — ตัวระบุที่ไม่ซ้ำกันสำหรับการพยายามส่งข้อมูลนี้
- **Endpoint** — ปลายทาง webhook ที่การส่งข้อมูลนี้ถูกส่งไป (เชื่อมโยงไปยังบันทึกปลายทาง)
- **Event Type** — เหตุการณ์ที่กระตุ้นการส่งข้อมูลนี้ (เช่น `order.paid`)
- **Status** — สถานะการส่งข้อมูลปัจจุบัน

### Payload

ส่วน **Payload** แสดงข้อมูล JSON ที่ถูกส่งไปยังปลายทางของคุณอย่างถูกต้อง ซึ่งรวมถึงประเภทเหตุการณ์ เวลาที่บันทึก และข้อมูลเหตุการณ์ทั้งหมด ใช้ส่วนนี้เพื่อตรวจสอบว่าเซิร์ฟเวอร์รับของคุณได้รับโครงสร้างข้อมูลที่ถูกต้องหรือไม่

### Response

ส่วน **Response** แสดงสิ่งที่เซิร์ฟเวอร์ของคุณตอบกลับมา:

- **Response Status Code** — รหัสสถานะ HTTP ที่เซิร์ฟเวอร์ของคุณคืนค่า แสดงเป็นสี: สีเขียวสำหรับ 2xx (สำเร็จ), สีเหลืองสำหรับ 4xx (ข้อผิดพลาดของไคลเอนต์), สีแดงสำหรับ 5xx (ข้อผิดพลาดของเซิร์ฟเวอร์)
- **Response Time** — เวลาที่เซิร์ฟเวอร์ของคุณใช้ในการตอบกลับในมิลลิวินาที แสดงเป็นสี: สีเขียวหากน้อยกว่า 500 มิลลิวินาที, สีเหลืองถ้าอยู่ในช่วงไม่เกิน 2 วินาที, สีแดงหากมากกว่า 2 วินาที
- **Response Body** — ตัวเนื้อหาของคำตอบจากเซิร์ฟเวอร์ของคุณ (ตัดทอนเหลือ 1,000 ตัวอักษร) สิ่งนี้สามารถช่วยระบุได้ว่าทำไมเซิร์ฟเวอร์ของคุณถึงปฏิเสธ webhook
- **Response Headers** — หัวข้อที่เซิร์ฟเวอร์ของคุณคืนค่า

### Error details

หากการส่งข้อมูลล้มเหลว ส่วน **Error Details** จะแสดงข้อความแสดงข้อผิดพลาด — ตัวอย่างเช่น `Connection refused`, `Timeout after 30s` หรือข้อผิดพลาด HTTP จากเซิร์ฟเวอร์ของคุณ

### Retry information

- **Attempt Count** — จำนวนการพยายามส่งข้อมูลที่ทำไปแล้ว (รวมถึงการพยายามครั้งแรก)
- **Next Retry At** — เวลาที่จะพยายามส่งข้อมูลครั้งต่อไป (แสดงเฉพาะสำหรับการส่งข้อมูลที่อยู่ในสถานะ **Retrying**)

การลองส่งข้อมูลอีกครั้งจะใช้ตารางการหลีกเลี่ยงแบบทวีคูณ — ช่วงเวลาที่ใช้ระหว่างการลองส่งข้อมูลจะเพิ่มขึ้นตามจำนวนการพยายาม เพื่อหลีกเลี่ยงการส่งข้อมูลมากเกินไปไปยังเซิร์ฟเวอร์ที่มีปัญชั่วคราว

ด้วยจำนวนการลองอีกสูงสุด 5 ครั้ง (ค่าเริ่มต้น) ตารางเวลาการลองอีกจะครอบคลุมหลายชั่วโมง

## การลองส่งอีกครั้งด้วยตนเอง

หากคุณต้องการลองส่งอีกครั้งทันทีโดยไม่ต้องรอตารางเวลาอัตโนมัติ:

1. เลือกช่องถัดจากส่งที่คุณต้องการลองอีกครั้ง
2. จากเมนูแบบลงล่าง **Action** เลือก **Retry selected deliveries**
3. คลิก **Go**

เฉพาะการส่งที่ไม่ได้อยู่ในสถานะ **Success** จะถูกเพิ่มเข้าคิวสำหรับการลองอีกครั้ง การส่งที่สำเร็จจะถูกข้าม

มันมีประโยชน์เมื่อคุณแก้ไขปัญหากับเซิร์ฟเวอร์รับของคุณแล้วและต้องการประมวลผลเหตุการณ์ที่ล้มเหลวโดยไม่ต้องรอ

## การวินิจฉัยความล้มเหลวทั่วไป

### รหัสตอบกลับ HTTP 4xx

การตอบกลับ 4xx จากเซิร์ฟเวอร์ของคุณมักหมายถึงมีปัญหากับคำขอ — การตรวจสอบความถูกต้องล้มเหลว, URL ปลายทางเปลี่ยนไป หรือเซิร์ฟเวอร์ของคุณปฏิเสธรูปแบบข้อมูลที่ส่งมา ตรวจสอบ:

- URL ปลายทางถูกต้องหรือไม่?
- เซิร์ฟเวอร์ของคุณตรวจสอบลายเซ็น HMAC ได้ถูกต้องหรือไม่? การไม่ตรงกันจะทำให้เซิร์ฟเวอร์หลายตัวคืนค่า 401 หรือ 403
- โครงสร้างข้อมูลเปลี่ยนไปหรือไม่? ตรวจสอบข้อมูลในบันทึกการส่งกับสิ่งที่เซิร์ฟเวอร์ของคุณคาดไว้

### รหัสตอบกลับ HTTP 5xx

การตอบกลับ 5xx หมายความว่าเซิร์ฟเวอร์ของคุณเกิดข้อผิดพลาดภายในขณะประมวลผล webhook ตรวจสอบบันทึกข้อผิดพลาดของเซิร์ฟเวอร์ของคุณเองเพื่อวินิจฉัยปัญหา

### Connection refused / Timeout

ข้อผิดพลาดเหล่านี้หมายความว่า Spwig ไม่สามารถเข้าถึงเซิร์ฟเวอร์ของคุณได้เลย:

- เซิร์ฟเวอร์กำลังทำงานและสามารถเข้าถึงได้จากสาธารณะหรือไม่?
- URL ถูกต้องหรือไม่ (รวมถึงโปรโตคอลที่ถูกต้อง — http หรือ https)?
- มีการปิดกั้นคำขอที่เข้ามาโดยไฟร์วอลล์หรือไม่?
- เวลาตอบกลับของเซิร์ฟเวอร์เกินค่า timeout ที่กำหนดไว้หรือไม่? หากใช่ ให้เพิ่มค่าการตั้งค่า **Timeout** ที่ปลายทาง หรือปรับปรุงตัวจัดการ webhook ของเซิร์ฟเวอร์ของคุณให้ตอบกลับเร็วขึ้น (โดยทั่วไปภายใน 5 วินาที)

### Sandbox Blocked

การส่งถูกบล็อกไปยัง URL ท้องถิ่นหรือที่อยู่เครือข่ายภายใน เว็บฮุคปลายทางต้องสามารถเข้าถึงได้จากสาธารณะ ใช้เครื่องมือเช่น ngrok ระหว่างการพัฒนาเพื่อเปิดเผยเซิร์ฟเวอร์ท้องถิ่นให้เข้าถึงได้จากสาธารณะ

## เคล็ดลับ

รักษาการจัดรูปแบบ markdown ทั้งหมด ค่าเส้นทางรูปภาพ บล็อกโค้ด และคำศัพท์ทางเทคนิคไว้

- ดำเนินการแก้ไขการจัดส่งที่ **ล้มเหลว** ทันที — ข้อมูลเหตุการณ์ยังคงอยู่ใน payload และคุณสามารถลองส่งอีกครั้งได้หลังจากแก้ไขปัญหาแล้ว
- หากคุณเห็นการจัดส่งที่กำลัง **ลองอีกครั้ง** จำนวนมากสำหรับหนึ่ง endpoint ให้เปิดบันทึก endpoint และตรวจสอบส่วน **Health** — endpoint อาจกำลังจะถูกปิดใช้งานอัตโนมัติ
- เวลาในการตอบสนองมีความสำคัญ: ตั้งค่า handler webhook ของคุณให้ตอบสนองอย่างรวดเร็ว (ภายในไม่กี่วินาที) และประมวลผล payload แบบอะซิงโครนัสในพื้นหลัง

handler ที่ช้าจะทำให้เกิดความล้มเหลวจาก timeout แม้ว่าตรรกะของคุณจะถูกต้อง
- ใช้ตัวกรอง **ประเภทเหตุการณ์** เพื่อตรวจสอบประวัติการจัดส่งสำหรับประเภทเหตุการณ์เฉพาะเมื่อตรวจสอบว่า integration ของคุณได้รับเหตุการณ์ที่ถูกต้องหรือไม่
- บันทึกการจัดส่งจะสะสมขึ้นเรื่อยๆ ตามเวลา

ใช้ตัวกรองวันที่เพื่อโฟกัสที่การจัดส่งล่าสุดและหลีกเลี่ยงการดูประวัติเก่าที่ไม่จำเป็น
