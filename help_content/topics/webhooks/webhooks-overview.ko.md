---
title: 웹후크 개요
---

웹후크는 스토어에서 일어나는 일(예: 주문이 생성됨, 결제가 완료됨, 제품이 품절됨)이 발생할 때마다 외부 시스템(예: 재고 도구, ERP, 출고 서비스, 커스텀 애플리케이션)에 자동으로 알릴 수 있도록 합니다. 이러한 시스템들이 반복적으로 "어떤 변화가 있었나요?"라고 묻는 대신, 스토어는 이벤트가 발생하는 즉시 알림을 전송합니다.

## 웹후크의 기능

스토어에서 이벤트가 발생할 때(예: 주문이 생성됨, 결제가 완료됨, 제품이 품절됨), Spwig는 설정한 URL로 이벤트 데이터를 포함한 HTTP POST 요청을 전송합니다. 수신 시스템은 이 데이터를 즉시 처리할 수 있습니다. 예를 들어, 재고를 업데이트하거나 배송 라벨을 생성하거나 커스텀 알림을 전송할 수 있습니다.

웹후크의 일반적인 사용 사례는 다음과 같습니다:

- 출고 파트너와 실시간으로 주문을 동기화함
- 재고가 변경될 때 ERP의 재고를 업데이트함
- 주문 상태 변경 시 SMS 또는 푸시 알림을 트리거함
- 보고를 위해 데이터 웨어하우스에 이벤트를 기록함
- Zapier 또는 Make와 같은 자동화 도구에 연결함

## 엔드포인트 보기 및 관리

**통합 > 웹후크**로 이동하여 설정된 모든 웹후크 엔드포인트를 볼 수 있습니다.

![웹후크 엔드포인트 목록](/static/core/admin/img/help/webhooks-overview/endpoint-list.webp)

목록은 각 엔드포인트의 이름, URL, 활성 상태, 이벤트 구독 수, 건강 상태, 마지막으로 배송을 받은 시간을 보여줍니다.

### 건강 상태 지표

**건강 상태** 열은 각 엔드포인트의 성능을 한눈에 볼 수 있도록 보여줍니다:

- **건강** — 최근 모든 배송이 성공적으로 완료됨
- **저하됨** — 최근 일부 실패가 있었지만 엔드포인트는 여전히 활성 상태임
- **불건강 / 비활성** — 너무 많은 연속 실패(기본값은 10회) 후 자동으로 비활성화됨. 기본 문제를 해결한 후 수동으로 다시 활성화해야 합니다.

## 웹후크 엔드포인트 생성

**+ 웹후크 엔드포인트 추가**를 클릭하여 설정 마법사를 엽니다. 마법사는 4단계를 따라 안내합니다.

### 단계 1: 기본 정보

- **이름** — 이 엔드포인트를 식별하는 친근한 라벨(예: `Order Fulfilment Service` 또는 `Inventory Sync`).
- **URL** — 이 웹훅 POST 요청을 받을 서버의 전체 URL입니다.

이 URL은 공개적으로 접근할 수 있어야 하며, 로컬호스트 URL은 사용할 수 없습니다.
- **설명** — 이 엔드포인트가 사용되는 목적에 대한 선택적 메모입니다.
- **활성화** — 이 엔드포인트가 요청을 받을지 여부를 결정합니다.

체크를 해제하면 엔드포인트를 삭제하지 않고 일시적으로 중지할 수 있습니다.

### 단계 2: 이벤트 구독

이 엔드포인트로 전송되어야 할 이벤트를 선택합니다. 이벤트는 범주별로 그룹화되어 있습니다:

#### 주문 이벤트

| 이벤트 | 발생 시점 |
|-------|---------------|
| `order.created` | 새 주문이 생성될 때 |
| `order.paid` | 주문에 대한 결제가 확인될 때 |
| `order.cancelled` | 주문이 취소될 때 |
| `order.fulfilled` | 주문의 모든 항목이 배송될 때 |
| `order.partially_fulfilled` | 주문의 일부 항목이 배송될 때 |
| `order.status_changed` | 주문 상태가 변경될 때 |
| `order.note_added` | 주문에 메모가 추가될 때 |

#### 결제 이벤트

| 이벤트 | 발생 시점 |
|-------|---------------|
| `payment.received` | 결제가 수령될 때 |
| `payment.failed` | 결제 시도가 실패할 때 |
| `payment.pending` | 결제가 확인을 기다릴 때 |

#### 배송 이벤트

| 이벤트 | 발생 시점 |
|-------|---------------|
| `shipment.created` | 배송이 생성될 때 |
| `shipment.shipped` | 배송이 발송될 때 |
| `shipment.delivered` | 배송이 도착할 때 |
| `shipment.returned` | 배송이 반품될 때 |
| `shipment.tracking_updated` | 추적 정보가 업데이트될 때 |

#### 재고 이벤트

| 이벤트 | 발생 시점 |
|-------|---------------|
| `inventory.low_stock` | 재고가 임계값 이하로 떨어질 때 |
| `inventory.out_of_stock` | 제품이 품절될 때 |
| `inventory.restocked` | 제품이 재고로 다시 입고될 때 |
| `inventory.adjusted` | 재고가 수동으로 조정될 때 |

#### 제품 이벤트

`product.created`, `product.updated`, `product.deleted`, `product.published`, `product.unpublished`

#### 고객 이벤트

`customer.created`, `customer.updated`, `customer.deleted`

#### 구독 이벤트

`subscription.created`, `subscription.activated`, `subscription.renewed`, `subscription.cancelled`, `subscription.expired`, `subscription.paused`, `subscription.resumed`, `subscription.payment_failed`

#### 기타 이벤트

`refund.created`, `refund.completed`, `refund.failed`, `cart.abandoned`, `cart.recovered`, `translation.job_completed`, `translation.job_failed`

모든 이벤트를 수신하려면 `*` (와일드카드)에 구독하세요. 이는 일반적인 로깅 엔드포인트에 유용하지만 더 많은 트래픽을 생성합니다 — 프로덕션 통합에 실제로 필요한 이벤트만 구독하세요.

### 단계 3: 구성

- **최대 재시도 횟수** — Spwig이 실패한 전송을 포기하기 전에 몇 번 재시도할지 지정합니다(기본값: 5). 각 재시도는 지수 백오프 간격을 사용합니다.
- **타임아웃(초)** — 수신 서버가 응답할 때까지 기다리는 시간(기본값: 30초). 서버가 느리다고 알려진 경우에만 이 값을 증가시킵니다.

### 단계 4: 보안

각 웹훅 엔드포인트는 자동으로 생성된 **서명 비밀**(64자 랜덤 키)을 받습니다. Spwig은 이 비밀을 사용하여 모든 웹훅 페이로드에 HMAC-SHA256 서명을 추가합니다.

서명은 `X-Webhook-Signature` 요청 헤더에 포함됩니다. 수신 서버는 이 서명을 확인하여 요청이 실제로 가게에서 발생했고 조작되지 않았는지 확인해야 합니다.

관리자에서 비밀은 마스킹되어 표시됩니다. 비밀을 확인하거나 회전하려면 Spwig API를 사용하세요. 비밀이 유출되었을 가능성이 있는 경우 즉시 비밀을 회전하세요.

## 엔드포인트 활성화 및 비활성화

한 개 이상의 엔드포인트를 빠르게 활성화하거나 비활성화하려면 각 엔드포인트를 열지 않고:

1.

변경하고자 하는 엔드포인트 옆의 체크박스를 선택하세요
2.

모든 마크다운 형식, 이미지 경로, 코드 블록 및 기술 용어를 유지하세요.

**Action** 드롭다운을 사용하여 **선택한 엔드포인트 활성화** 또는 **선택한 엔드포인트 비활성화**를 선택합니다.
3.

**Go**를 클릭합니다.

자동으로 실패로 인해 비활성화된 엔드포인트를 다시 활성화하려면 해당 엔드포인트를 선택하고 **실패 횟수 초기화** 작업을 수행한 후 다시 활성화합니다. 먼저 실패 원인을 수정한 후에만 다시 활성화해야 합니다. 그렇지 않으면 다시 빠르게 비활성화될 수 있습니다.

## Tips

- 실제로 필요한 이벤트만 구독하세요 — 불필요한 이벤트는 로그에 노이즈를 만들고 전송 부하를 증가시킵니다.
- 페이로드를 처리하기 전에 수신 서버에서 웹후크 서명을 항상 확인하세요. 이는 사기 요청으로부터 보호합니다.
- 이 엔드포인트가 어떤 시스템 또는 통합에 연결되어 있는지 기록하기 위해 **Description** 필드를 사용하세요. 이는 수개월 후 문제 해결 시 도움이 됩니다.
- 서버의 일반적인 응답 시간보다 약간 긴 **Timeout** 값을 설정하세요. 대부분의 통합에는 10–15초의 타임아웃이 충분합니다.
- 엔드포인트가 **Unhealthy** 상태가 되면 먼저 (참조: **Webhook Deliveries**) 전송 로그를 확인하여 실패 패턴을 이해한 후 다시 활성화해야 합니다.
- 테스트용으로 [webhook.site](https://webhook.site)와 같은 도구로 웹후크를 지정하여 실제 서버가 필요 없는 상태에서 원시 페이로드를 확인할 수 있습니다.
