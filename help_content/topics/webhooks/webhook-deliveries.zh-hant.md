# Webhook遞送記錄

每次您的商店嘗試發送 Webhook 時，都會創建一個遞送記錄。這些記錄讓您可以查看實際發送的內容、是否成功，以及在任何重試嘗試期間發生了什麼。本指南說明如何閱讀遞送記錄，並在遞送失敗時進行除錯。

## 查看遞送記錄

導航至 **整合 > Webhook 遞送**，以查看所有端點的所有 Webhook 遞送嘗試的完整歷史記錄。

![Webhook 遞送記錄](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

清單顯示每個遞送的端點名稱、事件類型、狀態、HTTP 回應碼、回應時間，以及嘗試次數。

遞送記錄是只讀的 — 當事件觸發時，它們會自動創建且無法編輯。

## 遞送狀態

每個遞送都有以下狀態之一：

| 狀態       | 含義                                       |
|------------|------------------------------------------|
| **待處理** | 遞送已排隊，尚未嘗試                     |
| **成功**   | 接收伺服器回應 HTTP 2xx 狀態碼 — 遞送確認 |
| **失敗**   | 所有遞送嘗試均已耗盡 — 遞送不會再次重試  |
| **重試中** | 最近一次嘗試失敗，但系統會在預定的重試時間再次嘗試 |
| **沙盒阻擋** | 遞送因端點 URL 在當前環境中無法訪問而被阻擋 |

當接收伺服器返回任何 HTTP 2xx 回應碼（例如 200、201、202 等）時，遞送會被視為成功。任何其他回應 — 包括 3xx 重定向或 4xx/5xx 錯誤 — 都會被視為失敗。

## 過濾遞送

使用右側的過濾面板來縮小清單範圍：

- **狀態** — 只查看失敗、重試中或成功的遞送
- **事件類型** — 查看特定事件的所有遞送（例如，所有 `order.created` 遞送）
- **端點** — 查看特定端點的遞送
- **建立時間** — 按日期範圍過濾

使用搜尋欄透過事件類型或端點名稱進行搜尋，或透過 ID 找到特定的遞送記錄。

## 閲讀遞送詳情

點擊任何遞送以查看其完整詳情。遞送記錄為只讀。

### 概要

- **ID** — 這次遞送嘗試的唯一識別碼
- **Endpoint** — 這個 Webhook 端點是發送到哪個端點（連結到端點記錄）
- **Event Type** — 觸發這次遞送的事件（例如，`order.paid`）
- **Status** — 當前遞送狀態

### 載入資料

**載入資料** 一節顯示傳送至您的端點的精確 JSON 資料。這包括事件類型、時間戳記和完整的事件資料。請使用此功能來驗證您的接收伺服器是否收到正確的資料結構。

### 回應

**回應** 一節顯示您的伺服器回應的內容：

- **Response Status Code** — 您的伺服器返回的 HTTP 狀態碼。顏色標示：2xx（成功）為綠色，4xx（客戶端錯誤）為黃色，5xx（伺服器錯誤）為紅色。
- **Response Time** — 您的伺服器回應所需的時間（以毫秒為單位）。顏色標示：低於 500 毫秒為綠色，高達 2 秒為黃色，超過 2 秒為紅色。
- **Response Body** — 您伺服器回應的內容（截斷為 1,000 個字元）。這有助於識別您的伺服器為何拒絕 Webhook。
- **Response Headers** — 您伺服器返回的標頭。

### 錯誤詳情

如果遞送失敗，**錯誤詳情** 一節會顯示錯誤訊息——例如，`Connection refused`、`Timeout after 30s`，或來自您伺服器的 HTTP 錯誤。

### 重試資訊

- **Attempt Count** — 已進行的遞送嘗試次數（包括首次嘗試）
- **Next Retry At** — 下一次重試的時間（僅在狀態為 **Retrying** 時顯示）

重試遵循指數退避排程——重試之間的間隔會隨著每次嘗試增加，以避免對暫時不可用的伺服器造成過度負載。

最多可進行 5 次重試（預設值），重試時間表會橫跨數小時。

## 手動重試失敗的遞送

如果您希望立即重試遞送，而無需等待自動時間表：

1. 勾選您要重試的遞送旁邊的複選框
2. 從 **Action** 下拉選單中選擇 **Retry selected deliveries**
3. 點擊 **Go**

只有狀態不是 **Success** 的遞送會被排入重試佇列。已經成功的遞送會被跳過。

當您在接收伺服器上解決了問題，並希望在不等待的情況下重新處理失敗事件時，這會很有幫助。

## 診斷常見失敗

### HTTP 4xx 回應碼

來自您伺服器的 4xx 回應通常表示請求有問題 — 認證失敗、端點 URL 更改，或您的伺服器拒絕了負載格式。請檢查：

- 端點 URL 是否正確？
- 您的伺服器是否正確驗證 HMAC 簽名？不匹配會導致許多伺服器返回 401 或 403。
- 負載結構是否已更改？請將遞送日誌中的負載與您的伺服器預期的內容進行比對。

### HTTP 5xx 回應碼

5xx 回應表示您的伺服器在處理 Webhook 時遇到了內部錯誤。請檢查您伺服器本身的錯誤日誌以診斷問題。

### 連線被拒絕 / 超時

這些錯誤表示 Spwig 完全無法連接到您的伺服器：

- 伺服器是否正在運行且可公開訪問？
- URL 是否正確（包括正確的協議 — http 或 https）？
- 防火牆是否阻止了進來的請求？
- 伺服器的回應時間是否超過設定的超時時間？如果是，請在端點上增加 **Timeout** 設定，或優化您的伺服器 Webhook 處理程式，使其快速回應（理想情況下在 5 秒內）。

### Sandbox Blocked

遞送到 localhost URL 或內部網絡地址會被阻止。Webhook 端點必須可以公開訪問。在開發過程中，使用類似 ngrok 的工具來公開本地伺服器。

## 小技巧

保留所有 markdown 格式、圖片路徑、程式碼塊和技術術語。

- 對 **失敗** 的遞送進行及時處理 — 事件資料仍然在負載中，問題解決後您可以手動重試一次。
- 如果看到某個端點有許多 **重試中** 的遞送，請打開該端點記錄並檢查 **健康狀態** 一節 — 這個端點可能即將被自動停用。
- 回應時間很重要：請配置您的 Webhook 處理器快速回應（在幾秒內），並在後台異步處理負載。

處理器速度慢會導致超時失敗，即使您的邏輯是正確的。
- 使用 **事件類型** 過濾器來檢查特定事件類型的遞送歷史，以確認您的整合是否收到了正確的事件。
- 遞送日誌會隨著時間累積。

使用日期過濾器來專注於最近的遞送，避免翻閱過往的歷史記錄。
