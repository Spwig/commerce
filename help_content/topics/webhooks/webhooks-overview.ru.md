---
title: Общее описание вебхуков
---

Вебхуки позволяют вашему магазину автоматически уведомлять внешние системы — такие как инвентарные инструменты, ERP, службы доставки или пользовательские приложения — всякий раз, когда происходит что-либо в вашем магазине. Вместо того чтобы эти системы постоянно спрашивали «что-нибудь изменилось?», ваш магазин отправляет уведомление в момент возникновения события.

## Что делают вебхуки

Когда происходит событие в вашем магазине (заказ оформлен, получена оплата, товар вышел из продажи), Spwig отправляет HTTP-запрос POST с данными события на URL, который вы настроите. Получающая система может сразу же действовать на основе этих данных — например, обновлять запасы, запускать маркировку доставки или отправлять пользовательское уведомление.

Часто используемые вебхуки включают в себя:

- Синхронизацию заказов в реальном времени с партнером по доставке
- Обновление запасов в ERP при изменении запасов
- Запуск SMS или уведомлений для изменения статуса заказа
- Запись событий в хранилище данных для отчетности
- Подключение к инструментам автоматизации, таким как Zapier или Make

## Просмотр и управление конечными точками

Перейдите в **Интеграции > Вебхуки**, чтобы увидеть все настроенные конечные точки вебхуков.

![Список конечных точек вебхуков](/static/core/admin/img/help/webhooks-overview/endpoint-list.webp)

Список показывает имя каждой конечной точки, URL, статус активности, количество событий, на которые она подписана, статус ее работоспособности и время последнего получения данных.

### Индикаторы работоспособности

Столбец **Работоспособность** показывает на первый взгляд, насколько хорошо работает каждая конечная точка:

- **Работоспособна** — Все последние доставки были успешными
- **Ограниченная работоспособность** — Некоторые недавние сбои, но конечная точка все еще активна
- **Неработоспособна / Отключена** — Конечная точка была автоматически отключена после слишком большого количества последовательных сбоев (по умолчанию 10). Вы должны вручную снова включить ее после устранения основной проблемы.

## Создание конечной точки вебхука

Нажмите **+ Добавить конечную точку вебхука**, чтобы открыть мастер настройки. Мастер проведет вас через четыре шага.

### Шаг 1: Основная информация

- **Название** — дружелюбная метка для идентификации этой конечной точки (например, `Сервис выполнения заказов` или `Синхронизация запасов`).
- **URL** — полный URL сервера, который будет получать POST-запросы вебхука. Это должно быть публично доступным URL (не локальный URL).
- **Описание** — необязательные примечания о том, для чего используется эта конечная точка.
- **Активна** — определяет, должна ли эта конечная точка получать данные. Снимите флажок, чтобы временно приостановить, не удаляя конечную точку.

### Шаг 2: Подписки на события

Выберите, какие события должны запускать доставку на эту конечную точку. События сгруппированы по категориям:

### События заказов

| Событие | Когда оно срабатывает |
|--------|----------------------|
| `order.created` | Создан новый заказ |
| `order.paid` | Оплата заказа подтверждена |
| `order.cancelled` | Заказ отменен |
| `order.fulfilled` | Все товары в заказе отправлены |
| `order.partially_fulfilled` | Некоторые товары в заказе отправлены |
| `order.status_changed` | Статус заказа изменен |
| `order.note_added` | Добавлен комментарий к заказу |

### События оплаты

| Событие | Когда оно срабатывает |
|--------|----------------------|
| `payment.received` | Получена оплата |
| `payment.failed` | Попытка оплаты завершилась неудачей |
| `payment.pending` | Оплата ожидает подтверждения |

### События доставки

| Событие | Когда оно срабатывает |
|--------|----------------------|
| `shipment.created` | Создана доставка |
| `shipment.shipped` | Доставка отправлена |
| `shipment.delivered` | Доставка получена |
| `shipment.returned` | Доставка возвращена |
| `shipment.tracking_updated` | Обновлена информация о трекинге |

### События запасов

| Событие | Когда оно срабатывает |
|--------|----------------------|
| `inventory.low_stock` | Запасы падают ниже порога |
| `inventory.out_of_stock` | Товар выходит из продажи |
| `inventory.restocked` | Товар пополняется |
| `inventory.adjusted` | Запасы вручную изменены |

### События товаров

`product.created`, `product.updated`, `product.deleted`, `product.published`, `product.unpublished`

### События клиентов

`customer.created`, `customer.updated`, `customer.deleted`

#### События подписки

`subscription.created`, `subscription.activated`, `subscription.renewed`, `subscription.cancelled`, `subscription.expired`, `subscription.paused`, `subscription.resumed`, `subscription.payment_failed`

#### Другие события

`refund.created`, `refund.completed`, `refund.failed`, `cart.abandoned`, `cart.recovered`, `translation.job_completed`, `translation.job_failed`

Чтобы получить все события, подпишитесь на `*` (диапазон). Это полезно для общего назначения логирование конечных точек, но создает больше трафика — подписывайтесь только на события, которые действительно нужны для производственных интеграций.

### Шаг 3: Настройка

- **Максимальное количество попыток** — сколько раз Spwig должен повторять доставку, прежде чем отказаться (по умолчанию: 5). Каждая повторная попытка использует экспоненциальное отступление.
- **Таймаут (секунды)** — сколько времени ждать, пока сервер получателя не ответит, прежде чем пометить доставку как неудачную (по умолчанию: 30 секунд). Увеличивайте это только если ваш сервер известен своей медлительностью.

### Шаг 4: Безопасность

Каждая конечная точка вебхука получает автоматически сгенерированный **секрет подписи** — 64-символьный случайный ключ. Spwig использует этот секрет для подписания каждого вебхук-пейлоада с помощью подписи HMAC-SHA256.

Подпись включается в заголовок запроса `X-Webhook-Signature`. Ваш сервер получателя должен проверить эту подпись, чтобы подтвердить, что запрос действительно пришел от вашего магазина и не был изменен.

Секрет отображается в маске в админке. Чтобы увидеть или обновить секрет, используйте API Spwig. Сразу обновите свой секрет, если подозреваете, что он скомпрометирован.

## Включение и отключение конечных точек

Чтобы быстро включить или отключить одну или несколько конечных точек без открытия каждой из них:

1. Выберите флажки рядом с конечными точками, которые вы хотите изменить
2. Используйте выпадающий список **Действие**, чтобы выбрать **Включить выбранные конечные точки** или **Отключить выбранные конечные точки**
3. Нажмите **Go**

Чтобы снова включить конечную точку, которая была автоматически отключена из-за сбоев, выберите ее и используйте действие **Сбросить счетчик сбоев**, затем снова включите. Сначала исправьте то, что вызвало сбои, иначе она снова будет отключена.

## Советы

- Подписывайтесь только на события, которые действительно нужны — ненужные события создают шум в ваших журналах и увеличивают нагрузку на доставку.
- Всегда проверяйте подпись вебхука на вашем сервере получателя перед обработкой пейлоада. Это защищает вас от поддельных запросов.
- Используйте поле **Описание**, чтобы записать, какую систему или интеграцию эта конечная точка подключает. Это помогает при устранении неполадок спустя месяцы.
- Установите **Таймаут** немного выше, чем типичное время отклика вашего сервера. Таймаут в 10–15 секунд достаточно для большинства интеграций.
- Если конечная точка становится **Нездоровой**, сначала проверьте журналы доставки (см. **Доставки вебхуков**), чтобы понять паттерн сбоя, прежде чем снова включить ее.
- Для тестирования направьте вебхуки на инструмент, например, [webhook.site](https://webhook.site), чтобы проверить сырые пейлоады без необходимости живого сервера.
