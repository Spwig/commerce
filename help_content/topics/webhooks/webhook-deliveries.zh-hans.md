---
title: Webhook交付日志
---

每次您的商店尝试发送webhook时，都会创建一条交付日志条目。这些日志可让您查看确切发送了什么内容、是否成功，以及在任何重试尝试期间发生了什么。本指南将解释如何阅读交付日志，并在交付失败时进行调试。

## 查看交付日志

导航到 **集成 > Webhook交付**，以查看所有端点的所有webhook交付尝试的完整历史记录。

![Webhook交付日志](/static/core/admin/img/help/webhook-deliveries/delivery-list.webp)

列表显示了每个交付的端点名称、事件类型、状态、HTTP响应代码、响应时间和尝试次数。

交付日志是只读的——当事件触发时，它们会自动创建，且无法编辑。

## 交付状态

每个交付都有以下状态之一：

| 状态 | 含义 |
|------|------|
| **待处理** | 交付已排队，尚未尝试 |
| **成功** | 接收服务器返回了HTTP 2xx状态代码——交付已确认 |
| **失败** | 所有交付尝试已用尽——交付将不再重试 |
| **重试中** | 最近一次尝试失败，但系统将在计划的重试时间再次尝试 |
| **沙盒阻止** | 由于端点URL在当前环境中无法访问，交付被阻止 |

当接收服务器返回任何HTTP 2xx响应代码（200、201、202等）时，交付被视为成功。任何其他响应——包括3xx重定向或4xx/5xx错误——都会被视为失败。

## 过滤交付

使用右侧的筛选面板来缩小列表范围：

- **状态** — 仅查看失败、重试中或成功的交付
- **事件类型** — 查看特定事件的所有交付（例如，所有 `order.created` 交付）
- **端点** — 查看特定端点的交付
- **创建时间** — 按日期范围筛选

使用搜索栏按事件类型或端点名称进行搜索，或通过其ID查找特定的投递记录。

## 阅读投递详情

点击任意投递以查看其完整详情。投递记录为只读。

### 概览

- **ID** — 此投递尝试的唯一标识符
- **端点** — 此投递发送到的 Webhook 端点（链接到端点记录）
- **事件类型** — 触发此投递的事件（例如，`order.paid`）
- **状态** — 当前投递状态

### 负载

**负载**部分显示发送到您的端点的确切 JSON 数据。这包括事件类型、时间戳和完整的事件数据。使用此部分验证您的接收服务器是否接收到正确的数据结构。

### 响应

**响应**部分显示您的服务器返回的内容：

- **响应状态码** — 您的服务器返回的 HTTP 状态码。颜色编码：绿色表示 2xx（成功），黄色表示 4xx（客户端错误），红色表示 5xx（服务器错误）。
- **响应时间** — 您的服务器响应所需的时间（以毫秒为单位）。颜色编码：绿色表示低于 500 毫秒，黄色表示最多 2 秒，红色表示超过 2 秒。
- **响应正文** — 您的服务器响应正文（截断为 1,000 个字符）。这有助于识别您的服务器为何拒绝 Webhook。
- **响应头** — 您的服务器返回的头信息。

### 错误详情

如果投递失败，**错误详情**部分会显示错误信息 — 例如，`连接被拒绝`、`30 秒后超时`，或来自您服务器的 HTTP 错误。

### 重试信息

- **尝试次数** — 已尝试的投递次数（包括首次尝试）
- **下次重试时间** — 下一次重试的时间（仅在投递状态为 **重试中** 时显示）

重试遵循指数退避计划 — 每次尝试之间的间隔会增加，以避免在服务器暂时不可用时造成过载。

最多允许 5 次重试（默认值），重试计划将持续数小时。

## 手动重试失败的交付

如果您希望立即重试交付，而无需等待自动计划：

1. 勾选您要重试的交付旁边的复选框
2. 从 **Action** 下拉菜单中选择 **Retry selected deliveries**（重试所选交付）
3. 点击 **Go**

只有状态不是 **Success** 的交付会被排队重试。成功的交付会被跳过。

当您修复了接收服务器的问题，并希望在不等待的情况下重新处理失败事件时，这非常有用。

## 诊断常见失败

### HTTP 4xx 响应代码

来自您的服务器的 4xx 响应通常表示请求存在问题 — 认证失败、端点 URL 已更改，或您的服务器拒绝了有效负载格式。请检查：

- 端点 URL 是否正确？
- 您的服务器是否正确验证了 HMAC 签名？不匹配会导致许多服务器返回 401 或 403。
- 有效负载结构是否已更改？将交付日志中的有效负载与您的服务器预期的内容进行对比。

### HTTP 5xx 响应代码

5xx 响应表示您的服务器在处理 Webhook 时遇到了内部错误。请检查您服务器自身的错误日志以诊断问题。

### Connection refused / Timeout（连接被拒绝 / 超时）

这些错误意味着 Spwig 完全无法连接到您的服务器：

- 服务器是否正在运行并且可以公开访问？
- URL 是否正确（包括正确的协议 — http 或 https）？
- 防火墙是否阻止了传入请求？
- 服务器的响应时间是否超过了配置的超时时间？如果是，请在端点上增加 **Timeout** 设置，或优化您的服务器 Webhook 处理程序以快速响应（理想情况下在 5 秒内）。

### Sandbox Blocked（沙箱被阻止）

交付被阻止到 localhost URL 或内部网络地址。Webhook 端点必须可以公开访问。在开发过程中使用类似 ngrok 的工具来公开本地服务器。

## Tips

保留所有 markdown 格式、图片路径、代码块和专业术语。

- 及时处理 **失败** 的配送 - 事件数据仍然在负载中，问题修复后可以手动重试一次。
- 如果看到某个端点有大量 **重试中** 的配送，请打开该端点记录并检查 **健康状况** 部分 - 该端点可能即将被自动停用。
- 响应时间很重要：配置你的 Webhook 处理程序以快速响应（几秒内），并在后台异步处理负载。

响应处理程序过慢会导致超时失败，即使你的逻辑是正确的。
- 使用 **事件类型** 过滤器来检查特定事件类型的配送历史，以调查你的集成是否接收到了正确的事件。
- 配送日志会随着时间累积。

使用日期过滤器来关注最近的配送，避免浏览过多的历史记录。
