---
title: ロイヤルティキャンペーン
---

ロイヤルティキャンペーンを使用すると、日常的な報酬ルールを超えた、時間限定のプロモーションや自動報酬を実行できます。週末のポイント2倍キャンペーン、顧客の誕生日を祝う、アクティブでない顧客を戻す、特定の会員グループにターゲットとしたボーナスを提供するなど、さまざまな目的で使用できます。

各キャンペーンは、トリガーまたはスケジュール、適用される会員、および実行するアクションを定義します。アクティブになると、キャンペーンは自動的に実行されます。一度設定すれば、Spwigが残りの処理をすべて行います。

## キャンペーンの種類

| 種類 | 実行されるタイミング |
|------|---------------|
| **トリガーに基づく** | 特定のイベントが発生したとき（例：購入が行われる、誕生日が検出される） |
| **スケジュールされた** | 繰り返しのスケジュールで（毎日、毎週、毎月） |
| **手動** | 管理画面から明示的に実行するときのみ |
| **行動に基づく** | 顧客が行動パターンに一致したとき（例：購入せずに閲覧する） |

## キャンペーンの作成

**プロモーション > ロイヤルティキャンペーン**に移動し、**+ ロイヤルティキャンペーンの追加**をクリックします。

### ステップ1: 基本情報

- **名前** — 管理画面内でのみ表示される、明確で説明的な名前（例: `Birthday Bonus — 200 Points`）
- **スラッグ** — 名前から自動生成され、内部で使用される
- **説明** — キャンペーンの目的に関するオプションのノート
- **キャンペーンの種類** — 上記の表から種類を選択します

### ステップ2: トリガーまたはスケジュール

**トリガーに基づくキャンペーン**の場合、キャンペーンを実行する**トリガーイベント**を設定します。利用可能なトリガーには以下が含まれます:

トリガー条件として **Trigger Conditions** をJSONオブジェクトとして追加して、キャンペーンが発火するタイミングをさらにフィルタリングできます。例えば、注文額が$100以上の場合にのみトリガーするには:

{
  "min_order_amount": 100
}

**定期的なキャンペーン**の場合は、**Schedule Type**（日次、週次、月次、またはカスタムCron）を設定し、**Schedule Config**フィールドでタイミングを構成します:

{
  "hour": 9,
  "minute": 0
}

### ステップ3: actions

**Actions**フィールドは、キャンペーンがトリガーされたときに何が起こるかを定義します。アクションオブジェクトのJSON配列を入力します。最も一般的なアクションはボーナスポイントの付与です:

[
  {
    "type": "award_points",
    "points": 200,
    "description": "バースデーボーナス — 会員になっていただきありがとうございます！"
  }
]

他の利用可能なアクションには、メール通知の送信やバッジの付与などがあります。提供元コンポーネントのドキュメントを参照して、フルリストを確認してください。

### ステップ4: targeting

ターゲティングフィールドを使用して、キャンペーンが適用される会員を制御します:

- **Target All Members** — デフォルトでチェックされています; このキャンペーンはすべてのアクティブなロイヤリティ会員に適用されます

- **Target Segment** — キャンペーンを特定のセグメントに属する会員に限定します（下記の[Segments](#managing-member-segments)を参照）

- **Target Tiers** — キャンペーンを特定のロイヤリティランクに属する会員に限定します

### ステップ5: 制限とクールダウン

- **1人の会員あたりの最大トリガー数** — 同じ会員がこのキャンペーンから利益を得られる回数。一度限りのボーナス（例: 誕生日報酬）には`1`に設定します。無制限にする場合は空白のままにしてください。
- **クールダウン日数** — 同じ会員に対するキャンペーンのトリガー間の最小日数。例えば、`365`に設定することで、誕生日キャンペーンが1年以内に複数回発火しないようにします。

### ステップ6: キャンペーン日時

**開始日**と**終了日**を設定して、キャンペーンを時間限定にします。両方を空白にすると、キャンペーンは継続されます。

キャンペーンは次のいずれかのステータスになります:

| ステータス | 説明 |
|--------|-------------|
| **下書き** | 作成済みだがまだアクティブになっていない。設定やテストに安全 |
| **アクティブ** | 実行中で、条件が満たされるとトリガーされます |
| **一時停止** | 設定を失うことなく一時的に停止 |
| **終了** | 終了日を過ぎており、もうトリガーされません |
| **アーカイブ** | アクティブなリストから非表示になりますが、記録のために保存されます |

すべてのフィールドを入力したら、**保存**をクリックしてください。その後、ステータスを**アクティブ**に変更してキャンペーンを開始してください。

## 実用的な例

### 例: ダブルポイントの週末

**シナリオ:** 特定の週末にすべての注文に対して2倍のポイントを付与します。

| フィールド | 値 |
|-------|-------|
| 名前 | `Double Points Weekend — March` |
| キャンペーンタイプ | Trigger-Based |
| トリガーイベント | Order Placed |
| アクション | `["{\"type\": \"award_points_multiplier\", \"multiplier\": 2.0}"]` |
| 開始日 | 金曜日の夜 |
| 終了日 | 日曜日の午前0時 |
| すべての会員を対象にする | 〇 |

### 例: 誕生日ボーナス

**シナリオ:** ロイヤルティ会員全員に誕生日に200ポイントのボーナスを付与します。

| フィールド | 値 |
|-------|-------|
| 名前 | `Birthday Bonus` |
| キャンペーンタイプ | Trigger-Based |
| トリガーイベント | Customer Birthday |
| アクション | `["{\"type\": \"award_points\", \"points\": 200, \"description\": \"Happy birthday from us!\"}"]` |
| 1人の会員あたりの最大トリガー数 | 1 |
| クールダウン日数 | 365 |
| すべての会員を対象にする | 〇 |

### 例: リカバリーキャンペーン


**シナリオ:** 90日間購入していない会員に100ポイントのボーナスを送信する。

| フィールド | 値 |
|-------|-------|
| 名前 | `90-Day Win-Back Bonus` |
| キャンペーンタイプ | トリガーに基づく |
| トリガーイベント | 90日間アクティブでない |
| アクション | `["{\"type\": \"award_points\", \"points\": 100, \"description\": \"We miss you — here are some bonus points\"}"]` |
| 会員ごとの最大トリガー数 | 1 |
| 冷却期間（日） | 180 |
| 全会員を対象にする | チェック済み |

## 会員セグメントの管理

セグメントを使用すると、ロイヤルティ会員の特定のグループをキャンペーンの対象にできます。**プロモーション > ロイヤルティセグメント**に移動して管理してください。

### セグメントタイプ

| タイプ | 説明 |
|------|-------------|
| **ルールベース** | ルールによって会員資格が決定される（例：1,000ポイント以上を持つ会員） |
| **動的計算** | 実時間の基準からオンデマンドで会員資格が計算される |
| **手動割り当て** | 会員をセグメントに手動で追加する |

### セグメントの作成

1. **プロモーション > ロイヤルティセグメント**に移動し、**+ Add Loyalty Segment**をクリックします。
2. 次の項目を入力します：
   - **名前** — 説明的な名前（例：`High-Value Customers`, `Silver Tier Members`）
   - **Slug** — 自動生成される
   - **Criteria Type** — 会員資格がどのように決定されるか
   - **Criteria Config** — 会員資格のルールを定義するJSONオブジェクト
3. **保存**をクリックします。

#### 例：500ポイント以上を持つ会員のセグメント

```json
{
  "min_available_points": 500
}
```

#### 例：ゴールドティア会員のみのセグメント

```json
{
  "tier_slugs": ["gold"]
}
```

セグメント一覧の**会員数**列は、現在一致する会員数を表示します。セグメントをクリックし、データが変更された場合、**Refresh Member Count**アクションを使用して再計算してください。

## キャンペーンのパフォーマンスの追跡

### キャンペーン実行履歴

**プロモーション > Campaign Executions**に移動して、どのキャンペーンがどの会員に対して実行されたかの記録を確認できます。各実行レコードは、どのキャンペーンが実行され、どの会員に対して実行され、その結果を表示します。

### キャンペーンのリーチの確認

キャンペーンのレコードを開いて、**Times Triggered**のカウントとキャンペーンが最後に実行されたタイミングを確認できます。

これにより、キャンペーンから利益を得た会員数を素早く把握できます。

## Tips

- まず、キャンペーンを**Draft**ステータスで作成して、本番公開する前にすべての設定を確認してください
- 一度限りのボーナスキャンペーン（誕生日、初購入、登録）では、**Max Triggers Per Member**を使用して、顧客がボーナスを複数回獲得できないようにしてください
- **Target Segment**とトリガーに基づくキャンペーンを組み合わせて、階層限定のプロモーションを実施できます — 例えば、ゴールドおよびプラチナ会員だけに購入時にポイントを2倍にするなど
- リカバリキャンペーンには**Cooldown Days**の値を設定して、会員が少量の購入後に再びアクティブでなくなると、キャンペーンが頻繁に送信されるのを防ぎます
- 現在アクティブなプロモーションを把握するための最良のツールはキャンペーン一覧です — 新しいオファーを公開する前に確認して、キャンペーンが意図せず重複しないようにしてください
- 終了したキャンペーンは削除する代わりにアーカイブしてください。これにより、過去に実施したプロモーションとその実施時期の履歴を保持できます
