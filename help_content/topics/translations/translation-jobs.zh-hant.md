---
title: 翻譯工作
---

翻譯工作可自動批量翻譯大量內容。與手動逐一翻譯產品不同，您可以創建一個工作，讓系統在後台翻譯整個目錄或特定子集。工作是異步執行的，因此在數百或數千個欄位自動翻譯的同時，您仍可繼續工作。

在啟用新語言、導入新產品或補上未翻譯內容時，使用翻譯工作。

## 什麼是翻譯工作？

翻譯工作是後台任務，可：

1. **掃描內容**，尋找可翻譯的欄位（產品、頁面、部落格文章等）
2. **根據工作範圍識別未翻譯或過時的欄位**
3. **將欄位發送到翻譯引擎**（本地 AI 模型或外部提供商）
4. **將翻譯結果保存回內容**
5. **報告完成情況**，並提供翻譯欄位的統計數據

工作透過 Celery 任務佇列執行，因此不會阻塞管理介面。

## 什麼時候使用翻譯工作

**啟用新語言**：
- 啟用德語作為新語言
- 創建工作：將所有產品從英語翻譯成德語
- 結果：數分鐘或數小時內（視規模而定），整個目錄即可用德語提供

**新產品導入**：
- 導入 500 個新英語產品
- 創建工作：將新產品翻譯成所有活動語言
- 結果：新庫存立即在所有市場可用

**彌補缺口**：
- 覆蓋報告顯示產品僅有 60% 翻譯成法語
- 創建工作：僅翻譯缺少的法語產品欄位
- 結果：法語覆蓋率提高至約 100%

**更新過時翻譯**：
- 翻譯模型改進或有新的提供商可用
- 創建工作：重新翻譯所有產品至西班牙語
- 結果：目錄中西班牙語翻譯的品質提高

## 創建翻譯工作

導航至 **設定 > 翻譯工作**，並點擊 **+ 創建工作**。

### 工作配置

**工作名稱** - 說明性標籤（例如，"將產品翻譯成德語"，"新部落格文章 - 所有語言"）

**內容類型** - 要翻譯的內容：
- 產品
- 產品分類
- 頁面
- 部落格文章
- SEO 元資料
- 電子郵件模板
- 所有內容類型

**來源語言** - 您要翻譯的語言（通常是預設語言）

**目標語言** - 要翻譯成的一種或多種語言（選擇多個以進行平行翻譯）

**範圍** - 要翻譯的內容子集：
- **所有項目** - 翻譯所有內容，不管現有翻譯情況
- **僅未翻譯** - 跳過已有翻譯的欄位
- **自某日期以來創建/更新** - 僅新或最近更改的內容
- **特定項目** - 選擇個別產品/頁面（高級過濾）

**翻譯引擎** - 使用哪個服務：
- 本地 AI 模型（預設，無 API 成本）
- 特定外部提供商（DeepL、Google、Azure、AWS）
- 自動選擇（使用配置的偏好）

**鎖定翻譯** - 是否鎖定翻譯欄位，防止未來自動覆寫（用於已審核的翻譯）

### 高級選項

**跳過已鎖定欄位** - 如果啟用，會尊重現有的鎖定翻譯（建議啟用）

**覆寫現有** - 即使已有翻譯，也重新翻譯（用於品質改進）

**欄位過濾器** - 僅翻譯特定欄位（例如，產品名稱和描述，跳過屬性）

**批次大小** - 一次處理多少項目（預設：50，如果伺服器能處理，可增加以加快處理速度）

**優先順序** - 高優先順序的工作會在正常優先順序的工作之前執行（請謹慎使用）

## 工作生命週期和狀態

工作會經歷這些狀態：

**已排隊** - 工作已創建，正在等待工作人員處理

**正在處理** - 工作人員正在積極翻譯內容

**已完成** - 所有翻譯成功完成

**失敗** - 工作遇到錯誤（請檢查錯誤日誌）

**已取消** - 被管理員手動停止

**已暫停** - 暫時中止（可重新啟動）

## 監控工作進度

工作詳情頁面顯示：

**進度條** - 完成百分比

**統計數據**：
- 總共需要翻譯的項目數
- 已完成項目
- 剩餘項目
- 預估剩餘時間

**即時日誌** - 翻譯活動的串流（用於除錯）

**錯誤數量** - 有多少欄位翻譯失敗（附帶原因）

## 工作結果和統計數據

當工作完成時，結果頁面顯示：

**摘要**：
- 總共處理的欄位數
- 成功翻譯的欄位
- 翻譯失敗的欄位
- 跳過的欄位（已翻譯、已鎖定或被過濾器排除）

**每個項目的細節**：
- 哪些產品/頁面被翻譯
- 每個項目有多少欄位
- 遇到的任何錯誤

**效能指標**：
- 總共經過的時間
- 每秒平均翻譯數
- 使用的翻譯引擎

## 處理翻譯失敗

如果某些翻譯失敗：

**審查錯誤日誌** - 確定哪些欄位失敗及原因

**常見失敗原因**：
- API 請求限制（外部提供商）
- 翻譯引擎逾時（非常長的文本）
- 無效欄位格式（JSON 解析錯誤）
- 模型不支援語言對

**重試選項**：
- 解決根本問題
- 為僅失敗的項目創建新工作
- 使用不同的翻譯引擎

## 取消和暫停工作

**取消** - 立即停止工作，丟棄任何進行中的翻譯（已完成的翻譯會被保存）

**暫停** - 暫時停止工作，之後可從中斷處重新啟動

**重新啟動** - 繼續暫停的工作

當您需要臨時釋放伺服器資源時，使用暫停/重新啟動功能。

## 批量工作策略

**策略 1：按語言逐個處理**：
- 為每個目標語言創建獨立的工作
- 更容易監控每個語言的進度
- 可優先處理重要語言
- 隨時間分散負載

**策略 2：一次性全部處理**：
- 單一工作翻譯成所有活動語言
- 整體完成更快
- 處理期間伺服器負載更高
- 工作管理更簡單

**策略 3：按內容類型逐個處理**：
- 首先翻譯產品（最高優先級）
- 然後是分類、頁面、部落格文章
- 允許逐步推出
- 更容易測試和驗證翻譯

根據您的伺服器容量、緊迫性和目錄規模來選擇策略。

## 工作排程

安排重複性工作，自動處理新內容：

**每日工作** - 翻譯過去 24 小時內創建/更新的產品

**每周工作** - 每週彌補任何翻譯缺口

**導入後** - 在批量產品導入後自動觸發工作

**語言啟用時** - 啟用新語言時自動創建工作

排程的工作可讓翻譯保持最新，無需手動操作。

## 效能考量

**本地 AI 模型**：
- ~100-500 翻譯/秒（視伺服器而定）
- 處理時 CPU 負荷高
- 無 API 請求限制
- 免費（無每翻譯費用）

**外部提供商**：
- 請求限制因提供商而異（DeepL：每月 500k 字符的免費層）
- API 延遲會增加負荷
- 品質更好但需要費用
- 並行請求限制

**大型工作**（>10,000 個欄位）：
- 在非高峰時間執行
- 監控伺服器資源
- 考慮拆分為較小批次
- 首先用子集進行測試

## 小技巧

- **從小規模開始** - 在執行完整目錄翻譯之前，先在子集（例如 10 個產品）上測試工作
- **使用 "僅未翻譯" 範圍** - 更快，並避免重新翻譯已良好的內容
- **密切監控第一個工作** - 在啟動較大的工作之前，觀察是否有錯誤或品質問題
- **在低流量時期安排工作** - 翻譯會佔用 CPU/API 資源
- **鎖定已審核的翻譯** - 避免批量工作覆寫您的手動編輯
- **保持工作專注** - 比 "翻譯所有內容" 的大規模工作，較小、針對性的工作更容易除錯
- **完成後審查樣本** - 在認為工作成功之前，隨機檢查翻譯品質
- **在進行重大工作前導出/備份** - 以防需要撤銷批量更改
