---
title: 紹介プログラム
---

紹介プログラムは、既存のお客様が家族や友人にユニークな紹介リンクを共有できるようにします。紹介された友人が初めての有効な購入をした場合、紹介者と新規のお客様の両方が報酬を受け取ることができます。これにより、口頭での宣伝を通じた新規顧客の獲得が促進されます。

## 紹介プログラムの仕組み

1. 顧客が友人に自分のユニークな紹介リンク（またはコード）を共有します。
2. 友人がリンクをクリックし、最大30日間（設定可能）クッキーでトラッキングされます。
3. 友人が登録し、初めての有効な注文を置きます。
4. システムは紹介属性記録を作成し、不正行為と資格のチェックを実行します。
5. 属性が承認された場合、両方の当事者に報酬が発行されます。

あなたの店舗には、1つの紹介プログラム設定があります。**マーケティング > 紹介プログラム**に移動して設定してください。

## 紹介プログラムの設定

### プログラムの状態

プログラムには3つの状態があります：

- **下書き** — プログラムは設定中ですが、まだ公開されていません。紹介リンクは無効です。
- **アクティブ** — プログラムは公開されています。顧客はリンクを共有し、報酬を獲得できます。
- **一時停止** — プログラムは一時的に停止されています。既存の属性は引き続き処理されますが、新しい紹介はトラッキングされません。

プログラムを公開する準備が整ったときに、**ステータス**を**アクティブ**に設定してください。いつでも一時停止できます。

### 報酬設定

紹介が成功したときに発行される報酬を定義します。このプログラムは**双方向報酬**をサポートしており、リンクを共有した紹介者（顧客）と、そのリンクを使用した新規顧客（被紹介者）の両方に報酬を支払うことができます。

**報酬設定**フィールドで、各受信者向けの報酬を設定してください。利用可能な報酬の種類は以下の通りです：

| 報酬の種類 | 説明 |
|-------------|-------------|
| **ストアクレジット** | 顧客のウォレットにクレジットを追加し、今後の注文で使用可能 |
| **クーポンコード** | 一意の割引バウチャー コードを生成 |
| **パーセンテージ割引** | チェックアウト時に使用できるパーセンテージ割引を発行 |
| **エクスクルーシブな特典** | カスタム特典（例：無料ギフト、優先アクセス）— 報酬の説明フィールドで記述 |

**例の設定** — リファラーラーに$10のストアクレジット、新規顧客に$10の割引:

```json
{
  "referrer": {"kind": "credit", "amount": 10},
  "referee": {"kind": "discount", "amount": 10},
  "double_sided": true
}
```

`"double_sided": false`に設定する場合は、リファラーラーだけに報酬を支給したい場合のみ。

### 権限ルール

権限ルールは、どのリファラルが報酬資格があるかを決定します。これらは **権限ルール** フィールドで設定します:

| ルール | その動作 |
|------|--------------|
| `new_customer_only` | `true`の場合、紹介された友人は新規顧客（以前の注文がない）でなければなりません |
| `min_order_value` | 紹介された友人が支払う必要がある最低注文額（あなたの店舗通貨で） |
| `exclude_discounts` | `true`の場合、紹介された顧客がバウチャーを使用した注文は資格外となります |
| `exclude_staff` | `true`の場合、スタッフアカウントはリファラーラーまたはリファレーラーとして使用できません |

**例** — 新規顧客のみ、最低$40の注文、スタッフは除外:

```json
{
  "new_customer_only": true,
  "min_order_value": 40.0,
  "exclude_discounts": false,
  "exclude_staff": true
}
```

### タイミング設定

**タイミング設定** フィールドは、資格のある注文後に報酬がいつ発行されるかを制御します:

| 設定 | その動作 |
|---------|--------------|
| `issue_on` | 報酬を発行するタイミング: `signup`（登録時に即時）、`first_purchase`（注文後に即時）、または `post_refund`（返金期間が終了した後） |
| `refund_window_days` | `post_refund`を使用する場合、報酬を発行する前に待つ日数（デフォルト: 14日） |


「post_refund」を使用することは最も慎重なアプローチです — 返品期限が過ぎた後に報酬を発行するため、後に返金される注文に報酬を支払うリスクを低減します。

| 設定 | その役割 |
|---------|--------------|
| `policy` | 全体の厳しさ: `strict`, `balanced`, または `lenient` |
| `auto_reject_threshold` | 承認自動拒否のリスクスコア（0–100）（デフォルト: 80） |
| `auto_approve_threshold` | 承認自動承認のリスクスコア（デフォルト: 30） |
| `check_ip` | `true`の場合、リファラーとリファレンディーが同じIPアドレスを持っているかを確認します |
| `check_device` | `true`の場合、リファラーとリファレンディーの間で共有されたデバイスのフィンガープリントを確認します |
| `check_velocity` | `true`の場合、単一のソースからの異常に高いリファラル率を監視します |
| `velocity_window_hours` | ベロシティチェックの時間窓（時間単位） |
| `max_referrals_per_window` | ベロシティ窓内から1つのソースが許可される最大リファラル数 |

リスクスコアが自動拒否と自動承認のしきい値の間にあるアトリビューションは、**Pending**ステータスになり、手動での確認が必要になります。

### 利用規約

**Terms & Conditions**フィールドにプログラムの法的利用規約を入力してください。このテキストは、顧客がリファラルプログラムを閲覧するときに表示されます。Markdown形式はサポートされています。

## リファラルアトリビューションの閲覧

**Marketing > Referral Attributions**に移動して、すべてのリファラルケース（リファラーとリファレンディーの間の関係）を確認してください。

![Referral attributions list](/static/core/admin/img/help/referral-program/attribution-list.webp)

各アトリビューションは、リファラー、リファレンディー、彼らが最初に注文した内容、現在のステータス、リスクスコアを表示します。

### アトリビューションステータス

| ステータス | その意味 |
|--------|---------------|
| **Pending** | 要確認 — リスクスコアが手動確認範囲内にあります |
| **Approved** | リファラルは有効 — 報酬が発行済みまたは発行予定です |
| **Rejected** | リファラルが対象外または不正とマークされました |
| **Expired** | 跟踪ウィンドウ内でリファラルが変換されませんでした |


### 承認または拒否の手動処理

**保留**ステータスのアトリビューションについては、アトリビューションレコードを開き、アクションボタンを使用して手動で承認または拒否できます。拒否する場合は、**拒否理由**を選択してください。

- 自己紹介
- 新規顧客ではない
- 最小注文額未満
- 一時的なメールアドレス
- キャップを超えた
- 欺骗リスク
- 注文が返金またはキャンセルされた
- 手動拒否

自分の記録のために**拒否のメモ**を追加することもできます。

### リスクレベルによるフィルタリング

サイドバーにある**リスクレベル**フィルタを使用して、確認が必要な高リスクのアトリビューションに焦点を当てます。

- 低リスク（スコア 0–30）— 自動承認
- 中リスク（スコア 31–70）— 手動確認
- 高リスク（スコア 71–89）— 手動確認、注意が必要
- 非常に高リスク（スコア 90+）— 自動拒否

## 発行された報酬の確認

**マーケティング > 発行された報酬**に移動して、承認されたアトリビューションの結果として発行されたすべての報酬を確認できます。

各報酬エントリには、顧客が紹介者か被紹介者か、報酬の種類と金額、および現在の利用ステータスが表示されます。

### 報酬ステータス

| ステータス | 意味 |
|----------|------|
| **保留** | 報酬は作成されたが、顧客にまだ配信されていません |
| **発行済み** | 報酬は有効で、顧客が使用できます |
| **利用済み** | 顧客が報酬を使用しました |
| **期限切れ** | 報酬は使用期限切れとなり、使用されていません |
| **取り消し済み** | 報酬が手動でキャンセルされました（例：報酬が発行された後に元の注文が返金された場合） |

### 報酬の取り消し

報酬をキャンセルする必要がある場合（例：対象の注文が返品された場合）は、報酬レコードを開き、**取り消し**アクションを使用してください。取り消し理由を記録するためにメモを追加してください。

## ヒント

- 最初に `post_refund` のタイミング設定から始めましょう。

報酬を発行する前に、返品ウィンドウが終了するのを待つことで、最終的に返品される注文に報酬を支払うことを防ぐことができます。
- `balanced` の詐欺防止ポリシーは、ほとんどの店舗にとって良いデフォルトです。

不審な少数のアカウントからのリファラル数が急激に増加している場合、`strict` に切り替えることを検討してください。
- 現実的な月間および生涯上限を設定してください。

報酬額が高い場合、1人あたり月間10〜20回の上限を設定することは、悪用を防ぐために適切です。
- **Pending** な属性を週に1回確認してください。

これらを長期間見過ごすと、報酬を待っている正当なリファララーが不満を抱く可能性があります。
- **Risk Level** フィルターを使用して、手動確認キューを優先順位付けましょう — 最高リスクの属性から始めて、中程度リスクに移ってください。
- 利用規約は短く、平易な言葉で記載してください。

顧客は、ルールが明確に理解できる場合、より参加しやすくなります。
