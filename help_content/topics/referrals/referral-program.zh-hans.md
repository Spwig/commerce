---
title: 推荐计划
---

推荐计划允许现有客户与朋友和家人分享唯一的推荐链接。当被推荐的朋友完成首次符合条件的购买时，推荐人和新客户都可以获得奖励——通过口碑营销促进新客户获取。

## 推荐计划的工作方式

1. 客户与朋友分享其唯一的推荐链接（或代码）。
2. 朋友点击链接，并通过 cookie 跟踪最多 30 天（可配置）。
3. 朋友注册并完成首次符合条件的订单。
4. 系统创建推荐归属记录并执行欺诈和资格检查。
5. 如果归属获得批准，双方将获得奖励。

您的商店有一个单一的推荐计划配置。导航到 **营销 > 推荐计划** 来设置它。

## 设置您的推荐计划

### 程序状态

该计划有三种状态：

- **草稿** — 程序正在配置但尚未上线。推荐链接处于非活动状态。
- **活动** — 程序已上线。客户可以分享链接并赚取奖励。
- **暂停** — 程序暂时停止。现有的归属仍然处理，但不会跟踪新的推荐。

当您准备上线时，将 **状态** 设置为 **活动**。您随时可以暂停它。

### 奖励配置

定义当推荐成功时发放的奖励。该计划支持 **双方面奖励** ——这意味着您可以奖励推荐人（分享链接的客户）和被推荐人（使用链接的新客户）。

在 **奖励配置** 字段中为每位接收者配置奖励。可用的奖励类型包括：

| 奖励类型 | 描述 |
|-------------|-------------|
| **店铺积分** | 向客户的钱包中添加积分，可用于未来的订单 |
| **优惠码** | 生成一个唯一的折扣优惠码 |
| **百分比折扣** | 在结账时提供一个百分比折扣 |
| **专属福利** | 一个自定义福利（例如，免费礼品、优先访问权）——在奖励的描述字段中描述 |

**示例配置** —— 推荐人获得10美元店铺积分，新客户获得10美元折扣：

```json
{
  "referrer": {"kind": "credit", "amount": 10},
  "referee": {"kind": "discount", "amount": 10},
  "double_sided": true
}
```

如果只想奖励推荐人，请设置 `"double_sided": false`。

### 资格规则

资格规则决定了哪些推荐符合奖励条件。在 **资格规则** 字段中配置这些规则：

| 规则 | 功能 |
|------|--------------|
| `new_customer_only` | 如果为 `true`，被推荐的朋友必须是新客户（没有之前的订单） |
| `min_order_value` | 被推荐的朋友必须花费的最低订单金额（以您店铺的货币单位计算） |
| `exclude_discounts` | 如果为 `true`，被推荐客户使用优惠券的订单不符合条件 |
| `exclude_staff` | 如果为 `true`，员工账户不能作为推荐人或被推荐人 |

**示例** —— 仅限新客户，最低订单40美元，排除员工：

```json
{
  "new_customer_only": true,
  "min_order_value": 40.0,
  "exclude_discounts": false,
  "exclude_staff": true
}
```

### 时间配置

**时间配置** 字段控制在符合条件的订单之后何时发放奖励：

| 设置 | 功能 |
|---------|--------------|
| `issue_on` | 发放奖励的时间：`signup`（注册时立即发放），`first_purchase`（订单后立即发放），或 `post_refund`（退款窗口期结束后发放） |
| `refund_window_days` | 使用 `post_refund` 时在发放奖励前等待的天数（默认：14天） |

使用 `post_refund` 是最谨慎的做法 —— 它会在退货窗口期结束后才发放奖励，从而降低因后续退货而错误发放奖励的风险。

| 设置 | 功能说明 |
|---------|--------------|
| `policy` | 整体严格程度：`strict`（严格）、`balanced`（平衡）或 `lenient`（宽松） |
| `auto_reject_threshold` | 风险评分（0–100）高于此值的推荐将被自动拒绝（默认值：80） |
| `auto_approve_threshold` | 风险评分低于此值的推荐将被自动批准（默认值：30） |
| `check_ip` | 如果为 `true`，检查推荐人和被推荐人是否共享相同的 IP 地址 |
| `check_device` | 如果为 `true`，检查推荐人和被推荐人是否共享相同的设备指纹 |
| `check_velocity` | 如果为 `true`，监控来自单一来源的异常高推荐率 |
| `velocity_window_hours` | 速度检查的时间窗口（以小时为单位） |
| `max_referrals_per_window` | 在速度窗口内允许来自一个来源的最大推荐数 |

风险评分处于自动拒绝和自动批准阈值之间的推荐将进入 **Pending**（待处理）状态，并需要人工审核。

### 条款和条件

在 **Terms & Conditions**（条款和条件）字段中输入该计划的任何法律条款和条件。当客户查看推荐计划时，将显示此文本。支持 Markdown 格式。

## 查看推荐关联

导航到 **Marketing > Referral Attributions** 以查看所有推荐案例 —— 推荐人和被推荐客户之间的关联。

![Referral attributions list](/static/core/admin/img/help/referral-program/attribution-list.webp)

每个关联显示推荐人、被推荐客户、他们放置的第一笔订单、当前状态和风险评分。

### 关联状态

| 状态 | 含义 |
|--------|---------------|
| **Pending** | 等待审核 —— 风险评分处于人工审核范围内 |
| **Approved** | 推荐有效 —— 奖励已发放或即将发放 |
| **Rejected** | 推荐不符合条件或被标记为欺诈 |
| **Expired** | 推荐在跟踪窗口内未被转换 |


### 手动批准或拒绝归属

对于状态为 **待处理** 的归属，您可以通过打开归属记录并使用操作按钮来手动批准或拒绝它们。拒绝时，请选择 **拒绝原因**：

- 自我推荐
- 非新客户
- 订单金额低于最低限额
- 一次性电子邮件
- 资本超额
- 欺诈风险
- 订单已退款或取消
- 手动拒绝

您还可以添加 **拒绝备注** 以供自己记录。

### 按风险等级筛选

使用侧边栏中的 **风险等级** 过滤器，专注于需要审核的高风险归属：

- 低风险（得分 0–30）——自动批准
- 中等风险（得分 31–70）——手动审核
- 高风险（得分 71–89）——手动审核，需谨慎处理
- 非常高风险（得分 90+）——自动拒绝

## 查看已发放的奖励

导航到 **营销 > 已发放奖励**，查看所有因已批准的归属而发放的奖励。

每个奖励条目会显示客户信息、他们是推荐人还是被推荐人、奖励类型和金额，以及当前的兑换状态。

### 奖励状态

| 状态 | 含义 |
|------|------|
| **待处理** | 奖励已创建但尚未交付给客户 |
| **已发放** | 奖励已激活，客户可以使用 |
| **已兑换** | 客户已使用奖励 |
| **已过期** | 奖励在过期日期前未被使用 |
| **已撤销** | 奖励被手动取消（例如，奖励发放后原始订单被退款） |

### 撤销奖励

如果需要取消奖励——例如，符合条件的订单被退货——打开奖励记录并使用 **撤销** 操作。添加备注说明撤销原因以供记录。

## 小贴士

- 从 `post_refund` 时间设置开始。

在发放奖励前等待退货窗口期结束，可防止对最终被退货的订单进行奖励。
- `balanced` 欺诈策略是大多数商店的良好默认设置。

如果发现来自少量账户的推荐量出现异常激增，请切换到 `strict`。
- 设置合理的每月和终身上限。

如果奖励价值较高，每月每个推荐者的上限设为 10–20 是合理的，以防止滥用。
- 每周审查 **Pending** 的归属记录。

让它们长时间未被审查可能会让等待奖励的合法推荐者感到沮丧。
- 使用 **Risk Level** 过滤器来优先处理你的手动审查队列 —— 从极高风险的归属记录开始，再处理中等风险的。
- 保持你的条款与条件简短且使用通俗语言。

当客户清楚了解规则时，他们更有可能参与。
